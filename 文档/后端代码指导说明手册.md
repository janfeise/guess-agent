# 后端代码指导说明手册

## 1. 手册目的

这份手册不是代码实现稿，而是给你自己手写后端代码时使用的开发说明。它的目标很明确：

- 保持 demo 阶段足够简单
- 让你按步骤独立完成后端
- 先跑通最小闭环，再逐步补功能
- 避免一开始就把结构做复杂

如果你要先完成一个可演示的后端，这份手册可以直接作为开发顺序和编码边界的参考。

## 2. 当前阶段只做什么

### 建议保留

- Docker 目录与 Docker Compose 配置
- 后端目录结构
- FastAPI 作为服务入口
- LangChain 作为模型调用层，主模型和小助手模型分开配置
- MongoDB 作为数据存储

### 暂时不做

- 前端页面
- 多人对战
- 高级记忆系统
- 向量数据库
- 复杂权限和账号体系

## 4. 代码分层规则

### 4.1 API 层

API 层只做三件事：

- 接收请求
- 校验请求参数
- 调用 service 并返回结果

不要把业务逻辑写进路由函数里。

### 4.2 Service 层

Service 层负责流程控制，比如：

- 创建游戏
- 获取游戏状态
- 组织上下文
- 调用 Agent
- 记录回合数据

如果后面你要加规则判断，优先加在 service 层。

### 4.3 Agent 层

Agent 层只负责主模型推理：

- 拼接 prompt
- 调用 DeepSeek
- 返回文本结果

不要让 Agent 层直接操作路由，也不要让它直接替代 Service 层。

小助手如果单独做成一个模块，也应该只读取自己的模型配置，不要和主模型混用同一组参数。

### 4.4 Repository 层

Repository 层只负责 MongoDB：

- 新增数据
- 查询数据
- 更新数据

不要把数据库操作分散到各个 service 里。

## 5. 最小可实现功能

### 5.1 健康检查

先写一个 `/health` 接口，保证服务能启动。

要求：

- 返回 `ok`
- 返回服务名
- 不依赖数据库和模型

### 5.2 创建游戏

再写一个创建游戏的接口。

建议输入：

- 用户答案
- 用户密钥
- 可选的 Agent 答案

建议输出：

- game_id
- status
- created_at

### 5.3 发送问题

写一个对话接口，接收：

- game_id
- question

返回：

- answer
- round_count

### 5.4 保存回合

每次问答都向 MongoDB 追加一条 round 记录。

至少保存：

- question
- answer
- note
- created_at

## 6. LangChain 接入方式

demo 阶段建议只做一条最短链路：

1. 读取 system prompt
2. 接收用户问题
3. 调用 DeepSeek
4. 返回文本结果

### 6.1 Prompt 要点

你的 system prompt 需要明确：

- 角色是猜词游戏 Agent
- 不能直接泄露答案
- 回答要短
- 只能基于给定上下文推理

小助手的 prompt 则应该强调：只能回答查询、解释和提示类问题，不能影响游戏公平。

### 6.2 上下文处理

先不要做复杂记忆。

建议先把历史回合拼成简单列表，传给模型即可。等 demo 跑通后再考虑更复杂的记忆结构。

### 6.3 错误处理

模型调用失败时，返回可读的错误信息，并在日志里记录原始异常。

## 7. MongoDB 设计建议

### 7.1 集合建议

先只用一个 `games` 集合。

### 7.2 文档字段

建议字段：

- `game_id`
- `status`
- `created_at`
- `updated_at`
- `user_answer_encrypted`
- `agent_answer_encrypted`
- `rounds`

### 7.3 索引建议

至少给下面字段建索引：

- `game_id`
- `status`

这样后续查找和更新会更稳定。

## 8. 加密建议

demo 里可以先保留一个简单的加密工具。

建议只做两件事：

- 加密答案
- 解密答案

实现上可以直接用对称加密，也可以先用你现成的密钥派生方式。

注意不要把明文答案写进数据库。

## 9. Docker 配合方式

你已经有 Docker 目录后，后端代码只需要配合下面几点：

- 代码从环境变量读取配置
- MongoDB 连接地址用容器内地址
- FastAPI 监听 `0.0.0.0`
- 镜像里安装依赖后再启动服务

如果你本地调试，直接用 `.env`；如果容器里跑，就让 Compose 注入环境变量。

## 10. 编码顺序

建议你按这个顺序写：

1. 写配置类
2. 写 MongoDB 连接
3. 写健康检查接口
4. 写 Agent 调用类
5. 写游戏 service
6. 写 repository
7. 写创建游戏接口
8. 写问答接口
9. 最后补加密

这个顺序能让你尽快看到结果。

## 11. 接口验收标准

### 11.1 健康检查

访问后能立刻返回成功。

### 11.2 创建游戏

接口返回一个新的 `game_id`，MongoDB 里能看到对应记录。

### 11.3 提问接口

传入 `game_id` 后，Agent 能返回回答，并把这一轮记录保存进数据库。

### 11.4 重启验证

重启服务后，已有游戏记录还能查到。

## 12. 开发中的常见问题

### 12.1 模型调用失败

优先检查：

- 主模型 API Key
- 主模型 Base URL
- 主模型名
- 小助手模型 API Key
- 小助手模型 Base URL
- 小助手模型名
- 网络连接

### 12.2 MongoDB 连接失败

优先检查：

- 容器是否启动
- URI 是否正确
- 用户名密码是否一致
- 端口是否冲突

### 12.3 路由返回 404

优先检查：

- 路由前缀是否写对
- 请求方法是否一致
- 路径是否拼错

### 12.4 环境变量不生效

优先检查：

- `.env` 路径是否正确
- 容器是否重新构建
- 配置类是否正确读取环境变量

## 13. 适合你的最小目标

如果你只想先做出一个 demo，最小目标应该是：

- Docker 一键启动
- `/health` 可用
- `/game/start` 可用
- `/agent/chat` 可用，且主模型独立生效
- 小助手配置可单独保留，后续再接入独立接口
- MongoDB 有记录
- DeepSeek 真能返回结果

先完成这些，再去扩展游戏规则和助手能力，成本会最低。
