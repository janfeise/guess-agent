从提供的日志和 JSON 数据来看，**轮次控制（Turn Control）** 的失效确实导致了系统从“猜词游戏”变成了“审讯式对话”。

在第 1 轮 user 提问后，从第 2 轮到第 21 轮，全部变成了 `agent_question` -> `user_answer` 的单向模式。原本应当是“你问一题，我问一题”的对等博弈，变成了 Agent 单方面通过排除法压榨用户的信息。

针对这一问题，建议从以下几个维度优化系统的**轮次调度逻辑**：

### 1. 显式状态机：引入 `next_actor` 约束

目前系统似乎只根据 `input_text` 的意图识别来驱动。应当在后端引入一个明确的状态字段 `next_actor`：

- **规则限制**：如果上一轮 `actor` 是 `agent` 且 `turn_type` 是 `agent_question`，那么下一轮的 `actor` 必须是 `user`，且 `turn_type` 必须是 `answer`。
- **强制切换**：在 `user` 完成一次 `answer` 后，系统应强制进入 `user` 的 `question` 阶段，而不是允许 `agent` 继续追问。

### 2. 优化 `turn_type` 的双向循环

一个标准的对等回合应包含以下四个子阶段的循环：

1.  **User Question**: 用户提问 -> 系统回答。
2.  **Agent Question**: 系统提问 -> 用户回答。

- **当前问题**：日志显示系统在 `round 3` 用户回答完后，直接跳到了 `round 4` 的 `agent_question`。
- **修复建议**：在 `user` 做出 `answer` 后，逻辑层必须拦截 Agent 的发言权，提示或等待用户发起 `question` 或 `guess`。

### 3. 提示词与 UI 引导

- **LLM 侧 (Judgement Agent)**：在解析 `user_intent` 时，如果当前是 Agent 的提问轮次，识别到 `answer` 后，返回的指令应包含 `action: wait_user_move`，阻止 Agent 连续发问。
- **前端/输出侧**：当用户回答完 Agent 的问题（如第 19 轮回答“是的”），系统不应立刻调用 Agent 生成第 20 轮的问题，而是输出：“回答正确！现在轮到你向我提问了，或者你可以直接猜答案。”

### 4. 异常处理：处理“抢跑”

如果用户连续提问，系统可以回答；但如果 Agent 试图连续提问（如日志中表现的那样），调度器应当判定为非法动作。

- **逻辑校验**：
  ```python
  if last_turn.turn_type == "answer" and last_turn.actor == "user":
      # 此时必须等待 user 发起 question 或 guess
      # 禁止触发 agent_question 逻辑
  ```

### 5. 猜测逻辑的触发时机

日志中第 20 轮 Agent 问“是列夫·托尔斯泰吗？”，第 22 轮才正式发起 `agent_guess`。

- **建议**：Agent 的提问权（Question Quota）应当受到严格限制（例如每轮仅限一个）。一旦 Agent 认为把握很高，它应该消耗自己的“提问轮次”直接转换为 `guess` 动作，而不是在提问中夹带猜测信息，导致用户体验上的冗余。

**总结：**
系统目前缺乏一个 **"轮次锁"**。修复的核心在于：**在 `user` 提交 `answer` 意图后，系统必须挂起 Agent 的主动动作，直到 `user` 提交了下一个 `question` 或 `guess` 意图为止。**
