# 猜词 Agent App 信息架构、低保真原型与核心用户路径

## 项目背景

本项目是一个基于大语言模型的回合制猜词应用，核心玩法是用户与 Agent 轮流通过提问、回答和猜测来缩小目标词范围，直到一方完成正确猜测并结束对局。

项目的产品目标不是做一个单纯的聊天机器人，而是把 LLM 放进一个有明确规则和状态机的游戏场景中，让模型负责理解意图、生成回答和做策略判断，让后端负责状态流转、胜负判定、数据持久化和加密保护。

从当前代码实现来看，系统已经具备以下基础能力：

- 创建新对局并保存局内状态
- 记录用户提问、系统回答、系统猜词和用户判定
- 通过 MongoDB 持久化回合历史和摘要
- 对目标词进行加密存储，避免明文暴露
- 提供独立的小助手能力，用于辅助查询和推理，但不参与胜负判定

因此，这份 IA 与低保真原型主要服务于一个目标：帮助产品、设计和研发对齐单局猜词闭环的页面结构、状态切换和核心交互路径。

## 设计前提

- 当前项目的核心闭环已经明确：创建对局、提交回合、系统提问、用户回答、用户猜词、系统判定、结束对局。
- 当前后端没有登录、排行榜、支付、多人房间等能力，因此本稿只覆盖单局猜词产品的核心闭环。
- 小助手是独立辅助层，只提供知识支持和提问建议，不参与胜负判定。

## 二、页面原型

### 页面名称：首页

### 功能说明

- 承接首次进入 App 的转化动作，让用户快速开始一局游戏。
- 收集开局词和难度，完成新对局创建。
- 提供规则入口和最近对局入口，降低学习成本。

    │ 顶部栏：Logo / 规则 / 历史    │
    ├──────────────────────────────┤
    │ Hero：一句话介绍 + 开始对局    │
    ├──────────────────────────────┤
    │ 开局表单：开局词 / 难度 / 开始  │
    ├──────────────────────────────┤
    │ 最近对局 / 新手引导            │
    └──────────────────────────────┘

### 核心组件

- 顶部导航栏
- 品牌说明区
- 开局词输入框
- 难度选择器
- 开始对局按钮
- 最近对局卡片
- 规则入口卡片
- 空状态提示

### 交互说明

- 输入开局词后即时校验长度和空白字符。

- 点击规则入口进入规则与帮助页。
- 点击最近对局进入对局详情或回放页。

### 页面名称：对局页

### 功能说明

- 这是产品的核心页面，负责承载整局游戏的主要交互。
- 展示当前状态、历史回合、系统回复和用户输入。
- 根据状态自动切换为“提问中”“等待回答”“等待判定”三种操作模式。
- 提供小助手入口，但不影响主局判定。

- 顶部状态栏

- 回合时间线

- 系统猜词确认卡
- 文本输入框
- 快捷动作按钮
- 小助手抽屉

### 交互说明

- 在 user_turn 状态下，用户可输入问题或直接猜词。
- 在 waiting_answer 状态下，输入区优先展示“是 / 否 / 不知道”快捷按钮，也允许补充文本。
- 在 awaiting_judgement 状态下，页面只保留“正确 / 错误”判断动作。
- 当系统给出猜词后，立即弹出确认卡，要求用户判断。
- 提交后更新回合历史，并同步刷新阶段提示。
- 若对局结束，输入区自动禁用，页面切换到结果导向提示。

### 低保真补充

- 移动端建议采用单列结构，先看状态，再看对话，再看输入。
- 桌面端可扩展为左右双栏，左侧放对话，右侧放小助手和回合摘要。

## 三、核心用户路径

### 路径 1：首次进入并完成开局

1. 用户进入首页，先看到游戏简介和开始对局表单。
2. 输入开局词，选择难度，点击开始对局。
3. 系统校验词语格式，创建新对局。
4. 自动跳转到对局页，页面进入 user_turn 状态。
5. 用户开始第一轮提问，正式进入游戏主循环。

### 路径 2：正常问答推进

1. 用户在对局页提出一个问题。
2. 系统记录问题并调用 Agent 生成回答或下一步动作。
3. 如果 Agent 继续提问，页面切换到 waiting_answer 状态。
4. 用户通过“是 / 否 / 不知道”或文本回答继续推进。
5. 系统更新回合历史，并在需要时切换到下一轮提问或猜词判定。

### 路径 3：用户主动猜词

1. 用户在 user_turn 状态直接提交一个猜测词。
2. 系统先识别这是不是猜词意图，再与密文目标词进行比对。
3. 如果猜中，立即结束对局并进入结果页。
4. 如果猜错，记录一轮失败，回到 user_turn，让系统继续提问。

### 路径 4：系统猜词与用户判定

1. Agent 根据历史信息和摘要判断自己可以猜词。
2. 系统展示猜词确认卡，要求用户判断是否正确。
3. 如果用户确认正确，对局结束并进入系统胜结果页。
4. 如果用户判错，系统清空待猜状态，回到 user_turn，继续下一轮。

## 四、状态机摘要

- user_turn：用户提问或主动猜词。
- waiting_answer：等待用户回答 Agent 的问题。
- awaiting_judgement：等待用户判断 Agent 的猜词。
- finished：对局结束，进入结果页。

## 五、产品补充建议

- 首页只保留一个主目标：开始对局，避免入口过重。
- 对局页必须显式展示当前阶段，否则用户很容易不知道此刻该做什么。
- 小助手应作为独立浮层存在，避免和主局状态混淆。
- 历史记录和规则页属于低成本增强项，适合在 MVP 后补齐。
