# FSM 重构对比：改革前后

## 核心改变：从分散到集中

```
┌─────────────────────────────────────────────────────────────────────┐
│                        【重构前】分散的状态判断                        │
└─────────────────────────────────────────────────────────────────────┘

submit_turn(game_id, question)
│
├─ phase = metadata.get("phase")
│
├─ if phase == "awaiting_judgement"          ← 判断 1
│   ├─ if intent not in {"judge", "answer"}
│   │   └─ return _reject_turn()
│   └─ return await _handle_agent_guess_judgement()
│
├─ if phase == "waiting_answer"             ← 判断 2
│   ├─ if intent != "answer"
│   │   └─ return _reject_turn()
│   └─ return await _handle_user_answer()
│
├─ if phase == "user_turn"                   ← 判断 3
│   ├─ if intent == "invalid"
│   │   └─ return _reject_turn()
│   ├─ if intent == "guess"
│   │   └─ return await _handle_user_guess()
│   ├─ if intent == "answer"
│   │   └─ return _reject_turn()
│   └─ if intent == "question"
│       └─ return await _handle_user_question()
│
└─ return _reject_turn()


问题：
❌ 状态判断分散在多个 if-else 链中
❌ 轮次锁检查散落在各处，容易遗漏
❌ 新增状态需要修改 game_service
❌ 代码复杂度高，难以理解业务流程


┌─────────────────────────────────────────────────────────────────────┐
│                       【重构后】集中的 FSM 管理                       │
└─────────────────────────────────────────────────────────────────────┘

submit_turn(game_id, question)
│
├─ fsm = GuessGameFSM(
│      initial_state=phase,                  ← 从数据库恢复状态
│      initial_round_count=round_count
│  )
│
├─ event = {
│      "user_intent": intent,                ← 用户意图
│      "system_judge": ...,                  ← 系统判定（若有）
│      ...
│  }
│
├─ fsm_result = fsm.handle_event(event)      ← 【核心】让 FSM 处理
│   │                                         （包括轮次锁、状态转换）
│   └─ 返回：{
│        "user_current_state": state,
│        "user_next_state": next_state,
│        "system_action": action,
│        ...
│      }
│
├─ handler_name = fsm.get_handler_name(fsm_result) ← 获取处理器名称
│
└─ if handler_name == "handle_user_question":     ← 单层分派
    return await self._handle_user_question(...)
   elif handler_name == "handle_user_guess":
    return await self._handle_user_guess(...)
   elif handler_name == "handle_user_answer":
    return await self._handle_user_answer(...)
   ...


优势：
✅ 状态转换逻辑集中在 FSM
✅ 轮次锁自动在 FSM 中执行
✅ game_service 专注业务逻辑
✅ 新增状态只需修改 FSM
✅ 代码清晰，流程易懂
```

## 状态管理对比

### 重构前：多层判断决策树

```
用户输入 → 解析意图 → 检查 phase
                    ├─ phase = "user_turn"
                    │   ├─ intent = "question"?  → handle_user_question
                    │   ├─ intent = "guess"?     → handle_user_guess
                    │   └─ intent = "invalid"?   → reject
                    │
                    ├─ phase = "waiting_answer"
                    │   ├─ intent = "answer"?    → handle_user_answer
                    │   └─ else                  → reject (轮次锁)
                    │
                    └─ phase = "awaiting_judgement"
                        ├─ intent = "judge"?    → handle_agent_guess_judgement
                        ├─ intent = "answer"?   → handle_agent_guess_judgement
                        └─ else                 → reject (轮次锁)

问题：轮次锁检查分散，每个分支都要手工判断
```

### 重构后：统一的 FSM 决策

```
用户输入 → 解析意图 → FSM.handle_event()
                   │
                   ├─ _validate_event()    [事件验证]
                   ├─ __next_state()       [状态转换 + 轮次锁]
                   ├─ __action()           [动作生成]
                   └─ _should_increment_round() [轮次管理]

                   返回：(next_state, system_action)
                      │
                   FSM.get_handler_name()  [处理器映射]
                      │
                   调用对应处理器

优势：轮次锁由 FSM 自动处理，game_service 无需关心
```

## 轮次锁对比

### 场景：用户在 WAITING_ANSWER 状态中错误提问

**重构前**：

```python
phase = "waiting_answer"
intent = "question"

if phase == "waiting_answer":
    if intent != "answer":
        return self._reject_turn(...)  ← 手工检查
```

问题：需要在 game_service 中记住"waiting_answer 只允许 answer"的规则

**重构后**：

```python
fsm = GuessGameFSM(initial_state="waiting_answer")

event = {"user_intent": "question"}
result = fsm.handle_event(event)
# FSM 内部：
#   - 检查当前状态 = waiting_answer
#   - 检查意图 = question ≠ answer
#   - 返回 next_state = waiting_answer（保持不变）
#   - 返回 system_action = "prompt_answer"

handler_name = fsm.get_handler_name(result)  # → "handle_rejected_turn"
```

优势：规则在 FSM 中定义，自动执行，无遗漏

## 代码示例：完整对比

### 用户提问的流程处理

**重构前**（散落的逻辑）：

```python
async def submit_turn(self, game_id: str, question: str, mode: str = "agent"):
    game = await self.repository.get_game(game_id)
    metadata = game.get("metadata", {}) or {}
    phase = metadata.get("phase", "user_turn")

    intent_result = await self.agent.parse_user_intent(question, ...)
    intent = intent_result.get("intent", "invalid")

    # 问题 1：多层判断
    if phase == "awaiting_judgement":
        if intent not in {"judge", "answer"}:
            return self._reject_turn(...)
        return await self._handle_agent_guess_judgement(...)

    # 问题 2：重复判断
    if phase == "waiting_answer":
        if intent != "answer":
            return self._reject_turn(...)
        return await self._handle_user_answer(...)

    # 问题 3：多个意图判断
    if phase == "user_turn":
        if intent == "invalid":
            return self._reject_turn(...)
        if intent == "guess":
            return await self._handle_user_guess(...)
        if intent == "answer":
            return self._reject_turn(...)
        if intent == "question":
            return await self._handle_user_question(game, question, history, mode)

    # 问题 4：兜底处理
    return self._reject_turn(game_id, phase, "当前轮次状态异常", history)
```

**重构后**（FSM 统一管理）：

```python
async def submit_turn(self, game_id: str, question: str, mode: str = "agent"):
    from app.core.guessGameFSM import GuessGameFSM

    game = await self.repository.get_game(game_id)
    metadata = game.get("metadata", {}) or {}
    phase = metadata.get("phase", "user_turn")
    history = game.get("history", [])

    # 创建 FSM，从数据库恢复状态
    fsm = GuessGameFSM(initial_state=phase, initial_round_count=len(history))

    # 解析用户意图
    intent_result = await self.agent.parse_user_intent(question, ...)
    intent = intent_result.get("intent", "invalid")

    # 构建事件（关键数据）
    event = {"user_intent": intent}
    if intent == GuessGameFSM.Intent.GUESS:
        event["system_judge"] = "correct" if self.is_guess_correct(...) else "incorrect"
    # ...其他事件字段...

    # FSM 处理：状态转换 + 轮次锁 + 动作生成
    fsm_result = fsm.handle_event(event)

    # 获取处理器名称
    handler_name = fsm.get_handler_name(fsm_result)

    # 简洁的处理器分派
    if handler_name == "handle_user_question":
        return await self._handle_user_question(game, question, history, mode)
    elif handler_name == "handle_user_guess":
        return await self._handle_user_guess(game, intent_result)
    elif handler_name == "handle_user_answer":
        return await self._handle_user_answer(game, question, history, intent_result)
    elif handler_name == "handle_agent_guess_judgement":
        return await self._handle_agent_guess_judgement(game, question)
    elif handler_name == "handle_rejected_turn":
        return self._reject_turn(game_id, phase, fsm_result.get("message"), history)
    # ...其他处理器...
    else:
        return self._reject_turn(game_id, phase, "系统内部错误", history)
```

**对比**：

- 重构前：12 个 if-else 嵌套判断
- 重构后：1 个处理器分派 if-elif 链
- 改进：逻辑清晰，易于维护
