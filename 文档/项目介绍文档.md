# Guess Agent 项目介绍文档

## 1. 项目简介

Guess Agent 是一个以“大语言模型 + 回合制规则 + 安全加密”为核心的猜词对抗游戏项目。用户和 AI 会围绕同一个目标词展开多轮问答、推理和猜测，最终由服务端完成命中判定与游戏结算。

这个项目的重点不只是“能聊天”，而是把 LLM 放到一个规则明确、边界清晰、结果可校验的游戏场景里：

- 用户和 AI 都需要遵守回合制约束
- 目标词在游戏过程中始终以加密形式保存
- LLM 负责理解意图、生成回答和辅助推理
- 最终胜负由后端服务端逻辑裁决，而不是由模型直接决定

项目当前包含后端 FastAPI 服务、React 前端，以及一套围绕 Agent、加密、状态机和历史记录展开的业务实现。

## 2. LLM 使用方式

项目中的 LLM 不是单一聊天入口，而是按职责拆分成两类模型：

- 主 Agent 模型：负责游戏中的推理、提问、猜测和对话生成
- 辅助模型：负责小助手场景下的知识查询和建议生成

### 2.1 模型调用方式

后端的 Agent 层通过 ChatOpenAI 构建模型实例，并根据不同模式选择不同的模型：

- `agent_model` 用于猜词主流程：deepseek-ai/DeepSeek-V3.2
- `helper_model` 用于小助手能力：Qwen/Qwen2.5-7B-Instruct

提示词不硬编码在逻辑里，而是通过 PromptLoader 按名称加载，例如：

- `guess_system`
- `helper_system`

### 2.2 上下文与记忆策略

Agent 不会把整局游戏无脑塞给模型，而是由 MemoryPolicy 控制上下文：

- 只保留最近的有效历史
- 对更早的内容做摘要
- 过滤掉不该进入推理上下文的内容

这样做的目标是兼顾：

- 对话连贯性
- 上下文长度控制
- 推理稳定性

### 2.3 安全边界

LLM 在这个项目里承担的是“推理与表达”职责，不承担最终裁决职责。

后端会在服务层做最终判断，例如：

- 用户是否真的猜中目标词
- 系统词是否需要解密展示
- 当前回合是否允许执行某种动作
- 游戏是否结束

也就是说，模型可以“建议”和“表达”，但不能直接决定结果。

## 3. 核心模块

### 3.1 后端核心模块

- `app/main.py`：后端入口，负责 FastAPI 初始化、CORS、数据库生命周期和路由挂载
- `app/api/`：接口层，负责接收请求、返回响应
- `app/services/game_service.py`：游戏业务核心，负责回合控制、结果判断、历史整理、详情组装
- `app/agents/guess_agent.py`：Agent 核心逻辑，负责 LLM 调用、提示词加载、上下文拼装、意图识别
- `app/repositories/`：数据访问层，负责游戏记录的持久化与查询
- `app/schemas/`：请求和响应模型定义
- `app/core/`：配置、数据库、加密、常量和主题词池等基础能力
- `app/agents/utils/`：流式输出、提示词加载、记忆策略等支撑工具

### 3.2 前端核心模块

- `src/App.tsx`：前端页面总入口，管理 home、game、history、rules、result 等页面切换
- `src/hooks/useGame.ts`：前端游戏状态 Hook，负责创建游戏、提交回合、加载详情、维护消息列表
- `src/services/gameApiClient.ts`：API 封装层，统一处理健康检查、创建游戏、提交回合和查询历史
- `src/pages/`：页面层，承载首页、游戏页、历史页、规则页和结果页
- `src/components/`：通用组件，如顶部栏、底部导航、聊天气泡等
- `src/store/`：前端状态存储，例如匿名身份
- `src/types/` 与 `src/lib/`：类型定义和数据转换辅助

## 4. 后端架构

后端采用的是“分层单体”的组织方式，适合当前阶段快速迭代，也方便后续把能力拆出去。

### 4.1 请求链路

典型的后端处理链路如下：

前端请求 → FastAPI 接口层 → GameService 业务层 → GuessAgent 推理层 → Repository 数据层 → MongoDB

### 4.2 分层职责

#### API 层

API 层只处理协议和参数，不承载复杂业务：

- 接收请求
- 参数校验
- 调用业务服务
- 返回 JSON 响应

#### Service 层

Service 层是游戏规则的真正控制中心，负责：

- 创建游戏
- 管理回合状态
- 处理用户提问、回答和猜测
- 组装可展示的历史记录
- 执行结果判定与结算
- 对敏感内容做脱敏或隐藏

#### Agent 层

Agent 层专注于 LLM 能力：

- 加载提示词
- 组织历史上下文
- 选择主模型或辅助模型
- 生成回答、建议和推理结果

#### Repository 层

Repository 层只负责数据读写：

- 保存游戏记录
- 查询游戏详情
- 查询用户历史

#### Core 层

Core 层提供底层基础能力：

- 配置读取
- 数据库连接
- 加密和解密
- 常量与主题词池

### 4.3 关键设计点

这个后端最重要的设计点有三个：

1. 目标词加密存储，避免游戏过程中泄露答案
2. 回合状态由服务端统一控制，避免前端或模型越权
3. LLM 只做推理和语言生成，最终结果由后端校验

## 5. 前端架构

当前主前端是 React + TypeScript 方案，整体结构比较清晰，适合做对话类游戏界面。

### 5.1 页面层

前端页面由 `App.tsx` 统一调度，主要页面包括：

- Home：开始游戏、进入历史、查看规则
- Game：核心对局页面，展示聊天和回合状态
- History：历史记录页面
- Rules：玩法说明页面
- Result：结算结果页面

页面切换由 App 统一维护，配合 motion/react 做过渡动画，让不同页面之间的切换更自然。

### 5.2 状态与交互层

`useGame.ts` 是前端最核心的状态组织点，负责：

- 创建新游戏
- 发送用户输入
- 提交答案或判断
- 加载游戏详情
- 同步消息列表和游戏状态

它把“界面状态”和“后端回合状态”连接起来，是前端业务的中枢。

### 5.3 接口层

`gameApiClient.ts` 统一封装所有接口调用，负责：

- 健康检查
- 创建游戏
- 提交回合
- 获取游戏详情
- 获取用户历史

这样做的好处是：

- 页面和 Hook 不需要关心具体 URL
- 便于统一处理错误
- 便于后续替换后端地址或增加缓存策略

### 5.4 组件层

组件层主要负责展示和复用：

- 顶部栏
- 底部导航
- 聊天气泡
- 游戏辅助卡片

页面只负责组合组件，不直接堆业务逻辑。

## 6. 核心业务流程

一局完整游戏的基本流程如下：

1. 用户在前端创建游戏
2. 前端通过 API 创建后端游戏记录
3. 后端为用户词和系统词建立加密与状态
4. 用户进入问答阶段，前端持续提交回合
5. 服务端根据阶段、历史和意图决定是否允许该动作
6. Agent 生成回答或推理建议
7. 服务端判断用户是否猜中、系统是否结束、是否进入下一阶段
8. 前端根据返回结果更新消息、状态和结算页

## 7. 项目定位总结

这个项目的价值在于把 LLM 放进一个“可控、可验、可复盘”的游戏系统里，而不是做一个泛聊天壳：

- 规则明确：回合、意图、阶段都有后端控制
- 结果可信：最终答案由服务端判定
- 体验完整：有历史、结算、规则和回放
- 扩展友好：Agent、API、UI 都按分层组织

如果后续要继续扩展，比较自然的方向是：

- 增加更多题材或学科池
- 扩展小助手能力
- 增加更丰富的历史分析和统计面板
- 把当前单体后端进一步拆成更清晰的服务边界
