# 双词系统隔离实施手册

## 1. 文档目标

这份手册不是在解释“LLM 为什么猜错”，而是在解决一个更底层的问题：**两个独立词源被塞进了同一条推理链**。

你的目标不是只修一个 prompt，而是把系统拆成两个互不污染的世界：

- 系统词只服务于系统裁判回合
- 用户词只服务于 Agent 推理回合
- 两者可以同时存在，但不能共用同一份 reasoning history

如果你想先做最稳的版本，也可以直接退回单词模式。但如果你要保留“双词玩法”，就必须按这份手册做隔离。

---

## 2. 问题复盘

从你给出的日志可以直接看出污染链路：

1. 用户提问：`是文科吗？`
2. 系统回答：基于系统词 `刑法`，返回 `yes`，并附带 `hint = 法律学科`
3. Agent 决策：把这条回答当成可用于推理用户词的信息
4. Agent 继续问：`它是否与法律学科相关？`
5. 后续所有判断都开始朝 `刑法` 的方向收敛

这里的核心错误不是模型“理解能力差”，而是**信息来源没有标记**。

当前 history 里只有“问了什么、答了什么”，没有说明：

- 这条答案是基于哪个词生成的
- 这条答案是否允许被 Agent 看到
- 这条 hint 是否只是展示信息，还是推理信息

只要这几个字段缺失，两个世界就一定会串线。

---

## 3. 设计原则

### 3.1 不要让 LLM 充当状态机

LLM 可以做的事情：

- 识别用户意图
- 生成自然语言回答
- 给出下一步动作建议

LLM 不能替代程序做的事情：

- 决定数据归属
- 决定 history 哪些内容可见
- 决定哪个词源进入哪个推理链
- 决定游戏状态是否切换

### 3.2 history 必须分层

建议把历史分成两层：

- `raw_history`：完整审计日志，保留全部事件
- `reasoning_history`：喂给 Agent 的过滤后历史，只保留允许推理的内容

程序写库时保存 `raw_history`，Agent 决策时只读 `reasoning_history`。

### 3.3 hint 不能默认进入推理链

你现在的 `answer_question()` 会返回 `hint = 法律学科`。

这个 hint 如果进入 `decide_agent_action()`，就等于把系统词特征直接喂给 Agent，污染会非常快。

所以必须明确：

- `hint` 可以给前端展示
- `hint` 可以留在 raw history
- `hint` 默认不能进入 Agent 决策 prompt

### 3.4 两个词源必须有明确归属

至少要区分下面几类信息：

- `source_word_owner`：这条信息来自 system 还是 user
- `visible_to_agent`：这条信息是否允许进入 Agent 推理
- `visible_to_player`：这条信息是否允许显示给前端

只要这三个字段存在，后续隔离就有抓手。

---

## 4. 推荐架构

### 4.1 推荐模式：保留双词，但推理隔离

推荐你保留双词玩法，但把它拆成两个完全独立的链路：

- 系统词链路：用户提问 -> 系统裁判回答
- 用户词链路：Agent 推理 -> 继续提问/猜测 -> 用户确认

这两个链路可以同存，但不能共享同一份可推理上下文。

### 4.2 如果要最快稳定，也可以先切成单词模式

如果你当前只是想先把项目跑通，最稳的做法其实是：

- 只保留一个真实答案
- 删除双词对抗逻辑
- 先把标准 20 Questions 链路跑稳

但如果你的目标明确就是“双词隔离”，那下面的方案才是正解。

---

## 5. 核心数据结构

### 5.1 history 记录建议

每条历史记录建议至少带上这些字段：

```json
{
  "round_no": 1,
  "actor": "user",
  "turn_type": "question",
  "input_text": "是文科吗？",
  "answer_text": "是哦，属于文科领域哦",
  "answer_label": "yes",
  "hint": "法律学科",
  "confidence": 1.0,
  "source_word_owner": "system",
  "visible_to_agent": false,
  "visible_to_player": true,
  "result": "刑法属于法学，法学是典型的文科门类"
}
```

如果这条记录是用户词推理链中的内容，则可以写成：

```json
{
  "round_no": 3,
  "actor": "user",
  "turn_type": "answer",
  "input_text": "无关",
  "answer_text": "no",
  "source_word_owner": "user",
  "visible_to_agent": true,
  "visible_to_player": true
}
```

### 5.2 metadata 建议

建议在游戏 metadata 里加上这些字段：

```json
{
  "phase": "user_turn",
  "game_mode": "dual_word_isolated",
  "reasoning_scope": "user_word",
  "awaiting_answer": false,
  "awaiting_judgement": false,
  "pending_question": "",
  "pending_guess": "",
  "agent_confidence": 0.0
}
```

这里最关键的是 `reasoning_scope`。它告诉后端：

- 当前 Agent 在推哪一边的词
- 哪些记录可以进入推理上下文
- 哪些记录只能留在审计日志里

---

## 6. 必须拆开的三个函数

你当前真正需要改的，不是一个大而全的控制器，而是三个关键点：

- `GameService._handle_user_question()`
- `GameService._handle_user_answer()`
- `GuessAgent.decide_agent_action()`

### 6.1 `_handle_user_question()` 不要直接喂原始 history

现在的问题是：

- 系统回答了用户问题
- 然后把整段 `updated_history` 直接喂给 `decide_agent_action()`

这一步会把系统词的回答强行变成 Agent 的推理材料。

正确做法是：

1. 先记录完整历史
2. 再构造一份 `reasoning_history`
3. 只把允许进入推理链的记录传给 Agent

示意如下：

```python
raw_history.append(system_answer_record)
reasoning_history = build_reasoning_history(raw_history, scope="user_word")
agent_action = await self.agent.decide_agent_action(reasoning_history, summary)
```

如果当前这条系统回答本来就不属于用户词推理链，那就不要让 Agent 看见它。

### 6.2 `_handle_user_answer()` 也要保持同样的隔离

用户回答阶段的记录，也要明确它属于哪一条链：

- 这是在回答系统词问题
- 还是在回答 Agent 对用户词的追问

不要把这两个回答混成同一个 `turn_type = answer` 然后直接丢进统一 history。

### 6.3 `decide_agent_action()` 只能吃过滤后的上下文

`decide_agent_action()` 的输入应该是：

- 过滤后的 history
- 或者已经过 scope 处理的 summary

不能直接接整份 raw history。

如果 prompt 里出现了系统词的 `hint`，或者系统裁判语义，Agent 就会自然沿着那条线继续推理。

---

## 7. 构造 reasoning history

建议在 `GameService` 里单独加一个辅助方法，例如：

```python
def build_reasoning_history(self, history: list[dict], target_owner: str) -> list[dict]:
    filtered = []
    for item in history:
        if item.get("source_word_owner") != target_owner:
            continue
        if not item.get("visible_to_agent", True):
            continue

        filtered.append({
            "round_no": item.get("round_no"),
            "actor": item.get("actor"),
            "turn_type": item.get("turn_type"),
            "input_text": item.get("input_text"),
            "answer_text": item.get("answer_text"),
            "answer_label": item.get("answer_label"),
            "confidence": item.get("confidence", 0.0),
        })

    return filtered
```

这个方法的意义是：

- 把“审计日志”与“推理上下文”拆开
- 去掉不该进入 Agent 的 hint 和内部说明
- 让 prompt 看到的是干净、单一来源的信息

---

## 8. prompt 也要拆，不然代码拆了还是会串

### 8.1 `answer_question` 的 prompt 只负责对单个词源作答

[backend/app/agents/guess_agent_fixed.py](../backend/app/agents/guess_agent_fixed.py) 里的 `answer_question()` 当前会接收真实答案，然后生成给用户看的自然语言回复。

这个 prompt 没问题，但你要补一个约束：

- 它的输出只用于当前回答
- 它的 `hint` 不得默认进入 Agent 决策
- 它不能暗示“这条信息属于哪一侧的推理链”

### 8.2 `decide_agent_action` 的 prompt 必须声明数据边界

`agent_decision.txt` 里要明确写清楚：

- 你只能使用传入的 reasoning history
- 不能使用未过滤的原始 history
- 如果某条信息标记为 `source_word_owner = system`，且当前目标是 `user_word`，则不得将其当作推理依据

### 8.3 `turn_router.txt` 也要和状态机对齐

用户意图识别 prompt 要继续保留，但它判断的不是“这句像不像问题”，而是“这句属于哪条回合链”。

也就是说，`question / answer / guess / invalid` 这些分类本身没问题，但返回结果之后，程序要再判断：

- 这条意图属于 system side 还是 user side
- 是否允许进入 Agent 上下文

## 9. 推荐的状态流转

下面这个流程适合双词隔离：

```text
用户输入
  -> parse_user_intent
  -> 判断这条输入属于哪个词源
  -> 如果是 system_word 的问题：只回答，不进入 Agent 推理
  -> 如果是 user_word 的问题：进入用户词推理链
  -> 记录 raw history
  -> 构造 reasoning history
  -> decide_agent_action
  -> if ask: 生成 Agent 问题
  -> if guess: 进入 awaiting_judgement
```

### 9.1 关键点

最容易出错的地方是这一步：

```text
系统回答了用户问题
  -> 直接拿这条回答去推 Agent 下一步
```

这一步在双词模式里是错的。

必须改成：

```text
系统回答了用户问题
  -> 只写 raw history
  -> 决定是否进入 reasoning history
  -> 再判断是否允许 Agent 看见
```

## 10. 具体改造顺序

### 第一步：先给 history 加来源字段

先改 [backend/app/services/game_service.py](../backend/app/services/game_service.py) 中所有写 history 的地方，统一补上：

- `source_word_owner`
- `visible_to_agent`
- `visible_to_player`

如果你不想一次改太多，至少先把 `source_word_owner` 加上。

### 第二步：拆出 reasoning history

新增一个构造函数，专门把 raw history 转成 Agent 可见历史。

这一步是隔离的核心。

### 第三步：改 `_handle_user_question()`

把当前“写完历史就直接决策”的流程改成：

1. 记录完整回答
2. 生成过滤后的 reasoning history
3. 再决定是否调用 `decide_agent_action()`

如果当前记录不属于 Agent 推理链，就直接返回，不调用 Agent。

### 第四步：改 `decide_agent_action()` 的入参

让 [backend/app/agents/guess_agent_fixed.py](../backend/app/agents/guess_agent_fixed.py) 接收的是过滤后的历史，而不是原始历史。

### 第五步：改 prompt

更新这些 prompt：

- [backend/app/agents/prompt/turn_router.txt](../backend/app/agents/prompt/turn_router.txt)
- [backend/app/agents/prompt/guess_system.txt](../backend/app/agents/prompt/guess_system.txt)
- [backend/app/agents/prompt/agent_decision.txt](../backend/app/agents/prompt/agent_decision.txt)

把“不同词源不能混读”写进 prompt 规则里。

### 第六步：检查返回值

如果 `answer_question()` 返回了 `hint`，确认：

- 前端能看到，但只是展示用途
- `decide_agent_action()` 看不到
- raw history 可以留，但 reasoning history 必须删掉

---

## 11. 最小可行改造建议

如果你想先做最小可行改造，我建议按下面的优先级：

1. 给 history 加 `source_word_owner` 和 `visible_to_agent`
2. 新增 `build_reasoning_history()`
3. 让 `_handle_user_question()` 先过滤再决策
4. 给 `agent_decision` prompt 加隔离规则
5. 把 `hint` 从 Agent 决策输入里彻底移除

这五步做完，污染会明显下降。

---

## 12. 验收标准

下面这些标准都满足后，才算真的隔离成功：

- `history` 里能看见完整事件，但每条记录都有来源字段
- `decide_agent_action()` 的输入里没有系统词的 `hint`
- Agent 不会把 `系统词的 yes/no` 当成 `用户词` 的证据
- 用户问“是文科吗？”时，Agent 不会因为 `法律学科` 继续朝系统词方向推理
- 用户回答 `无关` 时，只影响当前词源链路，不污染另一条链路

---

## 13. 结论

这次的问题不是“LLM 反应慢”或者“模型理解错了”。

本质是：**你把两个词源的数据混进了一份 history，然后又把这份 history 原样喂给了 Agent。**

只要你按下面两条做，问题就会消失很多：

1. history 必须带来源和可见性
2. Agent 只能看过滤后的 reasoning history

如果你愿意，下一步可以继续把这份手册拆成具体的后端改造清单，直接对应到 `game_service.py` 和两个 prompt 文件。
