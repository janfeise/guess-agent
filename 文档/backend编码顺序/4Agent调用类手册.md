# 4Agent调用类手册

## 1. 文档目的

这一章的目标是指导你完成后端的 Agent 调用类，也就是把 LangChain、主模型、小助手模型、提示词和游戏上下文串起来的核心模块。

这一章写完后，你应该能够理解并实现以下事情：

- 如何把 prompt 单独抽离成可配置文件
- 如何使用 LangChain 组织模型调用
- 如何实现流式输出
- 如何在“每局游戏”维度管理上下文和记忆
- 如何判断哪些记忆该保留，哪些记忆不该进入 prompt
- 是否还需要补充额外模块

这一章非常关键，因为 Agent 调用类决定了“模型怎么回答”，而不是简单地“把问题丢给模型”。

## 2. 为什么这一章要单独写

在这个项目里，Agent 不只是一个普通的聊天调用器，它承担的是“猜词推理核心”。

如果这一层设计不好，会出现这些问题：

- prompt 散落在代码里，后续改规则很麻烦
- 同一局游戏的上下文丢失
- 不同局游戏的记忆串线
- 无法做流式输出，前端体验差
- 主模型和小助手模型职责混乱

所以 Agent 调用类必须单独抽出来，并且把 prompt、模型、记忆、输出方式分开管理。

## 3. Agent 调用类的职责边界

Agent 调用类只负责“模型交互层”的工作。

它应该做的事情：

- 读取配置
- 加载 prompt
- 组装消息
- 调用主模型或小助手模型
- 支持流式输出
- 处理上下文窗口
- 返回模型结果

它不应该做的事情：

- 不直接保存 MongoDB 数据
- 不直接处理接口参数校验
- 不直接做游戏胜负判断
- 不直接替代 Service 层的流程控制

如果后面需要记录回合、判断猜中与否，应该交给 Service 层处理。

## 4. LangChain 在这个项目里的角色

你现在是初步接触 LangChain，所以这里先抓最核心的理解：

LangChain 不是“模型本体”，它是一个把模型、提示词、消息、工具、记忆串起来的编排框架。

在你的项目里，LangChain 主要负责：

- 把 system prompt、历史消息和当前问题组织成模型输入
- 把模型输出组织成可读响应
- 支持流式调用
- 以后如果需要工具调用或记忆增强，也能继续扩展

当前 demo 阶段不需要上来就做复杂 Agent 链路，先做“可控、稳定、可维护”的最小链路就够了。

## 5. 这一章要解决的核心问题

### 5.1 prompt 为什么要单独抽离

因为 prompt 会不断迭代。

如果把 prompt 直接写死在 Python 代码里，每次改规则都要改代码、重建、再验证，效率很低。

把 prompt 抽离成独立文件后，你可以：

- 不改代码只改文案
- 针对主模型和小助手分别维护不同 prompt
- 更方便做版本对比
- 更容易让后续规则迭代

### 5.2 为什么要做流式传输

猜词类场景非常适合流式输出。

原因是：

- 用户希望尽快看到模型开始思考
- 长回答如果等完整返回，体感会慢
- 流式输出更适合后续前端聊天区展示

### 5.3 记忆为什么要按“每局游戏”划分

这个项目的记忆不能按“全局聊天”来做，而应该按“game_id”隔离。

原因是：

- 每局游戏的答案不同
- 每局游戏的上下文不同
- 每局游戏的推理目标不同
- 不能让上一局游戏的线索污染下一局

所以记忆的基本边界应该是：**一局游戏一个上下文空间**。

## 6. prompt 文件应该怎么抽离

### 6.1 推荐目录

建议把 prompt 文件放在下面这个结构中：

```bash
backend/
└── app/
    └── agents/
        ├── guess_agent.py
        └── prompt/
            ├── guess_system.txt
            ├── helper_system.txt
            └── game_rule.txt
```

如果后面还要继续细分，也可以拆成：

- 主模型系统提示词
- 游戏规则提示词
- 小助手提示词
- 输出格式提示词

### 6.2 为什么建议单独文件

因为 prompt 属于“策略配置”，不是“业务代码”。

单独文件的好处：

- 可读性高
- 可快速迭代
- 方便对比多个版本
- 方便非代码方式调整模型行为

### 6.3 prompt 加载方式建议

建议由 Agent 调用类提供一个“读取 prompt 文件”的能力，而不是让外部模块自己去读。

这样后续你可以统一管理：

- 文件路径
- 读取编码
- 缺失时的默认兜底
- 不同模型对应不同 prompt

### 6.4 prompt 里应该写什么

主模型的 prompt 应该明确：

- 你是猜词游戏 Agent
- 不能直接泄露答案
- 只能基于当前游戏上下文推理
- 回答要简洁明确
- 必须遵守游戏规则

小助手的 prompt 应该明确：

- 只提供知识查询、解释和提示
- 不参与答案推理
- 不影响游戏公平
- 不应该直接接触答案明文

## 7. Agent 调用类的推荐结构

建议把 Agent 调用类理解成四层：

```bash
GuessAgent
├── 读取配置
├── 读取 prompt
├── 组装消息
└── 调用模型 / 流式输出
```

### 7.1 初始化层

负责读取配置里的：

- 主模型 API Key
- 主模型地址
- 主模型名称
- 主模型温度

如果是小助手，则读取对应的小助手模型配置。

### 7.2 Prompt 层

负责从 prompt 文件中读取 system prompt，并在必要时动态拼接游戏规则、当前局历史和当前问题。

### 7.3 消息组织层

负责把 system message、历史消息、当前用户问题，组织成模型可以理解的输入。

### 7.4 输出层

负责把模型输出返回给 Service 层，或者以流式形式逐步吐给前端。

## 8. LangChain 调用思路

### 8.1 基本理解

在你的项目里，LangChain 最基本的用法不是“神秘链条”，而是：

1. 准备 system prompt
2. 准备历史消息
3. 准备当前问题
4. 调用模型
5. 拿回结果

你可以把它理解为“消息编排器 + 模型适配器”。

### 8.2 为什么建议先从消息模型开始

因为你的项目主要是<u>对话式推理</u>，不是复杂工具链任务。

所以第一阶段建议先用消息式调用方式，把思路先跑通，再考虑后续是否需要工具、函数调用或更复杂链路。

### 8.3 为什么不建议一开始就做复杂链

因为你当前的目标是 demo，不是完整 Agent 平台。

如果一开始就做复杂链，会导致：

- 调试困难
- 问题定位慢
- prompt 与逻辑耦合
- 影响你对游戏规则本身的判断

## 9. 流式传输应该怎么理解

### 9.1 什么是流式传输

流式传输就是模型在生成回答的过程中，后台一边生成、一边把内容逐步传给前端，而不是等完整答案结束后一次性返回。

### 9.2 为什么适合这个项目

因为猜词游戏天然有“思考过程”，流式输出能让用户感受到：

- 模型正在推理
- 模型正在组织线索
- 回答不是机械地一次性弹出

### 9.3 推荐的传输协议

demo 阶段建议优先使用 SSE，也就是 Server-Sent Events。

原因是：

- 比 WebSocket 更轻量
- 单向输出就够用
- 更适合聊天式回答流
- 前端接入简单

如果后面你要做多人对战、实时协作或双向交互，再考虑 WebSocket。

### 9.4 流式传输在系统中的路径

建议路径如下：

```text
LangChain 模型流
  -> Agent 调用类
  -> FastAPI 流式响应
  -> 前端聊天框逐步显示
```

### 9.5 流式时要注意什么

- 不要把整段回答先拼完再输出
- 不要在流式输出中丢失上下文
- 不要让前端以为回答结束太早
- 最后记得发送结束标记

## 10. 记忆策略怎么设计

这是这一章最重要的部分之一。

### 10.1 先回答你的核心问题

**每局游戏是否独立？**

是，必须独立。

每个 `game_id` 都应该有自己独立的上下文空间，不能和别的局混在一起。

**每轮游戏是否独立？**

不完全独立。

在同一局游戏里，每一轮回合都应该继承前面的上下文，因为猜词本身就是递进式推理。

所以正确做法是：

- 局与局之间独立
- 局内轮次共享上下文
- 但上下文要控制长度，不能无限增长

### 10.2 短期记忆怎么用

短期记忆就是“当前这一局游戏里，模型真正要看的上下文”。

在这个项目里，短期记忆建议包括：

- 当前局的 system prompt
- 当前局的游戏规则摘要
- 当前局最近几轮问答
- 当前局最近的推理结论

短期记忆适合直接进入 prompt。

### 10.3 长期记忆怎么用

长期记忆不是当前 demo 的重点，但可以先规划。

长期记忆更适合存这些内容：

- 用户偏好
- 常用提问策略
- 历史游戏总结
- 高层经验归纳

但注意：长期记忆不能直接替代当前局上下文。

在你的项目里，长期记忆如果要用，应该是“辅助性”的，而不是“主上下文”。

### 10.4 当前 demo 是否需要长期记忆

建议答案是：**暂时不需要真正的长期记忆系统**。

原因：

- demo 重点是先跑通一局游戏
- 复杂长期记忆会增加调试成本
- 当前阶段更适合把 MongoDB 作为记录层，而不是复杂记忆层

换句话说：

- MongoDB 可以存全量历史
- 但真正喂给模型的，只保留短期上下文和摘要

### 10.5 记忆窗口应该怎么截断

如果一局游戏回合很多，不能把所有历史都原封不动塞进 prompt。

建议策略：

1. 保留 system prompt
2. 保留当前局规则摘要
3. 保留最近若干轮对话
4. 更早的内容压缩成摘要

一个实用的 demo 策略是：

- 最近 6 到 10 轮直接放入上下文
- 更早轮次做一句话摘要

具体保留多少轮，取决于模型上下文长度和回答质量。

### 10.6 每轮游戏在记忆上应该如何处理

每轮新问题到来时，Agent 调用类应该：

1. 按 `game_id` 取出当前局历史
2. 读取最近若干轮
3. 拼入当前问题
4. 调用模型
5. 将本轮结果返回给 Service 层保存

也就是说：

- 当前轮不是孤立的一次问答
- 当前轮是当前局上下文的一部分
- 每轮都会影响下一轮的推理

### 10.7 什么时候需要做摘要

当满足以下任一情况时，就应该考虑摘要：

- 历史轮数太多
- prompt 已经接近模型上下文上限
- 回答开始变慢
- 模型开始“忘记前文”

摘要的目的不是省略信息，而是压缩成模型更容易理解的形式。

## 11. 是否需要补充额外模块

结论先说：**建议补充，但只补必要的，不要过早复杂化**。

### 11.1 建议补充的内容

- prompt 加载器
- 上下文裁剪器
- 回合摘要器
- 流式事件格式定义
- 主模型与小助手的统一配置读取

### 11.2 暂时不建议补充的内容

- 复杂工具调用链
- 多 Agent 协作
- 向量数据库记忆
- 自动检索增强
- 过重的状态机设计

### 11.3 你现在最需要补什么

如果只考虑当前 demo，最优先的是：

1. prompt 单独文件化
2. 流式输出能力
3. 每局独立上下文
4. 最近轮次窗口
5. 简单摘要预留

## 12. 推荐的 Agent 调用类接口设计

这一节讲的是“逻辑接口”，不是具体代码。

建议你的 Agent 调用类至少提供这些能力：

- 初始化模型
- 加载 prompt
- 构建当前局上下文
- 普通一次性回答
- 流式回答
- 生成摘要

### 12.1 普通一次性回答

适合：

- 后台测试
- 接口不要求流式时
- 快速调试

### 12.2 流式回答

适合：

- 前端聊天体验
- 用户需要即时反馈
- 长回答推理场景

### 12.3 生成摘要

适合：

- 回合很多时
- 上下文太长时
- 需要压缩历史时

## 14. 代码实现时的关键原则

### 14.1 不要把 prompt 写死在类里

prompt 应该来自文件，这样更灵活。

### 14.2 不要把所有历史都喂给模型

上下文是有限的，必须做窗口控制。

### 14.3 不要让每局游戏共享同一份记忆

每个 game_id 都必须独立。

### 14.4 不要把 MongoDB 当成模型记忆直接读取

MongoDB 是存储层，不是实时上下文层。应该先读取，再筛选，再压缩，最后再喂给模型。

### 14.5 不要把小助手和主模型混成一套 prompt

两者职责不同，应该分别管理。

## 19. 详细实现步骤

这一节把你真正动手实现 Agent 调用类的顺序拆开说明。你可以把它当成实际编码路线图。

### 19.1 第一步：先确认 Agent 的输入输出边界

在动手写代码前，先确定这个类到底要接收什么、返回什么。

建议最小输入包括：

- `game_id`
- `question`
- `history`
- `mode`（主模型或小助手）

建议最小输出包括：

- 普通响应文本
- 流式响应事件
- 可选的摘要文本

这样后面 Service 层调用时就不会混乱。

### 19.2 第二步：准备 prompt 文件

先不要写 Agent 类本身，先把 prompt 文件准备好。

建议至少准备：

- 主模型系统提示词
- 小助手系统提示词
- 游戏规则提示词

先把它们写成独立文本文件，再由 Agent 调用类统一读取。

这样做的原因是：

- prompt 以后改动频繁
- 主模型和小助手的语气与任务不同
- 游戏规则可能会单独调整

### 19.3 第三步：设计 prompt 加载逻辑

你需要在 Agent 调用类里提供一个统一的“读取 prompt”能力。

建议读取逻辑满足这几个条件：

- 按文件名加载
- 支持不同模型读取不同文件
- 文件不存在时给出明确错误
- 保持编码一致

这一步完成后，Agent 类就不再依赖硬编码文本。

### 19.4 第四步：确定消息组织顺序

Agent 调用时，消息顺序应该是固定的。

推荐顺序：

1. system prompt
2. 游戏规则摘要
3. 当前局最近历史消息
4. 当前用户问题

这个顺序的好处是：模型会先知道身份和规则，再看上下文，再回答当前问题。

### 19.5 第五步：实现上下文裁剪

不要把整局所有内容直接塞进模型。

建议你先定义一个简单的上下文策略：

- 最近若干轮直接保留
- 更早内容压缩为摘要
- 同一局内保留一致的规则前提

如果一局已经很长，优先保留最近信息，因为它对当前推理最有帮助。

### 19.6 第六步：区分主模型和小助手

这一步很重要，必须从设计上分开。

建议把调用模式至少分为两类：

- 主模型模式：用于猜词推理
- 小助手模式：用于查询、解释和提示

两者的区别应该体现在：

- 配置不同
- prompt 不同
- 输出约束不同
- 业务权限不同

### 19.7 第七步：先实现非流式版本

在做流式输出之前，建议先保证普通一次性回答可用。

原因是：

- 你可以先验证 prompt 是否正确
- 先验证上下文是否拼接正常
- 先验证主模型和小助手是否都能正常返回

非流式版本稳定后，再加流式输出，会更容易定位问题。

### 19.8 第八步：再加流式输出

当普通版本能跑通后，再增加流式能力。

建议实现顺序：

1. 先让模型输出能逐段拿到 token
2. 再把 token 逐段转成 SSE 事件
3. 再让 FastAPI 转发给前端
4. 最后再补结束标记

流式输出的目标不是“更复杂”，而是“更符合用户体验”。

### 19.9 第九步：补摘要能力

当一局游戏历史变长后，你需要有一个摘要能力。

建议摘要使用场景：

- 历史轮次过多
- prompt 太长
- 模型开始忘记前文

摘要的结果不需要保存为完整对话，只需要保留能帮助下一轮推理的核心线索。

### 19.10 第十步：把输出交回 Service 层

Agent 调用类不要自己保存结果。

它应该把结果交给 Service 层，由 Service 层决定：

- 是否写数据库
- 是否更新回合
- 是否结束游戏

这样职责才不会混乱。

### 19.11 代码层面应该怎么组织实现

如果把这一章真正写成代码，Agent 调用类通常会被拆成几个内部职责块，而不是一个大函数直接写到底。

推荐的组织方式是：

```bash
GuessAgent
├── 初始化配置
├── 加载 prompt
├── 构建上下文
├── 选择调用模式
├── 执行普通调用或流式调用
└── 归一化返回结果
```

对应到代码思路上，可以理解为：

1. 初始化时先把主模型或小助手模型的配置读进来
2. 启动时把 prompt 文件读取到内存
3. 每次请求来时，根据 `game_id` 取历史
4. 按窗口截断历史，再拼成模型消息
5. 选择一次性返回或流式返回
6. 把输出统一整理后交给 Service 层

这样做的好处是：

- 代码不会过长
- 每一步都容易调试
- 未来替换模型时影响面小
- 主模型和小助手共用一套“组织方式”，但使用不同参数

### 19.12 代码实现时的核心思路

你可以把 Agent 调用类的实现理解成“三段式”：

#### 第一段：准备输入

这一段做的是“把杂乱信息变成标准输入”。

要处理的内容包括：

- 当前问题
- 历史问答
- 当前局规则
- prompt 文件内容
- 当前模式（主模型或小助手）

这一段的目标是把所有输入统一成模型容易理解的结构。

#### 第二段：调用模型

这一段做的是“把标准输入送进模型”。

如果是普通模式，就一次性拿结果；
如果是流式模式，就逐步接收 token 并往外发。

这一段的核心不是复杂，而是稳定。

#### 第三段：整理输出

这一段做的是“把模型输出变成系统可用结果”。

需要做的事情包括：

- 去掉多余空白
- 统一输出格式
- 区分普通结果和流式事件
- 记录必要日志

### 19.13 为什么要这样拆

这样拆的原因很简单：

- 输入准备、模型调用、输出整理是三种不同性质的工作
- 未来每一部分都可能独立变化
- 你后面要加记忆、加流式、加小助手时，不会把整个类重写

如果不拆，代码很容易变成一坨“读配置 + 拼 prompt + 调模型 + 保存结果”混在一起的大函数。

### 19.14 设计时如何考虑 prompt、上下文和记忆

你在实现时要始终记住三件事：

#### 1. prompt 是规则，不是历史

prompt 里写的是“怎么回答、不能做什么、风格是什么”。

#### 2. 上下文是本局历史，不是永久记忆

同一局游戏里的上下文要保留，但它不是无限增长的。

#### 3. 记忆是压缩后的经验，不是原文堆叠

长期记忆如果以后要做，应该以摘要、总结、经验形式存在，而不是把所有历史原封不动塞回 prompt。

### 19.15 设计时如何考虑主模型和小助手

主模型和小助手在代码层面可以使用同一个调用框架，但不能使用同一套策略文本。

建议思路是：

- 主模型：更强调推理、克制、不能泄露答案
- 小助手：更强调解释、建议、知识辅助

也就是说，**框架可以统一，策略必须分开**。

### 19.16 设计时如何考虑流式输出

流式输出不是最后加一个开关这么简单，它会影响整个数据流。

你在设计时要提前考虑：

- 模型是否能逐段返回内容
- FastAPI 是否能把内容逐段转发
- 前端是否能逐段渲染
- 最终结束标记怎么发

所以流式输出应该在第一版设计里就预留，而不是后补。

### 19.17 设计时如何考虑后续扩展

如果你以后要加更多能力，当前结构要尽量满足这些原则：

- prompt 可替换
- 模型可切换
- 记忆可裁剪
- 输出可流式
- 主模型和小助手可并行存在

这也是为什么这一章不建议把所有逻辑直接写进一个方法里。

### 19.18 实现前的自检问题

在真正编码前，你可以先问自己几个问题：

- 这一局游戏的历史从哪里来？
- 这一轮要保留多少历史？
- 这一轮输出是一次性还是流式？
- 这个调用是主模型还是小助手？
- 输出后谁来保存结果？

如果这些问题都能答清楚，代码实现会顺很多。

## 21. 调试时的推荐顺序

你真正动手时，建议按这个顺序调试：

1. 先打印 prompt 是否能正常读取
2. 再确认历史上下文是否按 game_id 取到
3. 再确认主模型是否能返回结果
4. 再确认小助手是否能返回结果
5. 再开启流式
6. 最后再看摘要是否需要

不要一开始就把所有能力一起打开，否则很难定位问题。

## 22. 验证点清单

完成这一章后，你应该检查下面这些点：

- prompt 文件是否独立存在
- 主模型和小助手的配置是否分开
- 每局游戏的历史是否按 game_id 取出
- 同一局内的历史是否按轮次保留
- 是否有最近窗口和摘要策略
- 普通回答是否先可用
- 流式输出是否能逐步返回

## 23. 衔接下一层

Agent 调用类完成后，下一层应该进入 `game_service`。

因为 Service 层会负责把“模型回答”转成“游戏流程”，包括：

- 开局
- 回合推进
- 回合记录
- 胜负判断
- 结束处理

也就是说，Agent 负责“怎么回答”，Service 负责“什么时候回答、回答后做什么”。
