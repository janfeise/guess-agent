## 8. 项目核心部分：猜词

核心流程：

```txt
用户输入
  -> LLM 识别意图
  -> 程序决定流程
  -> 如果是提问，LLM 生成回答
  -> 如果是猜测，程序判断是否猜中
  -> 如果游戏继续，Agent 决定 ask 或 guess
  -> 如果 Agent 猜测，等待用户确认
  -> 若确认正确，结束游戏
```

- `metadata.phase`：表示当前流程阶段
- `metadata.last_actor`：上一动作是谁执行的
- `metadata.last_intent`：上一轮用户意图
- `metadata.awaiting_judgement`：是否正在等待用户确认 Agent 猜测
- `metadata.agent_confidence`：Agent 猜测置信度
- `metadata.pending_guess`：Agent 当前猜测词

```python
metadata = {
    "phase": "user_turn",
    "last_actor": "system",
    "last_intent": None,
    "awaiting_judgement": False,
    "pending_guess": "",
    "agent_confidence": 0.0,
}
```

llm api 返回格式标准化：

```python
{
  "intent": "question | guess | invalid",
  "normalized_text": "用户输入的标准化文本",
  "guess_word": "如果是猜测则提取出的词",
  "answer": "yes | no | unknown",
  "confidence": 0.0, // 置信度，决定系统是否猜测
  "reason": "简短理由"
}

// 实际标准
{
  "action": "ask | guess",
  "question": "如果 action=ask，则这里放问题",
  "guess_word": "如果 action=guess，则这里放猜测词",
  "confidence": 0.0,
  "reason": "简短理由"
}
```

说明：

- `intent` 是最重要的字段
- `normalized_text` 用于清洗空格、标点和无关前缀
- `guess_word` 只有在 `guess` 时有效
- `answer` 适合在提问场景下由模型生成 yes / no / unknown
- `confidence` 用于记录解析置信度，不是强制字段，但建议保留
- `reason` 只做调试，不要给前端过多展示

## 9. 双词系统隔离

当前猜词系统的核心问题在于没有对“双词机制”进行隔离，导致系统在回答用户问题时始终基于“系统词”（如系统词“刑法”）生成信息，而Agent却将这些回答误当作“用户词”（如“伦敦”）的特征来推理，从而产生严重的信息污染和推理偏移；本质上是不同来源的数据被混用，缺乏明确的归属标识。要解决该问题，必须在双词系统中实现严格隔离：所有回答需明确区分是基于“系统词”还是“用户词”，并在对话历史中标记数据来源，同时确保Agent在推理时只使用与目标词一致的数据，从而保证推理链条的正确性与一致性

方案：过滤，传递给 llm api 的数据进行了过滤

## 10. 轮次控制——轮次锁

情况：原本应当是“你问一题，我问一题”的对等博弈，变成了 Agent 单方面通过排除法压榨用户的信息

缺少轮次控制实现每轮交替对话

在 meatadata 添加字段，为后续轮次锁实现做铺垫

- `next_actor`：下一步允许谁行动，取值建议为 `user` 或 `agent`
- `expected_turn_type`：下一步允许的动作类型，取值建议为 `question`、`answer`、`guess`、`judge`
- `last_actor`：上一次actor
- `last_intent`：上一次的动作
