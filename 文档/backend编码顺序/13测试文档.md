# 13 测试文档

## 1. 文档目标

这一章用于指导你对当前猜词项目进行测试，重点验证下面几条主链路是否正确：

- 创建游戏时，后端是否会调用小助手模型生成系统词
- 开局词 `user_word` 是否会被正确校验和保存
- `system_word_encrypted` 是否会被正确生成并入库
- 游戏创建接口是否能返回预期结果
- 非法输入时，接口是否能返回明确错误

本章以“手工联调 + 最小验证”为主，不依赖复杂自动化测试框架。

## 2. 当前项目的测试范围

当前建议重点测试这些模块：

- [backend/app/api/v1/games.py](../../backend/app/api/v1/games.py)
- [backend/app/services/game_service.py](../../backend/app/services/game_service.py)
- [backend/app/agents/guess_agent.py](../../backend/app/agents/guess_agent.py)
- [backend/app/agents/utils/prompt_loader.py](../../backend/app/agents/utils/prompt_loader.py)
- [backend/app/core/constants.py](../../backend/app/core/constants.py)
- [backend/app/repositories/game_repository.py](../../backend/app/repositories/game_repository.py)

### 2.1 当前开局接口的行为

当前开局接口的请求体包括：

- `owner_id`：可选
- `user_word`：必填
- `difficulty`：可选，默认 `medium`

当前开局接口的返回体包括：

- `game_id`
- `status`
- `round_count`
- `user_word`
- `difficulty`
- `system_word_encrypted`
- `created_at`

### 2.2 当前开局流程

创建游戏时，服务端会：

1. 校验 `user_word`
2. 调用小助手模型生成系统词
3. 校验系统词长度和格式
4. 生成随机系统密码
5. 使用系统密码加密系统词
6. 再用服务端密钥加密系统密码
7. 写入 MongoDB
8. 返回创建结果

## 4. 测试目标

这次测试重点确认以下问题：

- 创建游戏接口是否能成功返回 201
- `user_word` 是否按规则保存
- 系统词是否由小助手生成，而不是写死
- 返回值里是否包含 `system_word_encrypted`
- 数据库里是否写入了 `target_word_encrypted`
- 非法输入是否能触发 400
- 模型异常时是否能触发 500 或业务错误

## 6. 手工测试步骤

### 6.1 正常创建游戏

1. 启动后端服务
2. 打开 `/docs`
3. 找到 `POST /api/v1/games`
4. 发送一个合法请求
5. 检查返回状态码是否为 201
6. 检查返回体字段是否完整
7. 检查 MongoDB 是否插入新记录

### 6.2 检查数据库结果

创建成功后，应该在 `games` 集合里看到类似字段：

- `game_id`
- `owner_id`
- `status`
- `history`
- `summary`
- `round_count`
- `created_at`
- `updated_at`
- `difficulty`
- `user_word`
- `system_password_encrypted`
- `target_word_encrypted`
- `target_word_source`

### 6.3 检查加密结果

你要确认两件事：

- `system_word_encrypted` 在接口返回里存在
- 数据库里保存的是加密后的内容，不是明文系统词

## 7. 正向测试用例

### 用例 1：标准开局词

- 目标：验证正常创建游戏流程
- 请求：

```json
{
  "owner_id": "user-001",
  "user_word": "数据库",
  "difficulty": "medium"
}
```

- 预期结果：
  - 返回 201
  - 返回体包含 `game_id`
  - 返回体包含 `system_word_encrypted`
  - `user_word` 原样返回
  - 数据库成功插入记录

### 用例 2：默认难度

- 目标：验证 `difficulty` 不传时是否默认 `medium`
- 请求：

```json
{
  "owner_id": "user-001",
  "user_word": "数据库"
}
```

- 预期结果：
  - 返回 201
  - 返回体中的 `difficulty` 为 `medium`
  - 数据库中 `difficulty` 也为 `medium`

### 用例 3：无 owner_id

- 目标：验证 `owner_id` 可选
- 请求：

```json
{
  "user_word": "数据库",
  "difficulty": "medium"
}
```

- 预期结果：
  - 返回 201
  - `owner_id` 在数据库中可为空
  - 游戏仍然可创建成功

## 8. 负向测试用例

### 用例 4：user_word 为空

- 目标：验证空开局词是否被拒绝
- 请求：

```json
{
  "owner_id": "user-001",
  "user_word": "",
  "difficulty": "medium"
}
```

- 预期结果：
  - 返回 400
  - 错误信息提示开局词不能为空
  - 不写入数据库

### 用例 5：user_word 过长

- 目标：验证长度校验是否生效
- 请求：

```json
{
  "owner_id": "user-001",
  "user_word": "这是一个超过限制长度的词语",
  "difficulty": "medium"
}
```

- 预期结果：
  - 返回 400
  - 错误信息提示长度超限
  - 不写入数据库

### 用例 6：user_word 含标点或空白

- 目标：验证开局词格式校验
- 请求：

```json
{
  "owner_id": "user-001",
  "user_word": "数 据库",
  "difficulty": "medium"
}
```

- 预期结果：
  - 返回 400
  - 错误信息提示不能包含标点或空白字符
  - 不写入数据库

### 用例 7：小助手模型返回空值

- 目标：验证模型异常或空返回是否会中断创建
- 做法：
  - 临时把 helper 模型配置改成无效地址，或让 `generate_start_word()` 返回空字符串

- 预期结果：
  - 返回 400 或 500
  - 不写入数据库
  - 不会生成空目标词

### 用例 8：MongoDB 不可用

- 目标：验证数据库异常处理
- 做法：
  - 停止 MongoDB
  - 调用创建游戏接口

- 预期结果：
  - 返回 500
  - 后端不应假装创建成功

## 10. 数据库检查建议

创建成功后，建议直接查 MongoDB 的 `games` 集合，确认以下字段：

- `game_id` 是否存在
- `user_word` 是否为输入值
- `system_password_encrypted` 是否存在
- `target_word_encrypted` 是否存在
- `target_word_source` 是否为 `helper_llm`
- `round_count` 是否为 0
- `status` 是否为 `active`

### 10.1 你应该重点看什么

重点不是看明文系统词，而是看：

- 是否保存了密文
- 是否保存了来源信息
- 是否没有把系统词明文直接返回给前端

## 11. 常见问题排查

### 11.1 返回 500

可能原因：

- helper 模型配置错误
- MongoDB 不可用
- prompt 文件路径错误
- `encryption_secret` 配置缺失

### 11.2 返回 400

可能原因：

- `user_word` 为空
- `user_word` 太长
- `user_word` 含标点或空白
- 模型生成结果不符合规则

### 11.3 `/docs` 页面打不开

可能原因：

- FastAPI 服务没有启动
- 端口冲突
- 项目依赖没有安装完整
