# 24 轮次锁执行清单

## 1. 你现在要解决的不是“意图识别”，而是“轮次放行”

当前代码里，`parse_user_intent()` 已经能区分 question、answer、guess，但真正的问题是：**识别出来之后，系统没有再检查当前轮次是否允许这个动作**。

这就是轮次锁要解决的核心：

- 先判断当前轮次允许谁动
- 再判断这次输入属于什么意图
- 最后才决定要不要进入 Agent 逻辑

如果顺序反了，系统就会继续抢跑。

---

## 2. 先修正数据模型：轮次锁字段必须放进 metadata

### 2.1 你现在的代码哪里不一致

在 [backend/app/services/game_service.py](../../backend/app/services/game_service.py) 里，`_build_initial_metadata()` 已经返回了 `phase`、`waiting_answer`、`awaiting_judgement` 等字段。

但 `create_game()` 末尾又额外把这些字段写到了文档根上：

- `next_actor`
- `expected_type`
- `last_actor`
- `last_intent`

这是错位的。因为 [backend/app/repositories/game_repository.py](../../backend/app/repositories/game_repository.py) 的 `update_game_state()` 是把状态写到 `metadata.*`，不是写到根字段。

所以你必须统一成一种写法：**轮次锁字段只放 metadata，不要再往根上补一份。**

### 2.2 你要改成什么

把 `_build_initial_metadata()` 改成包含下面这些字段：

- `next_actor`: `user`
- `expected_turn_type`: `question`
- `last_actor`: `system`
- `last_intent`: `None`

建议初始结构如下：

```python
@staticmethod
def _build_initial_metadata() -> dict:
    return {
        "phase": "user_turn",
        "game_mode": "dual_word_isolated",
        "reasoning_scope": "user_word",
        "next_actor": "user",
        "expected_turn_type": "question",
        "last_actor": "system",
        "last_intent": None,
        "awaiting_judgement": False,
        "waiting_answer": False,
        "pending_guess": "",
        "pending_question": "",
        "agent_confidence": 0.0,
    }
```

然后把 `create_game()` 里那几行根字段删掉，只保留 `metadata=self._build_initial_metadata()`。

## 3. 第二步：新增轮次锁的辅助函数

建议在 `GameService` 里加三个小函数，不要把判断写散在各个分支里。

### 3.1 读取当前轮次状态

```python
@staticmethod
def _get_turn_lock_state(game: dict) -> dict:
    metadata = game.get("metadata", {}) or {}
    return {
        "phase": metadata.get("phase", "user_turn"),
        "next_actor": metadata.get("next_actor", "user"),
        "expected_turn_type": metadata.get("expected_turn_type", "question"),
        "last_actor": metadata.get("last_actor", "system"),
        "last_intent": metadata.get("last_intent"),
    }
```

### 3.2 校验当前输入是否合法

```python
@staticmethod
def _is_turn_allowed(phase: str, intent: str, expected_turn_type: str) -> bool:
    if phase == "waiting_answer":
        return intent == "answer"
    if phase == "awaiting_judgement":
        return intent == "judge"
    if phase == "user_turn":
        return intent in {"question", "guess"}
    return intent == expected_turn_type
```

### 3.3 统一更新轮次锁状态

```python
def _build_turn_lock_update(self, *, phase: str, next_actor: str, expected_turn_type: str, last_actor: str, last_intent: str) -> dict:
    return {
        "phase": phase,
        "next_actor": next_actor,
        "expected_turn_type": expected_turn_type,
        "last_actor": last_actor,
        "last_intent": last_intent,
    }
```

这个辅助函数的目标很简单：所有分支都用同一套字段更新，不再手写一堆散字段。

## 4. 第三步：先改 submit_turn 的总分发逻辑

当前 [backend/app/services/game_service.py](../../backend/app/services/game_service.py) 里的 `submit_turn()` 直接按 phase 和 turn_type 分流，最危险的地方是这里：

```python
if phase == "waiting_answer" or turn_type == "answer":
    ...
```

这句太宽了。它会让任何被标成 `answer` 的输入都直接进入答案分支，哪怕当前轮次根本不该答题。

### 4.1 正确顺序

你应该改成：

1. 先读 `metadata`
2. 再调用 `parse_user_intent()`
3. 再根据 `phase + intent` 做合法性校验
4. 最后才进入具体 handler

### 4.2 推荐分发顺序

```python
metadata = game.get("metadata", {}) or {}
phase = metadata.get("phase", "user_turn")
history = game.get("history", [])

intent_result = await self.agent.parse_user_intent(question, history)
intent = intent_result.get("intent", "invalid")

if phase == "awaiting_judgement":
    if intent not in {"judge", "answer"}:
        return self._reject_turn(game_id, phase, "请先判断系统猜测是否正确")
    return await self._handle_agent_guess_judgement(game, question)

if phase == "waiting_answer":
    if intent != "answer":
        return self._reject_turn(game_id, phase, "请先回答上一轮问题")
    return await self._handle_user_answer(game, question, history, intent_result)

if phase == "user_turn":
    if intent == "guess":
        return await self._handle_user_guess(game, intent_result)
    if intent == "question":
        return await self._handle_user_question(game, question, history, mode)
    if intent == "answer":
        return self._reject_turn(game_id, phase, "现在轮到你提问或猜词，不是回答阶段")

return self._reject_turn(game_id, phase, "当前轮次状态异常")
```

这里最重要的是：**不要再用 turn_type 直接决定分支。** `turn_type` 只能作为辅助信息，不能当作最终真相。

## 5. 第四步：把 user_answer 分支改成“记录完就让位”

这是轮次锁最关键的一步。

### 5.1 现在的错误做法

当前 `_handle_user_answer()` 的流程是：

1. 解析回答是 yes/no/unknown
2. 写入历史
3. 立刻调用 `decide_agent_action()`
4. 如果模型认为可以，就继续 `agent_question` 或 `agent_guess`

这会让系统在用户回答之后继续抢回主动权。

### 5.2 正确做法

回答分支只做三件事：

1. 记录用户回答
2. 更新 metadata 为 `phase=user_turn`、`next_actor=user`、`expected_turn_type=question`
3. 返回“轮到你提问或猜词了”的提示

不要在这个分支里自动调用 `decide_agent_action()`。

### 5.3 推荐改法

```python
raw_history = history + [
    self._make_history_item(
        round_no=game.get("round_count", 0) + 1,
        actor="user",
        turn_type="answer",
        input_text=user_input,
        answer_text=resolved_answer,
        result=(intent_result or {}).get("reason", ""),
        source_word_owner="user",
        visible_to_agent=True,
        visible_to_player=True,
    )
]

await self.repository.update_game_state(
    game["game_id"],
    history=raw_history,
    summary=self.memory_policy.build_summary(raw_history),
    round_count=len(raw_history),
    phase="user_turn",
    awaiting_judgement=False,
    waiting_answer=False,
    pending_guess="",
    pending_question="",
    agent_confidence=0.0,
    last_actor="user",
    last_intent="answer",
    game_mode=game.get("metadata", {}).get("game_mode"),
    reasoning_scope=game.get("metadata", {}).get("reasoning_scope"),
)

return {
    "game_id": game["game_id"],
    "status": game.get("status", "active"),
    "phase": "user_turn",
    "message": "回答已记录，现在轮到你提问或猜词。",
    "history": raw_history,
}
```

### 5.4 你要特别注意

如果你还想保留“系统继续追问”的玩法，那也必须单独做成开关，默认关闭。轮次锁模式下，答案分支结束后就应该停止。

## 6. 第五步：把 user_question 分支也收紧

这个地方很容易漏。

当前 `_handle_user_question()` 在系统回答用户问题之后，会继续把 history 喂给 `decide_agent_action()`，然后自动发出下一轮 `agent_question` 或 `agent_guess`。

如果你的目标是严格轮次锁，这里也要改。

### 6.1 推荐策略

用户问完问题后，系统只负责回答，不负责立即开启下一轮自动追问。

也就是说，处理完 `answer_question()` 后，直接：

1. 写 history
2. 更新 metadata 为 `phase=user_turn`、`next_actor=user`、`expected_turn_type=question`
3. 返回系统回答

不要再调用 `decide_agent_action()`。

### 6.2 如果你必须保留 Agent 建议

如果产品上必须保留“系统给出下一步建议”，那就把它降级成建议，不要自动执行。

例如：

```python
agent_action = await self.agent.decide_agent_action(reasoning_history, summary)

return {
    ...,
    "suggested_action": agent_action,
    "message": "回答已给出，现在轮到你继续提问或猜词。",
}
```

但默认不要把这条建议直接变成下一轮动作。

## 7. 第六步：修复 agent_guess_judgement 分支的状态回跳

当前 `_handle_agent_guess_judgement()` 判断完系统猜测后，已经会回到 `user_turn`，这部分方向是对的，但还要补齐轮次锁字段。

### 7.1 成功猜中时

直接结束游戏，不再走下一轮。

### 7.2 猜错时

必须更新为：

- `phase = user_turn`
- `next_actor = user`
- `expected_turn_type = question`
- `last_actor = user`
- `last_intent = answer`

这样系统不会在用户判错后继续抢跑。

## 8. 第七步：补一个统一的拒绝函数

你会发现轮次锁生效后，很多非法输入都要统一拦截。

建议加一个小函数，避免各分支自己写提示文案：

```python
def _reject_turn(self, game_id: str, phase: str, message: str, history: list[dict] | None = None):
    return {
        "game_id": game_id,
        "status": "active",
        "phase": phase,
        "message": message,
        "history": history or [],
    }
```

这个函数的价值是：

- 所有非法轮次请求都长得一样
- 前端可以稳定识别并提示用户
- 以后扩展“请先回答上一轮问题”之类的提示会更容易

## 9. 第八步：把提示词改成“只能建议，不能越权”

轮次锁不是只改代码，prompt 也要配合。

### 9.1 `turn_router.txt`

[backend/app/agents/prompt/turn_router.txt](../../backend/app/agents/prompt/turn_router.txt) 要强调两件事：

- 只负责分类，不负责调度
- 当前轮次不允许的动作，必须由后端拦截

### 9.2 `agent_decision.txt`

[backend/app/agents/prompt/agent_decision.txt](../../backend/app/agents/prompt/agent_decision.txt) 要加一条硬规则：

- 当当前轮次是 `user_turn` 时，不要自动继续生成 `agent_question`
- 当当前轮次是 `waiting_answer` 时，不要把没回答完的状态误判成可继续推进

### 9.3 `guess_system.txt`

[backend/app/agents/prompt/guess_system.txt](../../backend/app/agents/prompt/guess_system.txt) 主要防止系统回答里把“下一步动作”写进去。回答只给本轮答案，不要暗示后续追问。

## 10. 第九步：检查 repository 更新字段是否一致

[backend/app/repositories/game_repository.py](../../backend/app/repositories/game_repository.py) 的 `update_game_state()` 目前已经在更新这些字段：

- `metadata.phase`
- `metadata.awaiting_judgement`
- `metadata.waiting_answer`
- `metadata.pending_guess`
- `metadata.pending_question`
- `metadata.agent_confidence`
- `metadata.last_actor`
- `metadata.last_intent`
- `metadata.game_mode`
- `metadata.reasoning_scope`

所以你现在只需要把下面两个字段补进去：

- `metadata.next_actor`
- `metadata.expected_turn_type`

建议把 `update_game_state()` 的参数也一并加上，避免以后每次都手写 `$set`。

## 11. 最小改造顺序

如果你要按最稳的顺序执行，建议按这个顺序来：

1. 先把 `next_actor` 和 `expected_turn_type` 放进 `_build_initial_metadata()`。
2. 删掉 `create_game()` 里重复写到根上的轮次锁字段。
3. 扩展 `GameRepository.update_game_state()`，让它支持 `metadata.next_actor` 和 `metadata.expected_turn_type`。
4. 在 `submit_turn()` 里先做 phase + intent 校验，再进 handler。
5. 把 `_handle_user_answer()` 改成“记录完就回到 user_turn”。
6. 把 `_handle_user_question()` 改成“只回答，不自动发下一轮动作”。
7. 补 prompt 约束，防止 LLM 自己抢跑。

---

## 12. 验收清单

改完以后，逐条确认下面这些结果：

- `metadata` 里能看到 `next_actor` 和 `expected_turn_type`
- 用户回答完系统问题后，不会马上进入下一轮 `agent_question`
- 当前轮次是 `user_turn` 时，用户输入“是 / 不是”会被拒绝或提示改为提问
- 当前轮次是 `waiting_answer` 时，只有 answer 能通过
- 当前轮次是 `awaiting_judgement` 时，只允许 judge
- `update_game_state()` 每次落库后，轮次锁状态不会丢失

## 13. 你可以直接照着改的核心判断

下面这段逻辑是整个轮次锁的骨架：

```python
metadata = game.get("metadata", {}) or {}
phase = metadata.get("phase", "user_turn")

intent_result = await self.agent.parse_user_intent(question, history)
intent = intent_result.get("intent", "invalid")

if phase == "waiting_answer" and intent != "answer":
    return self._reject_turn(game_id, phase, "请先回答上一轮问题")

if phase == "awaiting_judgement" and intent not in {"judge", "answer"}:
    return self._reject_turn(game_id, phase, "请先判断系统猜测是否正确")

if phase == "user_turn" and intent == "answer":
    return self._reject_turn(game_id, phase, "现在轮到你提问或猜词，不是回答阶段")
```

这三段判断不要省。它们就是轮次锁。

---

## 14. 最终原则

你现在实现轮次锁，记住一句话就够了：

**意图识别只是分类，phase 和 next_actor 才是放行规则。**

只要把这两层分开，系统就不会再连续追问。
