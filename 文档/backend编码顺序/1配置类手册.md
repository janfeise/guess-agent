# 1配置类手册

## 1. 文档目的

这一节的目标是指导你先把后端项目的配置入口设计清楚，再开始写业务代码。配置类是整个后端的基础，它决定了程序如何读取环境变量、如何区分本地与容器环境、以及后续数据库和模型调用要从哪里取参数。

这份手册只讲配置类应该怎么设计，不写任何代码。你可以把它当成实现前的检查清单。

## 2. 为什么先做配置类

配置类应该排在最前面，原因很简单：

- 后端后面所有模块都要依赖配置
- 数据库地址、模型地址、密钥都必须先统一管理
- 如果配置设计混乱，后续每个模块都会重复读取环境变量
- demo 阶段最容易出问题的部分往往就是环境配置

先把配置类定好，后面的 MongoDB、LangChain、加密模块和 Docker 才能稳定衔接。

## 3. 配置类的职责边界

配置类只负责一件事：把外部环境中的参数，转换成程序内部可以稳定使用的配置对象。

它不应该负责：

- 业务判断
- 数据库操作
- 模型调用
- 日志格式化
- 路由处理

如果某个逻辑不是“读取配置”，就不要放进配置类。

## 4. 这个阶段需要哪些配置

demo 阶段建议先准备以下几类配置：

### 4.1 服务基础配置

- 服务名称
- 运行主机
- 运行端口
- 调试模式开关

这些配置主要用于 FastAPI 启动和本地开发。

### 4.2 模型调用配置

demo 阶段建议把模型配置分成两组：

- 主模型配置
  - API 密钥
  - 接口地址
  - 模型名称
  - 温度参数
- 小助手模型配置
  - API 密钥
  - 接口地址
  - 模型名称
  - 温度参数

这样主模型和小助手就可以各自独立调整，不会互相影响。

### 4.3 数据库配置

- MongoDB 连接地址
- 数据库名称
- 集合相关基础命名约定

这部分要同时兼容本地开发和 Docker 容器环境。

### 4.4 安全配置

- 加密密钥
- 答案加密所需的基础种子值

后续如果你扩展登录、权限或更复杂的加密策略，这部分还会继续扩展。

## 5. 配置类应该满足的要求

### 5.1 一致性

同一类配置应该只从一个地方读取，避免在不同文件里手动拼接同一个值。

### 5.2 可替换性

本地开发和 Docker 运行应该只改环境变量，不改业务代码。

### 5.3 可验证性

启动后，你应该能很快判断配置是否正确，比如服务名、端口、数据库连接串、模型地址是否已经读到。

### 5.4 默认值可控

适合 demo 的配置可以有默认值，但敏感配置不建议依赖默认值。

## 6. 配置文件之间的关系

你这个项目里，配置相关内容最好分成三层理解：

### 6.1 环境变量层

这是最外层，来自系统、Docker Compose 或本地 .env 文件。

### 6.2 配置对象层

这是程序内部统一使用的配置入口，后面的模块都应该通过它读取参数。

### 6.3 使用层

数据库连接、Agent 调用、加密工具、启动入口，都是配置对象的使用者。

这个层次划分清楚后，后续扩展会容易很多。

## 7. 设计建议

### 7.1 命名要稳定

配置项名称一旦定下来，后面尽量不要频繁改。尤其是数据库地址、模型名称和密钥相关字段。

### 7.2 统一大小写风格

环境变量建议保持统一风格，避免同一项目里混用多种命名方式。

### 7.3 敏感信息不要写死

像模型密钥、数据库密码、加密密钥这类信息都应该放在外部环境里，不要直接写进代码。

### 7.4 先小后大

demo 阶段只保留最必要的配置项，不要一开始就把未来可能用到的参数都写进去。

## 8. 建议的配置内容清单

你至少要考虑以下内容是否存在：

- 服务名
- 运行地址
- 运行端口
- 是否开启调试
- 主模型 API Key
- 主模型服务地址
- 主模型名称
- 主模型温度
- 小助手模型 API Key
- 小助手模型服务地址
- 小助手模型名称
- 小助手模型温度
- MongoDB 连接串
- MongoDB 数据库名
- 加密密钥

如果后面要做更完整的开发，还可以继续补：

- 日志级别
- CORS 允许来源
- 请求超时时间
- 默认回合数限制
- 默认 prompt 路径

## 9. 本地与容器的差异

配置类最容易出问题的地方，就是本地和 Docker 环境不一致。

你需要提前想清楚：

- 本地开发时数据库地址是什么
- 容器内数据库地址是什么
- 模型密钥从哪里注入
- 是否允许本地 .env 覆盖容器配置

建议原则是：代码不改，只改环境变量。

## 10. 配置类的验收标准

当你完成配置类后，应该能确认下面几点：

- 后端启动时能读取到服务名和端口
- 数据库连接信息是从统一入口读取的
- 模型相关参数可以被后续调用层使用
- 加密相关参数可以被后续工具层使用
- 本地与 Docker 下的配置方式一致

如果以上任何一点做不到，说明配置类还不够稳定。

## 11. 这一步完成后再做什么

配置类完成后，建议下一步立刻写 MongoDB 连接文档，再往后是健康检查接口和启动入口。不要跳过配置直接写业务逻辑，因为后面返工成本会更高。

## 12. 本章开发检查清单

- [ ] 已明确服务基础配置
- [ ] 已明确模型调用配置
- [ ] 已明确数据库配置
- [ ] 已明确安全配置
- [ ] 已区分本地和容器环境
- [ ] 已避免敏感信息写死
- [ ] 已限制配置范围在 demo 需要的最小集合
- [ ] 已准备好后续给数据库和 Agent 使用的统一入口

## 13. 结论

配置类不是一个“可有可无的小模块”，而是整个后端 demo 的起点。只要这里设计清楚，后面的接口、数据库、LangChain 和 Docker 才能按同一套参数体系工作。

你可以把这一章的目标理解成一句话：先让后端知道自己从哪里拿配置，再让其他模块去消费这些配置。

## 14. 配置类详细实现方案

这一节开始进入可直接实现的内容。建议你把配置类单独放在后端的核心配置目录中，由它统一负责读取环境变量，并向外提供一个稳定的配置对象。

### 14.1 文件位置建议

建议把配置类放在下面这个位置：

```bash
backend/
└── app/
	└── core/
		└── config.py
```

这样后续数据库连接、启动入口、Agent 调用层都可以直接引用这一个配置入口。

### 14.2 依赖建议

配置类建议配合以下 Python 包使用：

- pydantic-settings
- python-dotenv

这两个包可以帮助你更稳地从 `.env` 文件和系统环境变量中读取参数。

### 14.3 配置类职责拆分

建议把配置类拆成两个层次：

1. `Settings`：真正承载配置字段的类
2. `get_settings()`：对外提供单例式读取入口

这样做的好处是：

- 其他模块只需要调用一个入口
- 避免重复实例化配置对象
- 方便测试和替换

### 14.4 推荐实现代码

下面这份代码就是你可以直接放进 `backend/app/core/config.py` 的参考实现：

```python
from functools import lru_cache

from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
	model_config = SettingsConfigDict(
		env_file=".env",
		env_file_encoding="utf-8",
		extra="ignore",
	)

	app_name: str = "guess-agent-demo"
	app_host: str = "0.0.0.0"
	app_port: int = 8000
	debug: bool = False

	agent_model_api_key: str = ""
	agent_model_base_url: str = "https://api.siliconflow.cn/v1"
	agent_model_name: str = "deepseek-ai/DeepSeek-V3.2"
	agent_model_temperature: float = 0.2

	helper_model_api_key: str = ""
	helper_model_base_url: str = ""
	helper_model_name: str = "Qwen2.5-7B-Instruct"
	helper_model_temperature: float = 0.5

	mongo_uri: str = "mongodb://root:rootpassword@mongo:27017/guess_agent?authSource=admin"
	mongo_db_name: str = "guess_agent"

	encryption_secret: str = "guess-agent-demo-secret"
	cors_origins: list[str] = ["*"]


@lru_cache
def get_settings() -> Settings:
	return Settings()
```

### 14.5 这段代码的设计意图

这份实现里有几个关键点：

- `BaseSettings` 负责从环境变量和 `.env` 中读取配置
- `SettingsConfigDict` 用来指定 `.env` 文件位置和编码
- `extra="ignore"` 可以避免多余环境变量影响启动
- `get_settings()` 用缓存方式保证同一进程里只创建一次配置对象

### 14.6 为什么要有默认值

默认值的作用是让 demo 阶段更容易启动。

但需要注意：

- 服务名、端口这类配置可以有默认值
- 数据库地址、API Key、密钥这类敏感配置不要依赖默认值
- 默认值只应该帮助开发，不应该掩盖真实配置错误

### 14.7 为什么建议使用缓存

配置对象通常是只读的。既然它不会频繁变化，就没有必要在每次调用时重新创建。

使用缓存后，你可以获得以下好处：

- 启动更稳定
- 读取更统一
- 测试更容易控制

## 15. 环境变量示例

建议你在后端根目录准备一个 `.env.example`，内容可以按下面的方式组织：

```env
APP_NAME=guess-agent-demo
APP_HOST=0.0.0.0
APP_PORT=8000
DEBUG=false

AGENT_MODEL_API_KEY=replace_with_your_agent_key
AGENT_MODEL_BASE_URL=https://api.siliconflow.cn/v1
AGENT_MODEL_NAME=deepseek-ai/DeepSeek-V3.2
AGENT_MODEL_TEMPERATURE=0.2

HELPER_MODEL_API_KEY=replace_with_your_helper_key
HELPER_MODEL_BASE_URL=replace_with_your_helper_base_url
HELPER_MODEL_NAME=Qwen2.5-7B-Instruct
HELPER_MODEL_TEMPERATURE=0.5

MONGO_URI=mongodb://root:rootpassword@mongo:27017/guess_agent?authSource=admin
MONGO_DB_NAME=guess_agent

ENCRYPTION_SECRET=guess-agent-demo-secret
CORS_ORIGINS=*
```

### 15.1 字段说明

- `APP_NAME`：服务名，用于日志和健康检查返回
- `APP_HOST`：服务绑定地址，容器里一般用 `0.0.0.0`
- `APP_PORT`：服务端口
- `DEBUG`：是否开启调试模式
- `AGENT_MODEL_API_KEY`：主模型调用密钥
- `AGENT_MODEL_BASE_URL`：主模型接口地址
- `AGENT_MODEL_NAME`：主模型名称
- `AGENT_MODEL_TEMPERATURE`：主模型生成温度
- `HELPER_MODEL_API_KEY`：小助手模型调用密钥
- `HELPER_MODEL_BASE_URL`：小助手模型接口地址
- `HELPER_MODEL_NAME`：小助手模型名称
- `HELPER_MODEL_TEMPERATURE`：小助手模型生成温度
- `MONGO_URI`：MongoDB 连接串
- `MONGO_DB_NAME`：MongoDB 数据库名
- `ENCRYPTION_SECRET`：答案加密密钥
- `CORS_ORIGINS`：允许跨域的来源配置

### 15.2 本地开发和 Docker 的区别

本地开发时，你可以把 `.env.example` 复制成 `.env` 再修改。

Docker 环境下，建议由 `docker-compose.yml` 注入环境变量，避免容器里直接依赖本地文件。

## 16. 配置类在项目中的使用方式

配置类不是单独存在的，它会被多个模块引用。

### 16.1 启动入口使用方式

启动入口应该先读取配置，再决定应用名称、端口、调试模式等基础行为。

### 16.2 数据库连接使用方式

MongoDB 连接模块应该从配置对象中读取 `mongo_uri` 和 `mongo_db_name`，不要在数据库模块里写死地址。

### 16.3 Agent 使用方式

Agent 层应该从配置对象中读取主模型配置，至少包括 `agent_model_api_key`、`agent_model_base_url`、`agent_model_name` 和 `agent_model_temperature`。

小助手层应该从配置对象中读取小助手模型配置，至少包括 `helper_model_api_key`、`helper_model_base_url`、`helper_model_name` 和 `helper_model_temperature`。

### 16.4 加密工具使用方式

加密工具应该从配置对象中读取 `encryption_secret`，这样后续统一更换密钥会更简单。

## 17. 编码时的注意事项

### 17.1 不要混乱地读取配置

所有模块都应该尽量通过统一配置入口读取，不要一会儿读环境变量，一会儿写死常量。

### 17.2 不要把敏感信息提交到仓库

`.env` 文件不要直接提交。仓库里只保留 `.env.example`。

### 17.3 不要过早增加无关配置

demo 阶段先把最小字段写清楚，等后面真的需要再补。

### 17.4 不要让配置类负责业务判断

如果你发现自己在配置类里开始写“如果数据库连不上就做什么”的逻辑，那已经越界了。

## 18. 这一章写完后的验收方式

你可以用下面几个标准判断配置类是否写对了：

- 能从同一个入口拿到应用名、端口、模型参数和数据库参数
- `.env` 文件修改后，程序能正确读取到新值
- Docker 和本地开发使用的是同一套字段名
- 其他模块不需要自己拼接环境变量

## 19. 本章最终建议

如果你准备马上动手，建议你的顺序是：

1. 先创建配置类文件
2. 再准备 `.env.example`
3. 然后写配置读取入口
4. 最后在启动入口和数据库模块里引用它

只要这一章完成，后面的 MongoDB 连接、健康检查接口和 Agent 初始化都会顺很多。
