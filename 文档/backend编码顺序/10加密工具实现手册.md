# 10 加密工具实现手册

## 1. 文档目标

这一章的目标是指导你完成后端的加密工具层，也就是 `backend/app/core/security.py`。

这个模块是本阶段的安全底座，负责把知识词、答案或其他敏感信息加密后再写入 MongoDB，避免明文长期存在于数据库中。

写完这一章后，你应该能够明确下面这些事情：

- 加密工具层应该放在哪一层
- 为什么要把加密独立成一个 core 模块
- 加密和解密应该提供哪些函数
- 如何从配置里读取密钥
- 如何在 Service 层调用加密工具
- 如何为后续的知识词开局流程提供安全能力

这一章只关注加密工具本身，不展开完整业务流程。

---

## 2. 加密工具层在架构中的位置

推荐调用链路如下：

```text
API 层 -> Service 层 -> core/security.py -> Repository 层 -> MongoDB
```

### 2.1 为什么要把加密放在 core 层

加密不是业务逻辑，也不是数据库逻辑，而是一个基础能力。

把它放在 `core/` 目录下的好处是：

- 所有业务层都可以复用
- 以后更换算法时，只需要改一个地方
- 加密逻辑不会散落在 API、Service、Repository 中
- 更容易测试和维护

### 2.2 加密工具层负责什么

加密工具层负责：

- 生成密钥
- 将明文加密成密文
- 将密文解密回明文
- 提供统一的异常处理
- 可选地生成加密元数据

### 2.3 加密工具层不负责什么

加密工具层不应该负责：

- 不判断游戏是否结束
- 不拼接游戏历史
- 不组织 prompt
- 不直接操作 MongoDB
- 不决定什么时候开局或结算

它只关心“给我明文和密钥，我输出密文”，或者“给我密文和密钥，我输出明文”。

---

## 3. 为什么这一章很重要

在这个项目里，答案不可见是一个核心规则。

如果没有统一加密工具，会出现这些问题：

- Service 里加密逻辑到处重复
- Repository 里可能混入明文数据
- 开局流程和验证流程的密钥算法不一致
- 后续排障会很麻烦

所以这一章不是附属能力，而是整条“知识词开局 + 回合推理 + 结果验证”链路的基础。

---

## 4. 本章需要解决的核心问题

### 4.1 需要加密什么

当前阶段建议至少考虑加密这些信息：

- 用户知识词
- Agent 知识词
- 用户答案
- Agent 答案

如果后续还有敏感字段，也可以继续扩展。

### 4.2 为什么要加密知识词

因为知识词本身就是谜底或核心目标，不能明文保存在数据库里。

如果明文直接落库，会破坏游戏公平性，也会让后续验证和回放逻辑失去意义。

### 4.3 为什么要统一算法

统一算法可以保证：

- 开局写入时加密
- 结算验证时解密
- 同一条数据始终能被还原

## 5. 推荐依赖

### 5.1 加密库

建议使用 `cryptography` 包中的 `Fernet`。

理由：

- API 简单
- 对称加密能力足够
- 适合 demo 和中小型业务
- 可通过固定密钥快速接入

### 5.2 依赖新增

在 `backend/requirements.txt` 中建议加入：

```text
cryptography>=41.0.0
```

### 5.3 配置项

当前项目的 `backend/app/core/config.py` 已经包含：

- `encryption_secret`

这个配置项会作为服务端加密的基础密钥来源。

## 7. 加密模块职责边界

### 7.1 security.py 应该做的事情

`security.py` 应该负责：

- 口令派生密钥
- 明文加密
- 密文解密
- 提供统一异常
- 提供密钥生成辅助函数

### 7.2 security.py 不应该做的事情

`security.py` 不应该负责：

- 不读取业务数据库
- 不处理接口请求
- 不决定游戏流程
- 不拼接业务文档

### 7.3 核心原则

一句话概括：

**加密层只关心数据如何安全转换，不关心数据为什么被加密。**

---

## 8. 推荐实现思路

### 8.1 基本算法

当前阶段建议采用：

- SHA-256 进行口令摘要
- Base64 URL-safe 编码
- Fernet 进行对称加密

### 8.2 为什么这样设计

这个方案的优点是：

- 逻辑简单
- 实现稳定
- 适合 demo 阶段
- 后续可以继续替换成更复杂的策略

### 8.3 统一密钥来源

建议把加密密钥来源固定为：

- `settings.encryption_secret`

如果后续需要支持用户口令，也可以在这个基础上再派生一次。

## 9. 详细代码实现参考

下面给出 `security.py` 的建议实现骨架。

### 9.1 文件骨架

```python
# backend/app/core/security.py

from __future__ import annotations

import base64
import hashlib
from dataclasses import dataclass

from cryptography.fernet import Fernet, InvalidToken

from app.core.config import get_settings
```

### 9.2 密钥派生函数

```python
def derive_key(secret: str) -> bytes:
    """把任意字符串口令派生成 Fernet 可用密钥。"""
    digest = hashlib.sha256(secret.encode("utf-8")).digest()
    return base64.urlsafe_b64encode(digest)
```

#### 设计说明

- 输入是字符串口令
- 输出是 Fernet 可接受的密钥字节串
- 这样可以统一服务端密钥和用户口令的处理方式

### 9.3 加密函数

```python
def encrypt_text(plain_text: str, secret: str) -> str:
    """使用 secret 加密明文，返回密文字符串。"""
    key = derive_key(secret)
    fernet = Fernet(key)
    token = fernet.encrypt(plain_text.encode("utf-8"))
    return token.decode("utf-8")
```

#### 设计说明

- `plain_text` 是待加密内容
- `secret` 是密钥来源
- 返回值是可直接存入数据库的字符串

### 9.4 解密函数

```python
def decrypt_text(cipher_text: str, secret: str) -> str:
    """使用 secret 解密密文，返回明文字符串。"""
    key = derive_key(secret)
    fernet = Fernet(key)
    plain_text = fernet.decrypt(cipher_text.encode("utf-8"))
    return plain_text.decode("utf-8")
```

#### 设计说明

- 传入密文和密钥
- 返回原始明文
- 如果密文无效或密钥不匹配，会抛异常

### 9.5 异常包装

如果你希望 Service 层处理错误更统一，可以加一个轻量异常包装：

```python
@dataclass
class EncryptionError(Exception):
    message: str

    def __str__(self) -> str:
        return self.message
```

也可以不单独包装，直接使用 `ValueError` 或 `InvalidToken`，但建议在 Service 层前做一层转换，方便对外返回统一错误信息。

### 9.6 配置驱动的默认加密函数

如果你想让调用方更少传参数，可以再补一层默认方法：

```python
def encrypt_with_settings(plain_text: str) -> str:
    settings = get_settings()
    return encrypt_text(plain_text, settings.encryption_secret)


def decrypt_with_settings(cipher_text: str) -> str:
    settings = get_settings()
    return decrypt_text(cipher_text, settings.encryption_secret)
```

#### 设计说明

- 适合服务端统一加密的场景
- 调用者不用每次都传 secret
- 后续更换配置密钥时更方便

## 10. 推荐的扩展实现

如果后续你要支持“用户口令参与加密”，可以再扩展一个派生入口。

### 10.1 用户口令参与派生

```python
def build_secret_with_user_password(base_secret: str, user_password: str | None) -> str:
    if not user_password:
        return base_secret
    return f"{base_secret}:{user_password}"
```

### 10.2 使用方式

```python
secret = build_secret_with_user_password(settings.encryption_secret, user_password)
encrypted_text = encrypt_text(plain_text, secret)
```

### 10.3 设计考虑

- 简化版只使用服务端 secret
- 增强版可以让用户口令参与密钥派生
- 是否启用由业务流程决定

## 11. 与 Service 层的配合方式

### 11.1 开局时如何调用

在 `game_service.py` 里，开局时应在写数据库前先调用加密工具：

```python
from app.core.security import encrypt_text

user_answer_encrypted = encrypt_text(user_knowledge_word, encryption_secret)
agent_answer_encrypted = encrypt_text(agent_knowledge_word, encryption_secret)
```

### 11.2 为什么要在 Service 里调用

因为加密属于业务流程中的一步，但它的实现必须由 core/security.py 统一提供。

也就是说：

- `security.py` 负责算法
- `game_service.py` 负责什么时候调用算法

### 11.3 为什么不能在 Repository 里加密

Repository 的职责是写数据，不是决定数据如何生成。

如果把加密放到 Repository，会导致业务编排和存储逻辑混在一起。

---

## 12. 与数据库的关系

### 12.1 应该存什么

建议数据库中存：

- 密文
- 加密元数据
- 以及必要的业务状态字段

### 12.2 不应该存什么

不建议在主文档中长期保存：

- 明文知识词
- 明文答案
- 临时调试信息

### 12.3 为什么这样更稳

因为这样可以保证：

- 即使数据库被查看，也只能拿到密文
- 只有通过正确密钥才能恢复明文
- 后续验证流程更可控

---

## 13. 调试和测试建议

### 13.1 先做最小自测

在真正接入业务前，建议先单独验证这三个场景：

1. 明文可以被加密
2. 密文可以被正确解密
3. 错误密钥会解密失败

### 13.2 推荐测试数据

可以先用这些示例：

- 明文：`反射`
- 密钥：`guess-agent-secret`

### 13.3 简单测试代码

```python
from app.core.security import encrypt_text, decrypt_text

plain_text = "反射"
secret = "guess-agent-secret"

cipher_text = encrypt_text(plain_text, secret)
restored_text = decrypt_text(cipher_text, secret)

assert restored_text == plain_text
```

### 13.4 常见错误

- 密钥为空
- 密文被截断
- secret 和 decrypt 时不一致
- 依赖没有安装 `cryptography`

## 14. 建议的验收标准

这一章完成后，至少应该满足以下条件：

- `core/security.py` 可以独立导入
- `encrypt_text()` 可以把明文转换为密文
- `decrypt_text()` 可以把密文还原为明文
- 加密结果可以直接存储到 MongoDB
- 后续 Service 可以直接调用这个模块

---

## 15. 实现顺序建议

建议你把加密工具按下面顺序实现：

1. 先补依赖 `cryptography`
2. 再创建 `core/security.py`
3. 再写 `derive_key()`
4. 再写 `encrypt_text()` 和 `decrypt_text()`
5. 再做最小自测
6. 最后再接入 `game_service.py`

这个顺序的好处是：

- 先验证最核心的安全能力
- 再让 Service 层调用
- 避免一开始就把业务流程和加密算法混在一起

## 17. 本章结论

加密工具层是本阶段的安全底座。

它的目标很明确：把明文安全、稳定地转成密文，并在需要时还原回来。

只要这一层写稳了，后面的知识词开局、回合问答、最终验证都会更容易实现。
