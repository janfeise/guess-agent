# 23 轮次控制实现手册

## 1. 文档目标

这份手册专门解决一个核心问题：**系统在用户回答之后，不应该继续自动追问，而应该把控制权交还给用户**。

从你给出的日志可以直接看出来，当前流程已经从“轮流出题、轮流回答”变成了“系统连续提问、用户连续回答”。这会让猜词游戏退化成审讯式对话，用户只是在被动接招，而不是参与博弈。

这份手册的目标不是单纯修 prompt，而是把**轮次控制**做成后端的明确规则。

---

## 2. 问题复盘

从日志看，最明显的问题有两个：

1. 用户在第 3 轮、第 5 轮、第 7 轮……只是正常回答系统问题，但系统没有停下来，而是继续在下一轮自动生成新的 `agent_question`。
2. 第 21 轮用户已经明确回答“正确”之后，系统仍然继续向前推进到第 22 轮 `agent_guess`，说明当前流程没有一个真正的“轮次锁”。

换句话说，系统现在只是在“识别用户输入是什么”，但没有在“当前轮次允许谁行动”这件事上做硬约束。

---

## 3. 核心原则

轮次控制要遵守三条硬规则：

1. **每一轮只能有一个主动方。**
2. **上一轮结束后，必须显式切换 next_actor。**
3. **如果当前轮次不允许某个动作，就直接拒绝，而不是交给 LLM 自己猜。**

你可以把它理解成一个最小状态机，而不是自由对话。

---

## 4. 推荐状态设计

建议在游戏 metadata 中补两个字段：

- `next_actor`：下一步允许谁行动，取值建议为 `user` 或 `agent`
- `expected_turn_type`：下一步允许的动作类型，取值建议为 `question`、`answer`、`guess`、`judge`

现有的 `phase` 继续保留，但它只负责表达当前业务阶段，不要单独承担轮次控制职责。

建议的初始状态如下：

```json
{
  "phase": "user_turn",
  "next_actor": "user",
  "expected_turn_type": "question"
}
```

如果当前系统正在等用户回答系统问题，则应该是：

```json
{
  "phase": "awaiting_answer",
  "next_actor": "user",
  "expected_turn_type": "answer"
}
```

如果系统已经给出猜测，等待用户判断，则应该是：

```json
{
  "phase": "awaiting_judgement",
  "next_actor": "user",
  "expected_turn_type": "judge"
}
```

---

## 5. 状态机规则

建议把轮次控制按下面的方式固定下来。

| 当前 phase           | 允许输入              | 处理结果               | 下一状态                 |
| -------------------- | --------------------- | ---------------------- | ------------------------ |
| `user_turn`          | `question` 或 `guess` | 系统回答或判定         | `user_turn` / `finished` |
| `awaiting_answer`    | `answer`              | 记录用户回答并结束本轮 | `user_turn`              |
| `awaiting_judgement` | `judge`               | 判断系统猜测是否正确   | `user_turn` / `finished` |

这里最关键的一条是：

- **用户回答完系统问题之后，必须回到 `user_turn`，不能直接进入下一轮 `agent_question`。**

这就是轮次锁真正要做的事。

## 7. 第一层：入口校验

### 7.1 在 `submit_turn()` 里先看当前轮次

[backend/app/api/v1/submit_turn.py](../../backend/app/api/v1/submit_turn.py) 现在只是把 `turn_type` 和文本交给 service，不负责轮次判断。

建议先在 service 层读取 metadata，再根据 `next_actor` 和 `expected_turn_type` 判断这次输入是否允许进入流程。

建议校验逻辑如下：

```python
if metadata.get("next_actor") == "user":
    if intent not in {"question", "guess"}:
        return {
            "message": "当前轮到你提问或猜词了，请不要先回答系统问题。",
            ...
        }

if metadata.get("next_actor") == "agent":
    if intent not in {"answer"}:
        return {
            "message": "请先回答上一轮系统问题。",
            ...
        }
```

这里的意思很简单：**先看轮次，再看意图。** 不要反过来。

### 7.2 不要让 `parse_user_intent()` 单独决定一切

`parse_user_intent()` 的作用只是分类，不是调度器。

也就是说，哪怕 LLM 说用户输入像 `answer`，如果当前轮次不允许 `answer`，程序也必须拒绝。

---

## 8. 第二层：服务层轮次锁

### 8.1 重点改 `_handle_user_answer()`

现在的问题通常就出在这里：

1. 用户回答系统问题
2. service 记录 history
3. service 立即调用 `decide_agent_action()`
4. 系统马上再问一个新问题

这就是“连续追问”的根因。

正确做法是：

1. 记录用户回答
2. 更新状态为 `user_turn`
3. 明确把 `next_actor` 改成 `user`
4. 返回一条提示，告诉用户现在轮到自己提问或猜词
5. **不要在这里自动触发下一轮 `agent_question`**

### 8.2 建议的返回语义

用户回答完成后，返回文案建议类似：

> 回答已记录，现在轮到你向我提问，或者直接猜答案。

这句提示的目的不是装饰，而是明确告诉前端：**轮次已经切到用户。**

### 8.3 `agent_guess` 只在明确时机触发

如果你还想保留系统猜词能力，`agent_guess` 也不要放在“用户回答之后自动触发”的路径里。

建议只在下面两种情况下触发：

1. 当前轮次明确允许系统行动
2. 后端已经判断本轮可以结束并切换到 `awaiting_judgement`

否则就会出现“用户刚回答完，系统立刻猜答案”的抢跑。

---

## 9. 第三层：Agent 层只做建议，不做调度

### 9.1 `decide_agent_action()` 不能替代状态机

[backend/app/agents/guess_agent_fixed.py](../../backend/app/agents/guess_agent_fixed.py) 里的 `decide_agent_action()` 只能输出“建议动作”，不能直接决定轮次是否放行。

也就是说，即便 LLM 返回了 `ask` 或 `guess`，后端也必须再过一层状态校验。

### 9.2 `turn_router.txt` 要支持轮次约束

[backend/app/agents/prompt/turn_router.txt](../../backend/app/agents/prompt/turn_router.txt) 现在只负责识别用户输入是 `question`、`guess`、`answer` 还是 `invalid`。

建议补一条规则：

- 如果当前是用户轮次，就不要把“是 / 不是 / 对 / 不对”这类短句强行当作有效回答
- 如果当前是等待回答轮次，就不要把普通提问当作有效输入

这条规则的核心是：**意图识别要结合轮次上下文。**

### 9.3 `agent_decision.txt` 里明确写“不得越权”

[backend/app/agents/prompt/agent_decision.txt](../../backend/app/agents/prompt/agent_decision.txt) 需要明确告诉模型：

- 你只能根据过滤后的历史做决策
- 你不能无视当前轮次，擅自继续出题
- 当 `next_actor = user` 时，模型只能给出等待用户行动的建议，不能硬生成下一题

如果 prompt 不写清楚，模型会默认继续补充问题，轮次锁就失效了。

---

## 10. 推荐改造顺序

如果你要按最稳的方式实施，建议按下面顺序做：

### 第一步：先加状态字段

在 [backend/app/services/game_service.py](../../backend/app/services/game_service.py) 的初始 metadata 里补上：

- `next_actor`
- `expected_turn_type`
- `last_actor`
- `last_intent`

### 第二步：先做入口校验

在 `submit_turn()` 里加一层判断：当前轮次不允许的输入，直接返回提示，不进入 Agent 处理。

### 第三步：改 `_handle_user_answer()`

这是最关键的一步。把“用户回答后自动生成下一题”的流程改掉，改成“用户回答后回到用户轮次”。

### 第四步：补 prompt 约束

把轮次规则写进 `turn_router.txt` 和 `agent_decision.txt`，避免模型在语言层面继续抢跑。

### 第五步：补验收测试

至少验证下面三种情况：

1. 用户回答完系统问题后，系统不会立刻再返回新的 `agent_question`。
2. 用户在 `user_turn` 阶段输入“是 / 不是”时，会被拒绝或提示改为提问。
3. 系统猜词后进入 `awaiting_judgement`，不会被别的分支抢走状态。

## 11. 推荐伪代码

下面这段伪代码可以直接作为实现目标：

```python
metadata = game.get("metadata", {})
phase = metadata.get("phase", "user_turn")
next_actor = metadata.get("next_actor", "user")

intent_result = await agent.parse_user_intent(user_text, history)
intent = intent_result.get("intent")

allowed_intents = {
    "user_turn": {"question", "guess"},
    "awaiting_answer": {"answer"},
    "awaiting_judgement": {"judge"},
}

if intent not in allowed_intents.get(phase, {"question"}):
    return {
        "message": "当前轮次不允许这个动作，请按提示继续。",
        "history": history,
    }

if phase == "awaiting_answer":
    # 记录用户回答
    # 更新 next_actor = user
    # 不再自动调用 decide_agent_action()
    return {
        "message": "回答已记录，现在轮到你提问或猜词。",
    }
```

这段伪代码的核心就是一句话：**先校验轮次，再处理意图。**

## 12. 验收标准

完成以后，至少要满足下面这些结果：

- 用户回答完系统问题后，后端不会马上再返回新的 `agent_question`
- 用户在自己的提问轮次输入“是 / 不是”，系统会提示当前轮次不对
- `awaiting_answer`、`awaiting_judgement`、`user_turn` 三个状态之间切换清晰
- 轮次日志里能看到 `next_actor` 的显式变化
- 第 21 轮用户回答“正确”后，不再自动跳到第 22 轮 `agent_guess`

---

## 13. 常见错误

### 13.1 只改 prompt，不改状态机

这是最常见也最没用的修法。prompt 可以辅助，但不能替代后端轮次控制。

### 13.2 只看 `intent`，不看 `phase`

如果只相信 `parse_user_intent()`，系统会把“当前轮次不允许的输入”也当成合法动作。

### 13.3 用户回答后立刻继续调用 Agent

这会直接把系统变成连续追问器，用户没有机会重新组织提问。

### 13.4 允许短句在任何阶段通用

“是 / 不是 / 对 / 不对” 这类短句，只能在 `awaiting_answer` 阶段有效，不能全局有效。

---

## 14. 结论

轮次控制不是一个 UI 提示问题，而是后端的状态约束问题。

如果你要把猜词游戏重新拉回正常节奏，就必须把下面这件事做成硬规则：

**用户回答后，系统必须停下来，把控制权交还给用户，直到用户再次主动发起提问或猜词。**

这就是这份手册的核心。
