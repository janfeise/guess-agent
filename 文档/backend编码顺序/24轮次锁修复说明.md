# 轮次锁修复说明 - 正确的游戏交替循环

## 问题诊断

**原始问题**：用户提问后，系统立即返回 `user_turn`，导致用户可以继续提问而系统永远没有机会主动出击（提问或猜测）。

**错误流程**：

```
用户提问 → 系统回答 → 返回 user_turn
  ↓
用户继续提问 → 系统继续回答 → 返回 user_turn
  （无限循环，系统永远不提问或猜测）
```

**API 响应示例（错误）**：

```json
{
  "phase": "user_turn",
  "message": "回答完毕，现在轮到你继续提问或直接猜答案。"
}
```

此时系统应该根据现有信息进行决策（continue asking or make a guess），而不是让用户继续控制。

## 正确的游戏循环设计

### 三个交替阶段

```
┌─────────────────────────────────────────┐
│  user_turn                              │
│  用户可以：提问 或 猜测                  │
└──────────┬──────────────────────────────┘
           │
     ┌─────┴──────┐
     │            │
     ▼            ▼
  提问        猜测 (guess)
     │            │
     ▼            ▼
系统回答    awaiting_judgement
     │            │
     ▼            │
系统决策     用户判断
    ╱╲           │
   ╱  ╲          ▼
  ╱    ╲     对/错
 ╱      ╲      ╱ ╲
▼        ▼    ▼   ▼
提问    猜测  赢   回到user_turn
 │      │
 ▼      ▼
waiting_answer awaiting_judgement
```

### 四个阶段的定义

| 阶段                   | Actor | 允许意图        | 等待动作                   |
| ---------------------- | ----- | --------------- | -------------------------- |
| **user_turn**          | 用户  | question, guess | 用户提事问题或猜测         |
| **waiting_answer**     | 用户  | answer          | 用户回答系统之前提的问题   |
| **awaiting_judgement** | 用户  | judge           | 用户判断系统的猜测是否正确 |

## 修复方案

### 1. `_handle_user_question()` 修复

**变化**：用户提问后，系统不仅要回答，还要进行决策

```python
# 之前：只回答问题，返回 user_turn
await self.repository.update_game_state(
    phase="user_turn",
    next_actor="user",
    expected_turn_type="question"
)

# 现在：回答问题后，调用 decide_agent_action 进行决策
agent_decision = await self.agent.decide_agent_action(raw_history, summary)

# 根据系统的决策决定下一个阶段
if agent_decision.get("action") == "question":
    # 系统决定继续提问
    await self.repository.update_game_state(
        phase="waiting_answer",  # ← 等待用户回答
        next_actor="user",
        expected_turn_type="answer"
    )
    return {
        "phase": "waiting_answer",
        "system_question": agent_question,
        "message": f"我的下一个问题是：{agent_question}"
    }
else:  # action == "guess"
    # 系统决定开始猜测
    await self.repository.update_game_state(
        phase="awaiting_judgement",  # ← 等待用户判断
        next_actor="user",
        expected_turn_type="judge"
    )
    return {
        "phase": "awaiting_judgement",
        "system_guess": guess_word,
        "message": f"我猜你想的词是：{guess_word}。对吗？"
    }
```

**关键点**：

- 系统的决策（问题或猜测）被添加到 `history` 中
- `phase` 和 `next_actor`/`expected_turn_type` 基于系统的决策设置
- 用户看到系统的决策（下一个问题或猜测）

### 2. `_handle_user_answer()` 修复

**变化**：用户回答后，系统再次进行决策（继续提问或猜测）

```python
# 用户回答后，系统决策
agent_decision = await self.agent.decide_agent_action(raw_history, summary)

# 根据系统的决策：
if agent_decision.get("action") == "question":
    # 继续提问
    await self.repository.update_game_state(
        phase="waiting_answer",
        # ...
    )
else:  # action == "guess"
    # 猜测答案
    await self.repository.update_game_state(
        phase="awaiting_judgement",
        # ...
    )
```

### 3. `_handle_user_guess()` 消息修正

**变化**：用户猜测错误后，应该提示"现在轮到你继续提问或猜词"，而不是"接下来轮到系统继续提问"

```python
# 之前
"message": f"很遗憾，{user_guess} 不是目标词，接下来轮到系统继续提问"

# 现在
"message": f"很遗憾，{user_guess} 不是目标词，游戏继续。现在轮到你继续提问或猜词。"
```

### 4. `_handle_agent_guess_judgement()` 不变

用户判断系统猜测错误后，返回 `user_turn`，让用户继续提问或猜测。这个逻辑已经是正确的。

## 修复后的正确流程

### 完整游戏循环示例

```
【第1轮】
1. 用户提问："是理科吗？"
   → _handle_user_question() 调用
   → 系统回答："是，属于理科"
   → 系统决策：decide_agent_action()
   → 决策结果：action="question", question="是物理吗？"
   → 返回 waiting_answer 阶段
   → 用户看到："我的下一个问题是：是物理吗？"

【第2轮】
2. 用户回答系统的问题："是"
   → _handle_user_answer() 调用
   → 系统决策：decide_agent_action()
   → 决策结果：action="question", question="是力学吗？"
   → 返回 waiting_answer 阶段
   → 用户看到："我的下一个问题是：是力学吗？"

【第3轮】
3. 用户回答系统的问题："不是"
   → _handle_user_answer() 调用
   → 系统决策：decide_agent_action()
   → 决策结果：action="guess", guess_word="量子力学"
   → 返回 awaiting_judgement 阶段
   → 用户看到："我猜你想的词是：量子力学。对吗？"

【第4轮】
4. 用户判断系统的猜测："是"
   → _handle_agent_guess_judgement() 调用
   → 游戏结束！
   → 用户看到："系统猜测正确！游戏结束"

或者：

4. 用户判断系统的猜测："不是"
   → _handle_agent_guess_judgement() 调用
   → 返回 user_turn 阶段
   → 用户看到："系统猜测不正确，游戏继续。现在轮到你继续提问或猜词。"
   → 回到【第1轮】，用户可以继续提问或猜测
```

## 核心改动总结

### 改动文件

- `backend/app/services/game_service.py`

### 改动函数

| 函数                      | 改动                                         | 目的                     |
| ------------------------- | -------------------------------------------- | ------------------------ |
| `_handle_user_question()` | 添加 `decide_agent_action()` 调用 + 条件路由 | 系统在用户提问后进行决策 |
| `_handle_user_answer()`   | 添加 `decide_agent_action()` 调用 + 条件路由 | 系统在用户回答后进行决策 |
| `_handle_user_guess()`    | 修正错误消息                                 | 提示用户正确的轮次状态   |

### 状态转移表

用户提问/回答后：

```
┌──────────────────────────┐
│ decide_agent_action()    │
│ 返回 action 字段         │
└──────────┬───────────────┘
           │
     ┌─────┴──────┐
     │            │
  "question"   "guess"
     │            │
     ▼            ▼
phase=waiting_answer
next_actor=user
expected_turn_type=answer

phase=awaiting_judgement
next_actor=user
expected_turn_type=judge
```

## 后续优化

### 可选的 Prompt 补充

虽然代码层已经确保轮次锁的强制执行，为了提高 LLM 的理解，可在提示词中补充：

**在 `agent_decision.txt` 中添加**：

```
【轮次说明】
- 这是游戏每轮的决策阶段，你已经收集到用户提供的信息
- 根据收集到的信息，决定是继续提问还是开始猜测
- 如果信息还不充分，选择继续提问
- 如果信息已充分，选择进行猜测
- 你的决策将不再由 LLM 生成"下一个问题"，而是直接返回给用户
```

但这是**可选的**，因为代码层已经完全控制了流程。

## 性能影响

- 额外的 LLM 调用：`decide_agent_action()` 在 `_handle_user_question()` 和 `_handle_user_answer()` 中各调用一次
- 平均额外延迟：~2-3秒（LLM 响应时间）
- 这在游戏交互流程中是可以接受的

## 总结

修复后的轮次锁实现确保：

1. ✅ 游戏是真正的**交替制**（user ↔ system）
2. ✅ 系统在收集信息后**主动决策**（提问或猜测）
3. ✅ 用户不被动地被系统连续提问
4. ✅ 用户也不能连续提问（系统会在其他时间点插入决策）
5. ✅ 所有状态转移都通过 `phase` 和 `next_actor` 明确控制
