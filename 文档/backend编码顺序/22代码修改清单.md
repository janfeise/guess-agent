# 22 代码修改清单

## 1. 文档目标

这份手册的目标非常明确：**把“双词系统隔离”真正落到代码里**。

前面的文档已经讲清楚了原因、原则和目标架构，这一章只做一件事：把修改拆成可执行的代码清单，按文件、按步骤、按优先级列出来。

你可以把这一章当成施工图，而不是设计说明。

---

## 2. 修改总原则

在开始改代码前，先记住这三条硬规则：

1. `raw_history` 和 `reasoning_history` 必须分开。
2. 系统词的回答可以留在审计记录里，但不能直接进入 Agent 推理链。
3. `hint` 默认只能展示，不能默认参与 `decide_agent_action()`。

只要这三条不落地，双词系统就一定会串线。

## 3.1 先看最终目标代码形态

你可以先把目标代码理解成下面这个结构：

```text
submit_turn()
    -> parse_user_intent()
    -> 根据 phase / intent 选择分支
    -> 记录 raw_history
    -> 生成 reasoning_history
    -> decide_agent_action(reasoning_history)
    -> 写回数据库
```

要点是：

- `raw_history` 负责留档
- `reasoning_history` 负责推理
- `decide_agent_action()` 永远不要直接吃原始历史

如果这三个层次没有分开，后面改任何 prompt 都不稳定。

## 4. 第一阶段：先把数据结构改对

### 4.1 修改 `create_game()` 写入的 `metadata`

文件：[backend/app/services/game_service.py](../../backend/app/services/game_service.py)

在 `create_game()` 里，当前 `metadata` 只有：

- `phase`
- `awaiting_judgement`
- `awaiting_answer`
- `pending_guess`
- `pending_question`
- `agent_confidence`

需要新增这些字段：

- `game_mode`：建议固定为 `dual_word_isolated`
- `reasoning_scope`：当前推理目标，建议初始为 `user_word`
- `last_actor`：上一动作执行者
- `last_intent`：上一轮用户意图

建议目标结构如下：

```python
"metadata": {
    "phase": "user_turn",
    "game_mode": "dual_word_isolated",
    "reasoning_scope": "user_word",
    "last_actor": "system",
    "last_intent": None,
    "awaiting_judgement": False,
    "awaiting_answer": False,
    "pending_guess": "",
    "pending_question": "",
    "agent_confidence": 0.0,
}
```

建议你在 `create_game()` 里直接落一个帮助函数，避免以后每次手写同样的 metadata：

```python
def _build_initial_metadata(self) -> dict:
    return {
        "phase": "user_turn",
        "game_mode": "dual_word_isolated",
        "reasoning_scope": "user_word",
        "last_actor": "system",
        "last_intent": None,
        "awaiting_judgement": False,
        "awaiting_answer": False,
        "pending_guess": "",
        "pending_question": "",
        "agent_confidence": 0.0,
    }
```

然后在 `create_game()` 里直接使用：

```python
game_doc = {
    ...
    "metadata": self._build_initial_metadata(),
    ...
}
```

### 4.2 统一 history 记录字段

所有写入 `history` 的地方，都要统一补这些字段：

- `source_word_owner`
- `visible_to_agent`
- `visible_to_player`

建议约定：

- 系统词回答，默认 `source_word_owner = system`
- 用户词推理链，默认 `source_word_owner = user`
- 系统词相关回答默认 `visible_to_agent = false`
- 用户词链路默认 `visible_to_agent = true`

建议你再补两个辅助函数，统一生成 history 记录：

```python
def _make_history_item(
    self,
    *,
    round_no: int,
    actor: str,
    turn_type: str,
    input_text: str,
    answer_text: str = "",
    answer_label: str | None = None,
    hint: str | None = None,
    confidence: float = 0.0,
    result: str = "",
    source_word_owner: str = "user",
    visible_to_agent: bool = True,
    visible_to_player: bool = True,
) -> dict:
    item = {
        "round_no": round_no,
        "actor": actor,
        "turn_type": turn_type,
        "input_text": input_text,
        "answer_text": answer_text,
        "confidence": confidence,
        "result": result,
        "source_word_owner": source_word_owner,
        "visible_to_agent": visible_to_agent,
        "visible_to_player": visible_to_player,
    }
    if answer_label is not None:
        item["answer_label"] = answer_label
    if hint is not None:
        item["hint"] = hint
    return item
```

这样所有分支都能复用同一种写法，后面排查历史污染也更容易。

### 4.3 先不要改业务逻辑，先改落库结构

这一阶段不要急着删 prompt，也不要急着改 Agent 决策，只先把结构补齐。

原因很简单：

- 结构没补齐，后续过滤没法写
- history 没有来源字段，隔离规则无法执行

## 5. 第二阶段：改 `GameService` 的回合编排

### 5.1 改 `_handle_user_question()`

文件：[backend/app/services/game_service.py](../../backend/app/services/game_service.py)

这是本次改造最重要的地方。

当前逻辑是：

1. `answer_question()` 生成系统回答
2. 把回答写入 `updated_history`
3. 直接把 `updated_history` 送进 `decide_agent_action()`

这个流程在双词模式下会污染 Agent。

#### 需要改成什么

改成两步：

1. 先记录完整历史，也就是 `raw_history`
2. 再构造 `reasoning_history`，只把允许推理的信息传给 Agent

#### 需要新增的辅助函数

建议在 `GameService` 中新增：

```python
def build_reasoning_history(self, history: list[dict], target_owner: str) -> list[dict]:
    ...
```

建议你按下面的方式实现它：

```python
def build_reasoning_history(self, history: list[dict], target_owner: str) -> list[dict]:
    reasoning_history = []
    for item in history:
        if item.get("source_word_owner") != target_owner:
            continue
        if not item.get("visible_to_agent", True):
            continue

        reasoning_history.append({
            "round_no": item.get("round_no"),
            "actor": item.get("actor"),
            "turn_type": item.get("turn_type"),
            "input_text": item.get("input_text", ""),
            "answer_text": item.get("answer_text", ""),
            "answer_label": item.get("answer_label"),
            "confidence": item.get("confidence", 0.0),
        })

    return reasoning_history
```

如果你当前要推理的是用户词，就传 `target_owner = "user"`。如果当前要推理的是系统词，就传 `target_owner = "system"`。

#### `_handle_user_question()` 的推荐代码骨架

```python
async def _handle_user_question(self, game: dict, question: str, history: list[dict], mode: str = "agent"):
    target_word = self.reveal_target_word(game, self.settings)
    logger.info("处理普通提问: game_id=%s question=%s", game["game_id"], self._preview_text(question))

    answer_result = await self.agent.answer_question(question, target_word)

    if isinstance(answer_result, dict):
        answer_label = answer_result.get("answer", "unknown")
        answer_text = answer_result.get("response_text") or self._compose_user_response(answer_label, answer_result.get("hint", ""))
        answer_reason = answer_result.get("reason", "")
        answer_hint = answer_result.get("hint", "")
        answer_confidence = answer_result.get("confidence", 0.0)
    else:
        answer_label = "unknown"
        answer_text = str(answer_result)
        answer_reason = ""
        answer_hint = ""
        answer_confidence = 0.0

    raw_history = history + [self._make_history_item(
        round_no=game.get("round_count", 0) + 1,
        actor="user",
        turn_type="question",
        input_text=question,
        answer_text=answer_text,
        answer_label=answer_label,
        hint=answer_hint,
        confidence=answer_confidence,
        result=answer_reason or f"answer={answer_label}",
        source_word_owner="system",
        visible_to_agent=False,
        visible_to_player=True,
    )]

    reasoning_history = self.build_reasoning_history(raw_history, target_owner="user")
    summary = self.memory_policy.build_summary(reasoning_history)
    agent_action = await self.agent.decide_agent_action(reasoning_history, summary)

    ...
```

注意这里的关键点：

- `raw_history` 存完整事件
- `reasoning_history` 才进入 Agent
- 系统词的问答记录默认 `source_word_owner = system` 且 `visible_to_agent = False`

过滤规则至少要包含：

- `source_word_owner == target_owner`
- `visible_to_agent == True`
- 去掉不必要的 `hint`
- 去掉内部状态说明

### 5.2 改 `_handle_user_answer()`

文件：[backend/app/services/game_service.py](../../backend/app/services/game_service.py)

用户回答阶段同样不能直接把原始 `history` 送进 Agent。

这里要做的事情和 `_handle_user_question()` 一样：

1. 先写 `raw_history`
2. 再生成过滤后的 `reasoning_history`
3. 用过滤后的上下文调用 `decide_agent_action()`

你可以直接复用同一套流程，只是把历史条目的归属改成当前正在推理的词源。

推荐代码骨架如下：

```python
async def _handle_user_answer(self, game: dict, user_input: str, history: list[dict], intent_result: dict | None = None):
    resolved_answer = self._resolve_user_answer(user_input, intent_result)
    if resolved_answer is None:
        return {
            "game_id": game["game_id"],
            "status": game.get("status", "active"),
            "phase": game.get("metadata", {}).get("phase", "awaiting_answer"),
            "message": "请回答是、否或不知道",
            "history": history,
        }

    raw_history = history + [self._make_history_item(
        round_no=game.get("round_count", 0) + 1,
        actor="user",
        turn_type="answer",
        input_text=user_input,
        answer_text=resolved_answer,
        result=(intent_result or {}).get("reason", ""),
        source_word_owner="user",
        visible_to_agent=True,
        visible_to_player=True,
    )]

    reasoning_history = self.build_reasoning_history(raw_history, target_owner="user")
    summary = self.memory_policy.build_summary(reasoning_history)
    agent_action = await self.agent.decide_agent_action(reasoning_history, summary)

    ...
```

这里的重点不是代码格式，而是数据边界：用户回答进入了哪条链，必须明确写出来。

## 6. 第三阶段：改 `GuessAgent`

### 6.1 改 `decide_agent_action()` 的输入假设

文件：[backend/app/agents/guess_agent_fixed.py](../../backend/app/agents/guess_agent_fixed.py)

这个方法本身不一定要知道“双词”概念，但它必须明确：

- 传进来的就是过滤后的推理历史
- 传进来的不是原始审计历史
- 它不能假设自己拿到的是完整世界状态

如果你想让这个方法更清晰，可以先在函数开头加一个轻量检查：

```python
def _ensure_reasoning_history(self, history: list[dict]) -> list[dict]:
    if not history:
        return []
    return history
```

然后 `decide_agent_action()` 只接受这类已过滤数据。这样以后如果有人误传 raw_history，也容易在入口处加校验。

### 6.2 给 `parse_user_intent()` 保留最近一轮上下文

这一步在前面的 20 号手册里已经提过了，这里再列一次是为了不漏改。

建议保持：

- 最近一轮问题
- 最近一轮回答
- 当前用户输入

三者一起进入意图解析。

但注意：这里是“解析意图”，不是“进入 Agent 推理链”。

推荐写成下面这样：

```python
async def parse_user_intent(self, user_input: str, history: list[dict]) -> dict:
    messages = [
        {"role": "system", "content": self.prompt_loader.load_prompt("turn_router")},
    ]

    if history:
        last_turn = history[-1]
        previous_question = last_turn.get("input_text", "")
        previous_answer = last_turn.get("answer_text", "")
        if previous_question or previous_answer:
            messages.append({
                "role": "assistant",
                "content": f"上一轮问题：{previous_question}",
            })
            messages.append({
                "role": "user",
                "content": f"上一轮回答：{previous_answer}",
            })

    messages.append({"role": "user", "content": f"请分析用户输入的意图：{user_input}"})
    ...
```

这个写法比直接把整段 history 丢进去稳定得多。

### 6.3 `answer_question()` 的 `hint` 只做展示值

文件：[backend/app/agents/guess_agent_fixed.py](../../backend/app/agents/guess_agent_fixed.py)

当前 `answer_question()` 会返回：

- `answer`
- `response_text`
- `hint`
- `confidence`

这里建议明确：

- `hint` 可以保留
- 但服务层写入 history 后，要决定它是否能进入 Agent 上下文
- 如果当前回答属于系统词链路，`hint` 默认不能传进 `decide_agent_action()`

建议你在 service 层写历史时，直接决定是否保留 `hint`：

```python
show_hint = answer_hint if answer_result.get("source_word_owner") == "user" else ""
```

如果你暂时不想加更复杂的来源逻辑，最少也要保证：系统词回答后的 `hint` 不进入 Agent 决策输入。

### 6.4 不要让 Agent 自己判断数据归属

Agent 只应该做：

- 读过滤后的上下文
- 输出下一步建议

不要让它自己判断“这条信息属于 system 还是 user”。这件事必须由后端程序控制。

## 7. 第四阶段：改 prompt

### 7.1 改 `turn_router.txt`

文件：[backend/app/agents/prompt/turn_router.txt](../../backend/app/agents/prompt/turn_router.txt)

这里要加的不是新分类，而是新约束：

### 7.2 改 `guess_system.txt`

文件：[backend/app/agents/prompt/guess_system.txt](../../backend/app/agents/prompt/guess_system.txt)

这里主要管“系统回答用户问题”的风格。

需要强调：

- 输出要简短
- 不要泄露答案本身
- `hint` 必须抽象、短小
- 不要把系统词的具体语义直接暴露成可推理结论

### 7.3 改 `agent_decision.txt`

文件：[backend/app/agents/prompt/agent_decision.txt](../../backend/app/agents/prompt/agent_decision.txt)

这是双词隔离最关键的 prompt 文件。

必须补的规则：

- 只能使用传入的 `reasoning_history`
- 不要使用未过滤的原始历史
- 不要把系统词回答当成用户词证据
- 如果信息来源不属于当前推理目标，就不要采纳

建议你直接把下面这类约束加到 prompt 中：

```text
你只能使用传入的 reasoning_history，不要使用未过滤的原始 history。
如果历史记录标记为 source_word_owner=system，且当前目标是 user_word，则不得把这条信息当作推理依据。
如果历史记录中的 hint 来自系统词链路，则默认忽略。
```

这几句话比抽象地说“不要串线”更有效，因为它直接限制了模型的可见范围。

### 7.4 如有需要，给 prompt 增加“来源边界”说明

建议把这类话直接写进 prompt：

- 当前目标是推理用户词时，系统词的回答不能当作依据
- 当前目标是系统词时，用户词链路的数据不能混用

这不是“风格优化”，而是防止模型乱用上下文。

## 9. 推荐实施顺序

### 第 1 步：补字段

先给 `create_game()` 和所有 `history` 写入位置加上来源字段。

先改 `_handle_user_question()`，再改 `_handle_user_answer()`。

### 第 4 步：收口 Agent 输入

让 `decide_agent_action()` 只吃过滤后的上下文。

### 第 5 步：补 prompt 约束

最后再改三个 prompt，避免模型继续顺着错误上下文推理。

### 第 6 步：补一个最小验证脚本

建议你在本地验证时，至少观察这三类日志：

- `parse_user_intent` 的输入是否包含最近一轮上下文
- `decide_agent_action` 的输入是否已经过滤掉系统词记录
- `history` 写库时是否补齐了来源字段

如果这三处都对了，双词隔离就已经成功了一大半。

## 10. 验收清单

下面这些条件都满足，才算本章真正完成：

- `history` 里的每条记录都有来源字段
- 系统词相关回答不会直接进入 Agent 决策输入
- `hint` 不再默认污染 `decide_agent_action()`
- Agent 只基于过滤后的历史继续提问或猜测
- 用户问“是文科吗？”时，不会再因为系统词 `刑法` 的回答把整局带偏
- 用户答“无关”时，只影响当前推理链，不影响另一条词源链

你还可以额外检查一条更直接的现象：

- 当系统词是“刑法”、用户词是“伦敦”时，用户问“是文科吗？”后，Agent 不应该继续往“法律学科”方向追问

如果这条现象还存在，说明 `reasoning_history` 仍然把系统词信息带进去了，需要回头检查 `_handle_user_question()` 的过滤逻辑。

## 11. 结论

这份清单的核心不是“多改几个文件”，而是把修改顺序固定下来：
