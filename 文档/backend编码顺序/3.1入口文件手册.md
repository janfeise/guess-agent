# 3.1 入口文件手册

## 1. 文档目的

这一章的目标是指导你完成后端入口文件，也就是 `backend/app/main.py`。

入口文件是整个后端服务的总开关，它负责把配置、数据库、路由、生命周期和中间件串起来。前面的配置类、MongoDB 连接、健康接口，最终都要通过这个入口文件真正跑起来。

这一章完成后，你应该能够做到：

- 启动 FastAPI 应用
- 正确加载统一配置
- 在启动时初始化 MongoDB
- 注册健康接口路由
- 为后续 Repository、Service、Agent 接口预留统一入口
- 支持本地运行和 Docker 运行

## 2. 为什么入口文件必须单独写

入口文件不是普通业务模块，它是应用的组织中心。

如果没有入口文件，下面这些事情都无法稳定成立：

- 路由无法统一注册
- 数据库无法在启动阶段初始化
- 健康检查无法挂载到应用上
- Docker 启动命令没有明确入口
- 中间件和跨域配置没有统一位置

所以入口文件应该作为后端最先落地的核心文件之一。

## 3. 入口文件的职责边界

入口文件只负责“把应用组装起来”，不负责具体业务。

它应该做的事情：

- 创建 FastAPI 实例
- 加载配置
- 处理应用生命周期
- 注册路由
- 注册中间件
- 暴露启动对象 `app`

它不应该做的事情：

- 不直接编写业务逻辑
- 不直接操作游戏规则
- 不直接写数据库增删改查
- 不直接实现 Agent 推理

这些事情应该分别放在 Service、Repository、Agent 层中完成。

## 4. 推荐文件位置

建议入口文件放在：

```bash
backend/
└── app/
    └── main.py
```

这是 FastAPI 项目里最清晰、最常见的方式。

## 5. 入口文件需要依赖什么

入口文件至少需要依赖以下模块：

- 配置类：`app.core.config`
- MongoDB 连接：`app.core.database`
- 健康接口路由：`app.api.v1.health`

后面如果你再写其他接口，也会继续在这里注册：

- `game.py`
- `agent.py`
- `helper.py`

## 6. 入口文件的执行顺序

入口文件最重要的是“顺序感”，建议按下面这个流程组织：

1. 导入配置和路由
2. 定义应用生命周期
3. 创建 FastAPI 实例
4. 注册中间件
5. 注册路由
6. 暴露 `app`

这个顺序非常重要，因为很多启动问题都来自初始化顺序不合理。

## 7. 推荐的生命周期设计

### 7.1 为什么要有生命周期

MongoDB 初始化、索引创建、连接关闭，这些都不适合放在请求处理函数里，而应该放在应用启动和关闭阶段。

### 7.2 推荐方式

建议使用 FastAPI 的 `lifespan` 机制来管理启动和关闭过程。

这样做的好处是：

- 启动时自动初始化 MongoDB
- 关闭时自动释放连接
- 逻辑集中，不会散落到多个地方
- 更容易和 Docker 配合

## 8. 推荐实现结构

建议把入口文件拆成以下几个核心部分：

```bash
main.py
├── 引入 settings
├── 引入 init_db / close_db
├── 定义 lifespan
├── 创建 FastAPI app
├── include_router
├── 可选：中间件
└── 可选：根路径或调试接口
```

## 9. 推荐实现代码

下面是一份可以直接参考的入口文件实现示例。你可以把它作为 `backend/app/main.py` 的实现模板。

```python
from contextlib import asynccontextmanager

from fastapi import FastAPI

from app.api.v1.health import router as health_router
from app.core.config import get_settings
from app.core.database import close_db, init_db


settings = get_settings()


@asynccontextmanager
async def lifespan(app: FastAPI):
    await init_db()
    yield
    await close_db()


app = FastAPI(
    title=settings.app_name,
    lifespan=lifespan,
)

app.include_router(health_router, prefix="/api/v1")


@app.get("/")
async def root():
    return {
        "status": "ok",
        "service": settings.app_name,
    }
```

## 10. 代码说明

### 10.1 为什么要先读取 settings

入口文件是整个应用最早接触配置的地方。它先读取 `settings`，可以保证服务名、端口、环境变量和后续路由注册逻辑都使用同一套配置。

### 10.2 为什么要在 lifespan 中调用 init_db()

这样 MongoDB 的初始化会在应用启动时自动执行，而不是等到某个接口第一次访问时才去初始化。

### 10.3 为什么要在 lifespan 结束时调用 close_db()

这样可以保证服务关闭时连接被正确释放，避免开发模式热重载或异常退出后留下资源。

### 10.4 为什么要 include_router

FastAPI 的路由不会自动生效，必须在入口文件中显式注册。这样后续的 health、game、agent、helper 路由才能对外访问。

## 11. 路由组织建议

demo 阶段建议给所有业务接口统一加一个版本前缀，例如：

- `/api/v1/health`
- `/api/v1/game/start`
- `/api/v1/agent/chat`

这样后面如果你要做第二版 API，迁移会更容易。

### 11.1 健康接口的注册方式

健康接口现在已经编写完成，入口文件需要把它注册进来。

建议方式：

- 导入 `health_router`
- 使用 `app.include_router(health_router, prefix="/api/v1")`

这样最终健康接口地址就是：

- `GET /api/v1/health`

## 12. 中间件建议

demo 阶段入口文件不需要太多中间件，但有两个建议可以提前考虑：

### 12.1 CORS

如果后面前端要联调，建议在入口文件里统一加 CORS 中间件。

### 12.2 请求日志

如果你想在调试时更容易定位问题，也可以在入口文件中统一做请求日志中间件。

但这些都不是当前最优先的事情，入口文件先保证能启动、能注册路由、能初始化数据库就足够了。

## 13. 启动命令配合方式

你的 Docker Compose 已经使用了类似下面的启动方式：

- `uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload`

这说明入口文件必须提供一个可被 Uvicorn 读取的 `app` 对象。

所以你写 `main.py` 的时候，必须确保：

- 顶层变量名叫 `app`
- `app` 是 FastAPI 实例
- 路由在导入后已经注册完成

否则 Docker 启动时会找不到入口。

## 14. 与 Docker 的配合方式

入口文件在 Docker 环境下尤其重要，因为它决定容器启动后服务是否能真正起来。

建议你在完成入口文件后立刻做下面几件事：

- 确认 `app.main:app` 可以被 Uvicorn 识别
- 确认 `/api/v1/health` 可以访问
- 确认 MongoDB 初始化不会在容器启动时报错

### 14.1 容器内启动注意点

- `host` 必须是 `0.0.0.0`
- 端口要和 Compose 保持一致
- 环境变量要由 Compose 注入

## 15. 常见错误

### 15.1 找不到 `app.main:app`

通常说明：

- `main.py` 不存在
- `app` 变量名写错了
- 入口文件路径不对

### 15.2 路由访问 404

通常说明：

- `include_router` 没写
- 路由前缀不一致
- 健康接口文件没有被导入

### 15.3 启动时 MongoDB 报错

通常说明：

- 数据库服务还没起来
- `MONGO_URI` 不正确
- `init_db()` 里索引创建失败

## 16. 这一章完成后的验收标准

你可以用下面几个标准判断入口文件是否正确：

- `uvicorn app.main:app` 能启动
- 应用启动时能自动执行 MongoDB 初始化
- `/api/v1/health` 可以访问
- `/` 根路径能返回简单状态信息
- Docker 容器启动后不再因为缺少入口文件而退出
