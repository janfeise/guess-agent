# 12 游戏开始小助手调用手册

## 1. 文档目标

这一章的目标是把“游戏开始时调用小助手 LLM API，传递提示词，获取返回词，并把这个词作为本局目标词”的链路定义清楚。

写完这一章后，你应该能明确这些事情：

- 游戏开始时，为什么要调用小助手模型
- 提示词应该放在哪个文件里
- 请求参数应该怎么传给模型
- 模型返回词后，后端应该如何校验、清洗、保存
- API 层、Service 层、Agent 层分别做什么
- 当前项目里已经有哪些代码可以直接复用

这一章只讲“开局取词”这一件事，不展开后续回合问答。

### 2.1 这些文件已经分别做了什么

- `game_start.txt`：已经写好了开局提示词，核心目标是让模型只输出一个词
- `guess_agent.py`：已经封装了主模型和小助手模型
- `prompt_loader.py`：已经提供了从文件读取 prompt 的能力
- `config.py`：已经配置了小助手模型的 API Key、Base URL、模型名和温度
- `game_service.py`：已经是游戏流程编排层，适合把“开局取词”放进来
- `games.py`：已经是创建游戏接口所在位置，适合接收开局请求
- `security.py`：已经提供加密和解密能力，适合把模型返回词加密后保存
- `game_repository.py`：已经提供写入和查询游戏记录的能力

### 2.2 当前代码里有一个需要注意的点

`PromptLoader` 目前提供的方法名是 `load_prompt()`，但是 `GuessAgent` 里 `_load_prompt()` 调用的是 `self.prompt_loader.load(...)`。

这意味着你在真正接入开局取词逻辑前，先要把方法名统一掉，不然模型提示词会加载失败。

### 2.3 关于“学科领域”字段的判断

这个字段不作为前端开局请求参数，是合理的。

原因很直接：

- 用户开局时只需要开始猜词，不一定需要先知道领域
- 领域信息如果暴露给用户，会削弱游戏探索感
- 领域更适合由后端内部控制，而不是由客户端显式传入
  但要注意一致性：

- 如果你仍然保留 [backend/app/agents/prompt/game_start.txt](../../backend/app/agents/prompt/game_start.txt) 里“必须属于指定学科领域”这个约束，那么后端就要自己决定领域
- 如果你不想维护内部领域，就应该同步修改 prompt，把这条约束删掉

结论是：

**学科领域可以不对外暴露，但 prompt 和服务端逻辑必须保持一致。**

## 3. 这条链路的目标行为

游戏开始时，后端应该做下面这些事：

1. 接收开局请求
2. 判断本局是否需要由小助手生成目标词
3. 从 [backend/app/agents/prompt/game_start.txt](../../backend/app/agents/prompt/game_start.txt) 读取系统提示词
4. 额外补一条用户消息，告诉模型本局的难度约束和输出格式；如果保留领域约束，则由后端内部补充，不由前端传入
5. 调用小助手模型
6. 拿到模型返回词
7. 清洗返回值，只保留单个词
8. 校验长度、字符、歧义和格式
9. 把结果写入 MongoDB
10. 返回创建成功结果，但不返回目标词明文

### 3.1 为什么要走小助手模型

因为开局取词不是普通问答，而是“按约束生成谜底”。

小助手模型更适合做这个动作：

- 生成一个符合题目要求的目标词
- 避免主模型和后续推理链路混在一起
- 方便把“取词”与“猜词”拆成两个职责

### 3.2 为什么不能直接在代码里写死目标词

因为写死目标词会导致：

- 每局游戏都一样，无法变化
- 无法体现“动态开局”
- 后续测试和扩展都不灵活

用 LLM 生成目标词，可以把开局策略和题目约束放进 prompt 里，后续只需要改 prompt 就能调整规则。

## 4. 推荐的调用链路

推荐的调用链路如下：

```text
前端 / 客户端
  -> API 路由
  -> GameService.create_game() 或 GameService.start_game()
  -> GuessAgent.generate_start_word()
  -> PromptLoader.load_prompt("game_start")
  -> helper_model
  -> 返回目标词
  -> 校验 / 清洗 / 加密
  -> GameRepository.create_game()
  -> MongoDB
```

API 层负责：

- 接收开局请求参数
- 做最基础的字段校验
- 把参数传给 Service 层
- 把 Service 层返回的结果转换成 HTTP 响应

### 4.2 Service 层负责什么

Service 层负责：

- 先调用小助手模型取词
- 再做词语校验和清洗
- 决定是否加密后落库
- 把最终游戏记录写入数据库

### 4.3 Agent 层负责什么

Agent 层负责：

- 读取提示词
- 选择 helper 模型
- 把 system prompt 和用户约束拼成消息
- 调用 LLM API
- 返回原始回答文本

### 4.4 Repository 层负责什么

Repository 层负责：

- 把开局生成的游戏数据写入 MongoDB
- 按 `game_id` 读取或更新游戏记录
- 不参与模型推理

## 5. 请求与返回设计

### 5.1 推荐请求字段

建议开局请求至少包含：

- `owner_id`：可选，表示创建者
- `difficulty`：可选，表示难度，例如 `easy`、`medium`、`hard`
- `subject`：不建议作为外部请求字段，除非你明确要让前端控制领域

### 5.2 推荐请求示例

```json
{
  "owner_id": "user-001",
  "difficulty": "medium"
}
```

### 5.3 推荐返回字段

建议返回：

- `game_id`
- `status`
- `round_count`
- `created_at`

不建议返回：

- 目标词明文
- 模型原始 prompt
- 模型调用密钥
- 系统内部加密材料

### 5.4 为什么目标词不直接返回

因为目标词就是本局的谜底，直接返回会破坏游戏规则。

如果前端需要展示开局结果，只展示：

- 游戏已创建
- 当前状态
- 回合数

目标词应只保存在后端。

## 6. Prompt 设计方式

### 6.1 `game_start.txt` 的角色

[backend/app/agents/prompt/game_start.txt](../../backend/app/agents/prompt/game_start.txt) 应该作为小助手开局的 system prompt。

它已经表达了这几个核心要求：

- 只能返回一个词语
- 必须是明确名词
- 必须属于指定学科领域
- 难度适中
- 不能解释

### 6.2 为什么还要再传一条用户消息

因为 system prompt 只能告诉模型规则，但不能告诉它本局具体要生成哪个领域。

所以你还要额外传一条 user message，例如：

```text
请根据以下约束生成一个目标词：
难度：medium
要求：只返回一个词语，不要解释。
```

如果你保留“领域”这一约束，建议由后端内部拼接到这段用户消息里，而不是让前端显式传入。

### 6.3 推荐消息结构

```python
messages = [
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": user_prompt},
]
```

### 6.4 为什么不把 subject 写死在 prompt 文件里

因为 `game_start.txt` 应该是通用模板，而不是某一个固定学科的专用 prompt。

如果你要保留领域约束，建议把 `subject` 放到后端内部传入，而不是作为前端请求字段。这样才能支持：

- 计算机局
- 数学局
- 历史局
- 后续新增任何学科局

## 8. 核心代码参考

下面这部分是最关键的实现参考。你可以直接按这个思路改。

### 8.1 GuessAgent 增加开局取词方法

```python
# backend/app/agents/guess_agent.py

class GuessAgent:
    def __init__(self, settings, prompt_loader, memory_policy):
        self.settings = settings
        self.prompt_loader = prompt_loader
        self.memory_policy = memory_policy
        self.agent_model = self._build_agent_model()
        self.helper_model = self._build_helper_model()

    def _load_prompt(self, mode):
        if mode == "helper":
            return self.prompt_loader.load_prompt("game_start")
        return self.prompt_loader.load_prompt("guess_system")

    def _select_model(self, mode: str):
        if mode == "helper":
            return self.helper_model
        return self.agent_model

    def _normalize_answer(self, answer: str) -> str:
        cleaned = answer.strip()
        for token in ["\"", "'", "“", "”", "。", "！", "？", ",", "，", "\n", "\r", "\t"]:
            cleaned = cleaned.replace(token, "")
        return cleaned

    async def generate_start_word(self, difficulty: str | None = None, subject: str | None = None) -> str:
        system_prompt = self.prompt_loader.load_prompt("game_start")
        user_prompt = (
            f"请根据以下约束生成目标词。\n"
            f"难度：{difficulty or 'medium'}\n"
            f"要求：只返回一个词语，不要解释。"
        )

        if subject:
            user_prompt = f"{user_prompt}\n学科领域：{subject}"

        messages = [
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt},
        ]

        response = await self.helper_model.agenerate(messages)
        return self._normalize_answer(response.generations[0][0].text)
```

### 8.2 这里为什么要单独写 `generate_start_word()`

因为开局取词和普通问答不是一类问题：

- 普通问答依赖 history
- 开局取词只依赖 prompt、难度和后端内部约束
- 普通问答要控制上下文记忆
- 开局取词要控制输出格式和词语合法性

把它们拆开，代码会更清晰。

### 8.3 GameService 里组织开局流程

```python
# backend/app/services/game_service.py

import uuid
from datetime import datetime

from app.core.security import encrypt
from app.core.constants import validate_word

class GameService:
    def __init__(self, settings, agent, repository, memory_policy):
        self.settings = settings
        self.agent = agent
        self.repository = repository
        self.memory_policy = memory_policy

    async def create_game(self, owner_id=None, difficulty: str | None = None):
        target_word = await self.agent.generate_start_word(difficulty=difficulty)
        normalized_target_word = validate_word(target_word)

        game_id = str(uuid.uuid4())
        now = datetime.utcnow()
        target_word_encrypted = encrypt(normalized_target_word, self.settings.encryption_secret)

        game_doc = {
            "game_id": game_id,
            "owner_id": owner_id,
            "status": "active",
            "history": [],
            "summary": "",
            "round_count": 0,
            "created_at": now,
            "updated_at": now,
            "difficulty": difficulty or "medium",
            "target_word_encrypted": target_word_encrypted,
            "target_word_source": "helper_llm",
        }

        await self.repository.create_game(game_doc)

        return {
            "game_id": game_id,
            "status": "active",
            "round_count": 0,
            "created_at": now,
        }
```

### 8.4 为什么要先校验再加密

因为模型返回值不一定完全符合要求。

你至少要校验：

- 不能为空
- 长度在 2 到 6 个字之间
- 不能有多余标点
- 不能是整句
- 不能返回解释文字

先校验，后加密，才能保证入库的是可用目标词。

### 8.5 API 路由里传参

```python
# backend/app/api/v1/games.py

from fastapi import APIRouter, Depends, HTTPException, status
from pydantic import BaseModel, Field

class CreateGameRequest(BaseModel):
    owner_id: str | None = Field(default=None, description="游戏创建者ID")
    difficulty: str | None = Field(default="medium", description="难度，例如 easy / medium / hard")

@router.post("", status_code=status.HTTP_201_CREATED)
async def create_game(request: CreateGameRequest, service: GameService = Depends(get_game_service)):
    try:
        return await service.create_game(
            owner_id=request.owner_id,
            difficulty=request.difficulty,
        )
    except ValueError as exc:
        raise HTTPException(status_code=status.HTTP_400_BAD_REQUEST, detail=str(exc))
```

### 8.6 依赖注入时要真正创建 GuessAgent

你当前 [backend/app/api/v1/games.py](../../backend/app/api/v1/games.py) 里的 `get_game_service()` 还在传 `agent=None`。

如果要让开局时真的能调用小助手模型，这里必须改成真正实例化 `GuessAgent`，例如：

```python
from app.agents.guess_agent import GuessAgent
from app.agents.utils.prompt_loader import PromptLoader
from app.agents.utils.memory_policy import MemoryPolicy
from app.core.config import get_settings
from app.core.database import get_database
```

## 9. 词语校验建议

### 9.1 目前的 `validate_word()` 还不够专门

你现在的 [backend/app/core/constants.py](../../backend/app/core/constants.py) 里有 `validate_word()`，它只做了长度校验和空字符串校验。

但开局取词还有更多要求：

- 只能返回一个词
- 不能出现句号、逗号、空格、换行
- 最好是 2 到 6 个中文字符
- 如果你保留内部领域约束，则还要额外检查是否符合后端选择的领域

所以建议你新增一个更专门的校验函数，例如：

```python
def validate_start_word(word: str) -> str:
    normalized = word.strip()
    if len(normalized) == 0:
        raise ValueError("词不能为空")
    if len(normalized) < 2 or len(normalized) > 6:
        raise ValueError("开局目标词长度必须在 2 到 6 个字符之间")
    forbidden_chars = [" ", "\n", "\r", "\t", "。", "，", "！", "？", ",", ".", ";", ":"]
    if any(ch in normalized for ch in forbidden_chars):
        raise ValueError("目标词不能包含标点或空白字符")
    return normalized
```

### 9.2 为什么要分开普通词校验和开局词校验

因为两类词的规则不一样：

- 普通词：更偏通用长度检查
- 开局词：更偏游戏规则约束

分开以后，后续扩展更安全。

## 10. 失败处理与重试策略

### 10.1 模型返回空字符串怎么办

如果模型返回空字符串，建议：

1. 重试一次
2. 仍然失败就报错
3. 不把空值写入数据库

### 10.2 模型返回解释怎么办

如果模型返回了类似下面的内容：

- `答案是数据库，因为它属于计算机领域。`
- `我推荐“微积分”。`

那就说明 prompt 约束还不够硬，或者模型没有严格遵守规则。

这种情况可以：

- 先做字符串清洗
- 再做严格校验
- 必要时重新调用一次模型，并在 user prompt 里加强限制

### 10.3 推荐的重试提示

第二次调用时可以补一条更严格的用户消息：

```text
你刚才的输出不符合格式。
请只返回一个 2 到 6 个字的词语，不要标点，不要解释，不要换行。
```

### 10.4 为什么建议控制重试次数

因为 LLM 调用会产生：

- 成本
- 延迟
- 不确定性

开局阶段最多重试 1 到 2 次就够了，再多意义不大。

### 11.1 推荐游戏记录结构

```json

  "game_id": "uuid",

  "target_word_encrypted": "...",
  "target_word_source": "helper_llm"
}
```

### 11.2 为什么要存 `target_word_source`

因为后续排查问题时，你会想知道这个词是：

- 规则库选出来的
- LLM 生成的
- 人工指定的

记录来源后，调试会更容易。

### 11.3 为什么建议只存密文

因为目标词本身就是本局秘密。

只存密文可以降低：

- 明文泄露风险
- 日志误打风险
- 数据库直接可读风险

## 14. 最后要记住的边界

这一章最重要的边界只有一个：

**小助手模型负责生成词，Service 负责校验和落库，API 负责接收请求，Repository 负责保存数据。**

如果你把这四层混在一起，后面会很难维护。

如果你按这个边界写，后续扩展其他学科、其他难度、其他模型都很容易。
