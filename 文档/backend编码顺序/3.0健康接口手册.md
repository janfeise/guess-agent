# 3健康接口手册

## 1. 文档目的

这一章的目标是指导你实现后端的健康检查接口。它的作用不是承载业务，而是快速判断后端服务、配置类、MongoDB 连接初始化是否正常。

这一章完成后，你应该能够做到：

- 访问接口后立即知道服务是否在线
- 确认配置类已经被正确加载
- 确认 MongoDB 初始化流程已经执行
- 为后续 Repository、Service 和 Agent 开发提供基础验收点

健康接口是最适合作为 demo 阶段第一个可验证接口的模块，因为它既简单，又能覆盖很多基础问题。

## 2. 为什么先写健康接口

虽然你已经完成了配置类和 MongoDB 连接，但这些模块还没有经过完整的对外验证。健康接口的意义就在于把“内部可用”变成“外部可见”。

先写健康接口有几个好处：

- 可以快速确认 FastAPI 服务是否真正启动
- 可以验证配置类读取是否成功
- 可以验证 MongoDB 初始化是否成功
- 可以为 Docker 启动后的自动检查提供一个固定入口
- 可以在继续写 Repository 前先排除基础问题

如果健康接口都不稳定，后面的 repository 和业务接口会更难排查。

## 3. 健康接口的职责边界

健康接口只负责“状态返回”，不负责业务逻辑。

它应该做的事情：

- 返回服务名
- 返回服务状态
- 返回时间戳
- 返回数据库状态或数据库初始化状态

它不应该做的事情：

- 不执行具体业务逻辑
- 不查询游戏数据
- 不修改数据库内容
- 不调用 Agent 做推理

如果一个接口开始承担业务判断，它就不再是健康接口。

## 4. 推荐接口形式

建议先实现一个最小的健康接口：

- 路由：`GET /health`
- 作用：检查服务是否在线
- 响应：JSON

如果你想更明确地区分基础健康和数据库健康，也可以扩展为：

- `GET /health`：基础健康检查
- `GET /health/db`：数据库连通性检查

但 demo 阶段建议先做一个统一入口，避免接口过多。

## 5. 推荐返回结构

健康接口返回内容建议保持简单、稳定、可读。

推荐字段：

- `status`：服务状态，通常是 `ok`
- `service`：服务名
- `timestamp`：当前时间
- `database`：数据库状态，建议返回 `ready` 或 `not_ready`
- `details`：可选，便于调试

### 5.1 示例返回

```json
{
  "status": "ok",
  "service": "guess-agent-demo",
  "timestamp": "2026-04-11T10:30:00Z",
  "database": "ready"
}
```

### 5.2 设计原则

- 返回值不要太长
- 字段名称不要频繁变化
- 让前端或脚本能够稳定读取
- 服务异常时仍尽量返回可读错误信息

## 6. 健康接口要检查什么

demo 阶段建议至少检查以下三类内容：

### 6.1 服务本身是否在线

也就是 FastAPI 是否已经启动。

### 6.2 配置类是否可用

确认 `get_settings()` 可以正常读取配置，例如：

- `app_name`
- `app_port`
- `mongo_uri`
- `mongo_db_name`

### 6.3 MongoDB 是否已初始化

确认 `init_db()` 是否执行过，或者至少确认数据库连接对象能正常获取。

如果你后续要把健康接口做得更完整，也可以增加：

- 配置加载状态
- 数据库 ping 状态
- 路由可用状态

但 demo 阶段先别做太复杂。

## 7. 推荐实现位置

建议把健康接口放在 API 层，和后续业务接口保持一致。

推荐路径：

```bash
backend/
└── app/
    └── api/
        └── v1/
            └── health.py
```

如果你想更简化，也可以先暂时放在 `main.py`，但长期来看还是建议单独文件管理。

## 8. 推荐实现思路

健康接口的实现可以分成两步：

1. 读取配置
2. 检查数据库连接是否可用

### 8.1 配置读取

从配置对象中读取：

- 服务名
- 数据库名

### 8.2 数据库检查

可以用 MongoDB 的基础操作做一次连通性确认，比如：

- 获取数据库对象
- 执行一个轻量检查

demo 阶段不建议做重查询，只做轻量校验即可。

## 9. 推荐实现代码

下面是一份可以直接参考的健康接口实现示例。你可以把它作为 `backend/app/api/v1/health.py` 的参考写法。

```python
from datetime import datetime, timezone

from fastapi import APIRouter

from app.core.config import get_settings
from app.core.database import get_database

router = APIRouter(tags=["health"])


@router.get("/health")
async def health_check():
    settings = get_settings()
    database_status = "ready"

    try:
        database = get_database()
        await database.command("ping")
    except Exception:
        database_status = "not_ready"

    return {
        "status": "ok",
        "service": settings.app_name,
        "timestamp": datetime.now(timezone.utc).isoformat(),
        "database": database_status,
    }
```

## 10. 代码说明

### 10.1 为什么要在健康接口里读取配置

这样你可以立刻确认配置类是否真的已经接入主流程，而不是只存在于文件里。

### 10.2 为什么要做数据库 ping

因为 MongoDB 是否能成功连接，是 demo 阶段最常见的环境问题之一。健康接口里做一次轻量 ping，可以尽早发现问题。

### 10.3 为什么失败时不直接抛出异常

健康接口的目标是尽量给出可读状态，而不是让整个接口报错。

所以更适合把数据库状态标记为 `not_ready`，让调用方先知道问题，而不是直接中断。

## 11. 生命周期接入建议

健康接口本身不负责数据库初始化，它只负责验证结果。

数据库初始化仍然应该放在应用生命周期中，例如 FastAPI 的 startup 或 lifespan 阶段。

健康接口只做“检查”，不做“初始化”。

这种分工可以避免以下问题：

- 每次请求都重复初始化数据库
- 健康接口变成副作用接口
- 测试时难以判断真正的启动流程

## 12. 路由组织建议

建议你把健康接口和后续业务接口统一放在版本化路由下，例如：

- `/api/v1/health`

如果你想让外部监控更方便，也可以同时暴露一个简单的 `/health` 入口。

demo 阶段建议先二选一，不要过早增加复杂路由结构。

## 13. 健康接口的验收标准

你可以用下面几个标准判断健康接口是否写对了：

- 接口可以被正常访问
- 返回中包含服务名
- 返回中包含 `status: ok`
- MongoDB 正常时返回 `database: ready`
- MongoDB 异常时返回 `database: not_ready`
- 接口不会修改任何业务数据

## 14. 与 Docker 的配合方式

健康接口在 Docker 环境下尤其重要，因为它可以作为容器启动后的检查点。

建议你后续在 `docker-compose.yml` 中给 backend 服务加入健康检查时，也使用这个接口作为目标。

这样容器管理工具就能快速知道服务是否真正可用。

## 15. 常见问题

### 15.1 返回 `not_ready`

说明数据库连接没有成功，优先检查：

- MongoDB 容器是否启动
- `MONGO_URI` 是否正确
- `MONGO_DB_NAME` 是否正确
- `database.py` 是否成功复用配置

### 15.2 接口返回 500

优先检查：

- 路由是否正确注册
- 配置类是否正确加载
- MongoDB 客户端是否能创建

### 15.3 访问不到接口

优先检查：

- 路由前缀是否写对
- 应用是否已经把该 router include 进去
- 容器端口是否已映射

## 16. 这一章完成后再做什么

健康接口完成后，下一步建议写 Repository 层。

原因是：你已经有了稳定的连接和可验证入口，接下来最自然的事情就是围绕 games 集合封装数据读写。

## 18. 最终建议

健康接口的核心不是“看起来很简单”，而是“能替整个后端做最早、最稳定的验收”。

只要这一章写好，你后面写 Repository、Service 和 Agent 时，排错会轻松很多。
