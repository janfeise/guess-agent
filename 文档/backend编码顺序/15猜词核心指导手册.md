# 15 猜词核心指导手册

## 1. 文档目标

这份手册的目标不是“重新设计概念”，而是直接指导你把猜词游戏的核心闭环做出来：

- 用户输入之后，系统能判断这是提问、猜测，还是无效输入
- 用户提问时，系统能给出稳定的回答
- 用户猜测时，系统能判断是否猜中，并决定是否结束游戏
- Agent 能在合适的时候主动提问或主动猜测
- 整个流程由程序控制状态，LLM 只负责理解和决策

一句话总结：

**LLM 负责“说该做什么”，程序负责“决定是否执行，以及什么时候切状态”。**

---

## 2. 当前实现和目标模型对齐

在开始实现前，先把现状和目标对齐，不然很容易把 LLM 当成“万能控制器”，最后整个流程不可控。

### 2.1 当前已经完成的部分

当前仓库里，以下能力已经可用：

- [backend/app/agents/utils/prompt_loader.py](../../backend/app/agents/utils/prompt_loader.py) 已经能按文件读取 prompt
- [backend/app/repositories/game_repository.py](../../backend/app/repositories/game_repository.py) 已经能创建、查询、更新和结束游戏
- [backend/app/core/config.py](../../backend/app/core/config.py) 已经把主模型、小助手模型、MongoDB 和加密配置统一起来

### 2.2 当前缺口

下面这些能力还没有真正落地，是这份手册要解决的重点：

- `submit_turn.py` 里 `agent=None`，问答接口无法真正调用 LLM
- `GuessAgent` 目前还是纯文本问答，没有结构化 JSON 输出
- 没有用户意图识别，无法稳定判断是提问、猜测还是无效输入
- 没有 Agent 自主决策逻辑，无法决定该提问还是猜测
- 没有统一的状态机，`active / finished` 还不够表达游戏过程
- 没有明确的游戏结束规则，无法稳定完成“猜中即结束”

### 2.3 目标状态

实现完成后，建议形成下面这个闭环：

```text
用户输入
  -> LLM 识别意图
  -> 程序决定流程
  -> 如果是提问，LLM 生成回答
  -> 如果是猜测，程序判断是否猜中
  -> 如果游戏继续，Agent 决定 ask 或 guess
  -> 如果 Agent 猜测，等待用户确认
  -> 若确认正确，结束游戏
```

## 3. 核心设计原则

### 3.1 不要让 LLM 控制流程

LLM 只适合做：

- 理解用户输入
- 生成自然语言回答
- 生成下一步行动建议
- 生成猜测词或提问

LLM 不适合做：

- 直接修改数据库
- 直接决定整个状态机如何跳转
- 直接承担“程序是否结束”的职责

### 3.2 程序必须拥有最终控制权

下面这些判断应该由程序决定：

- 游戏是否已结束
- 当前轮次是谁的回合
- 用户猜测是否与答案一致
- Agent 的动作是否被接受
- 状态是否写回数据库

### 3.3 用户猜测的正确性优先使用确定性判断

这是一个很关键的实现原则：

- 用户猜测是否正确，优先用程序做字符串比较
- 如果后续你想支持同义词、别名、近义表达，再考虑让 LLM 做辅助判断
- 默认不要让 LLM 代替程序做“猜中判断”，否则会带来不稳定性

### 3.4 每一轮只处理一个明确动作

核心原则是“一轮一动作”：

- 用户提问
- 用户猜测
- Agent 提问
- Agent 猜测
- 用户确认 Agent 猜测是否正确

不要把多个动作揉在一轮里，否则历史记录、状态流转和前端展示都会变乱。

## 4. 游戏状态机设计

### 4.1 建议的状态字段

当前数据库里已经有 `status` 和 `metadata`，建议这样用：

- `status`：只表示大状态，取值建议为 `active`、`finished`
- `metadata.phase`：表示当前流程阶段
- `metadata.last_actor`：上一动作是谁执行的
- `metadata.last_intent`：上一轮用户意图
- `metadata.awaiting_judgement`：是否正在等待用户确认 Agent 猜测
- `metadata.agent_confidence`：Agent 猜测置信度
- `metadata.pending_guess`：Agent 当前猜测词

建议阶段设计如下：

- `user_turn`：等待用户输入
- `agent_turn`：等待 Agent 决策
- `awaiting_judgement`：等待用户判断 Agent 猜测是否正确
- `finished`：游戏结束

### 4.2 推荐的最小状态模型

如果你不想一次改很多字段，可以先保持 `status` 不变，只在 `metadata` 里增加阶段信息：

```python
metadata = {
    "phase": "user_turn",
    "last_actor": "system",
    "last_intent": None,
    "awaiting_judgement": False,
    "pending_guess": "",
    "agent_confidence": 0.0,
}
```

### 4.3 核心流程图

```text
创建游戏
  -> phase = user_turn

用户输入
  -> LLM 判断 intent
  -> 如果 invalid：返回提示，phase 不变
  -> 如果 question：生成回答，写历史，phase = agent_turn
  -> 如果 guess：程序比对答案
      -> 猜中：status = finished
      -> 猜错：写历史，phase = agent_turn

Agent 回合
  -> LLM 决策 ask 或 guess
  -> 如果 ask：输出问题，写历史，phase = user_turn
  -> 如果 guess：输出猜测词，写历史，phase = awaiting_judgement

用户确认 Agent 猜测
  -> 如果正确：status = finished
  -> 如果错误：写历史，phase = user_turn
```

### 4.4 为什么需要 `awaiting_judgement`

因为 Agent 猜测不是最终答案，而是“系统向用户提交一个猜测”。

这个状态表示：

- 系统已经给出猜测
- 现在轮到用户确认对错
- 如果用户确认正确，游戏结束
- 如果用户确认错误，回到用户提问阶段

## 5. LLM 结构化 JSON 协议

这一部分是整份文档最核心的实现约束。

### 5.1 用户意图解析协议

用户输入先经过意图解析，返回一个结构化结果：

```json
{
  "intent": "question | guess | invalid",
  "normalized_text": "用户输入的标准化文本",
  "guess_word": "如果是猜测则提取出的词",
  "answer": "yes | no | unknown",
  "confidence": 0.0,
  "reason": "简短理由"
}
```

说明：

- `intent` 是最重要的字段
- `normalized_text` 用于清洗空格、标点和无关前缀
- `guess_word` 只有在 `guess` 时有效
- `answer` 适合在提问场景下由模型生成 yes / no / unknown
- `confidence` 用于记录解析置信度，不是强制字段，但建议保留
- `reason` 只做调试，不要给前端过多展示

### 5.2 Agent 决策协议

Agent 决定下一步动作时，也应该输出 JSON：

```json
{
  "action": "ask | guess",
  "question": "如果 action=ask，则这里放问题",
  "guess_word": "如果 action=guess，则这里放猜测词",
  "confidence": 0.0,
  "reason": "简短理由"
}
```

说明：

- `ask` 表示继续发问
- `guess` 表示认为信息已经足够，直接猜答案
- `confidence` 建议取 0 到 1 之间的浮点数
- `question` 和 `guess_word` 只能二选一生效

### 5.3 用户问题回答协议

当用户输入是提问时，系统应该返回一个稳定答案：

```json
{
  "answer": "yes | no | unknown",
  "reason": "简短理由"
}
```

说明：

- 这里的 `answer` 要尽量严格，只允许 `yes`、`no`、`unknown`
- 不要让模型输出长篇解释
- 不要让模型泄露答案明文

### 5.4 解析失败时的兜底规则

任何 JSON 输出都要有兜底：

- JSON 解析失败时，重新请求一次模型
- 第二次仍失败时，进入保守降级
- 降级策略建议：把输入视为 `invalid`，要求用户重新输入

建议统一降级响应：

```json
{
  "intent": "invalid",
  "reason": "模型输出格式不合法，请重试"
}
```

## 6. 推荐的实现顺序

这部分是你真正动手实现时最重要的顺序。

### 6.1 第一步：先把数据结构定义清楚

建议先补两个 Pydantic 模型：

- 用户意图解析结果
- Agent 决策结果

示例：

```python
from pydantic import BaseModel, Field
from typing import Literal


class UserIntentResult(BaseModel):
    intent: Literal["question", "guess", "invalid"]
    normalized_text: str = ""
    guess_word: str = ""
    answer: Literal["yes", "no", "unknown"] = "unknown"
    confidence: float = Field(default=0.0, ge=0.0, le=1.0)
    reason: str = ""


class AgentDecisionResult(BaseModel):
    action: Literal["ask", "guess"]
    question: str = ""
    guess_word: str = ""
    confidence: float = Field(default=0.0, ge=0.0, le=1.0)
    reason: str = ""
```

### 6.2 第二步：重写 prompt 文件

建议把 prompt 拆成三类：

- `turn_router.txt`：负责识别用户输入是提问、猜测还是无效
- `guess_system.txt`：负责用户问题回答和主模型规则
- `agent_decision.txt`：负责 Agent 决策 ask 或 guess
- `helper_system.txt`：负责开局生成词与知识辅助

如果你不想马上增加文件数量，也可以先把这些 prompt 写在 `guess_system.txt` 内部分段管理，但长期来看建议拆开。

### 6.3 第三步：给 `GuessAgent` 增加结构化方法

建议增加下面这些方法：

- `parse_user_intent()`：解析用户输入
- `answer_question()`：回答用户是/否问题
- `decide_agent_action()`：决定 ask 还是 guess
- `generate_agent_guess()`：生成猜测词
- `generate_agent_question()`：生成下一问

### 6.4 第四步：改造 `GameService.submit_turn()`

`GameService.submit_turn()` 不要只做“把问题丢给模型”。

它应该负责：

- 读取游戏状态
- 判断当前 phase
- 调用意图解析
- 处理 question / guess / invalid
- 根据 Agent 决策切换状态
- 写回数据库
- 返回给前端一个清晰的结果

### 6.5 第五步：补齐 repository 的状态更新能力

建议增加一个统一的更新方法，专门更新：

- `history`
- `summary`
- `round_count`
- `metadata.phase`
- `metadata.pending_guess`
- `metadata.awaiting_judgement`
- `updated_at`

### 6.6 第六步：最后再补测试

测试顺序建议是：

- 用户输入是正常提问
- 用户输入是正常猜测
- 用户输入是无效输入
- 用户猜中答案
- 用户猜错答案
- Agent 决定 ask
- Agent 决定 guess
- Agent 猜测正确
- Agent 猜测错误
- 游戏结束后不允许继续提交

### 7.4 `GuessAgent` 方法骨架

```python
from pydantic import BaseModel


class GuessAgent:
    async def parse_user_intent(self, user_input: str, history: list[dict]) -> dict:
        messages = [
            {"role": "system", "content": self.prompt_loader.load_prompt("turn_router")},
            {"role": "user", "content": user_input},
        ]
        response = await self.agent_model.ainvoke(messages)
        return self._parse_json_response(response.content)

    async def answer_question(self, question: str, answer: str) -> dict:
        messages = [
            {"role": "system", "content": self.prompt_loader.load_prompt("guess_system")},
            {"role": "system", "content": f"真实答案：{answer}"},
            {"role": "user", "content": question},
        ]
        response = await self.agent_model.ainvoke(messages)
        return self._parse_json_response(response.content)

    async def decide_agent_action(self, history: list[dict], summary: str) -> dict:
        messages = [
            {"role": "system", "content": self.prompt_loader.load_prompt("agent_decision")},
            {"role": "user", "content": f"历史对话：{history}\n\n摘要：{summary}"},
        ]
        response = await self.agent_model.ainvoke(messages)
        return self._parse_json_response(response.content)
```

### 7.5 `GameService.submit_turn()` 的推荐骨架

```python
class GameService:
    async def submit_turn(self, game_id: str, question: str, mode: str = "agent"):
        game = await self.repository.get_game(game_id)
        if not game:
            raise ValueError(f"Game with id {game_id} not found")

        if game.get("status") == "finished":
            raise ValueError(f"Game with id {game_id} has already finished")

        metadata = game.get("metadata", {})
        phase = metadata.get("phase", "user_turn")
        history = game.get("history", [])
        round_count = game.get("round_count", 0)

        if phase == "awaiting_judgement":
            return await self._handle_agent_guess_judgement(game, question)

        intent_result = await self.agent.parse_user_intent(question, history)

        if intent_result["intent"] == "invalid":
            return {
                "game_id": game_id,
                "status": game.get("status"),
                "phase": phase,
                "message": "请输入有效的提问或猜测",
                "history": history,
            }

        if intent_result["intent"] == "guess":
            return await self._handle_user_guess(game, intent_result)

        answer = await self._answer_user_question(game, question)
        updated_history = history + [{
            "round_no": round_count + 1,
            "actor": "user",
            "turn_type": "question",
            "input_text": question,
            "answer_text": answer["answer"],
            "result": answer.get("reason", ""),
        }]

        agent_action = await self.agent.decide_agent_action(
            updated_history,
            self.memory_policy.build_summary(updated_history),
        )

        if agent_action["action"] == "guess":
            updated_history.append({
                "round_no": round_count + 2,
                "actor": "agent",
                "turn_type": "agent_guess",
                "input_text": agent_action["guess_word"],
                "answer_text": "awaiting_judgement",
                "result": f"confidence={agent_action.get('confidence', 0.0)}",
            })
            await self.repository.update_game_state(
                game_id,
                history=updated_history,
                summary=self.memory_policy.build_summary(updated_history),
                round_count=len(updated_history),
                phase="awaiting_judgement",
                pending_guess=agent_action["guess_word"],
                awaiting_judgement=True,
                agent_confidence=agent_action.get("confidence", 0.0),
            )
            return {
                "game_id": game_id,
                "phase": "awaiting_judgement",
                "agent_guess": agent_action["guess_word"],
                "history": updated_history,
                "message": "系统已给出猜测，请判断是否正确",
            }

        updated_history.append({
            "round_no": round_count + 2,
            "actor": "agent",
            "turn_type": "agent_question",
            "input_text": agent_action["question"],
            "answer_text": "",
            "result": f"confidence={agent_action.get('confidence', 0.0)}",
        })
        await self.repository.update_game_state(
            game_id,
            history=updated_history,
            summary=self.memory_policy.build_summary(updated_history),
            round_count=len(updated_history),
            phase="user_turn",
            awaiting_judgement=False,
        )
        return {
            "game_id": game_id,
            "phase": "user_turn",
            "question": agent_action["question"],
            "history": updated_history,
        }
```

### 7.6 用户猜测的正确性判断

推荐做法是程序优先比较，不要把这个动作完全交给模型：

```python
def normalize_guess_text(text: str) -> str:
    return text.strip().lower()


def is_guess_correct(user_guess: str, target_word: str) -> bool:
    return normalize_guess_text(user_guess) == normalize_guess_text(target_word)
```

如果你仍然保留数据库中的明文字段，可先做兼容；但长期建议只从加密字段解密得到目标词：

```python
from app.core.security import decrypt


def reveal_target_word(game_doc: dict, settings) -> str:
    system_password = decrypt(game_doc["system_password_encrypted"], settings.encryption_secret)
    target_word = decrypt(game_doc["system_word_encrypted"], system_password)
    return target_word
```

### 7.8 处理 Agent 猜测后的用户确认

```python
async def _handle_agent_guess_judgement(self, game: dict, user_input: str):
    pending_guess = game.get("metadata", {}).get("pending_guess", "")
    target_word = reveal_target_word(game, self.settings)

    judgement_text = user_input.strip().lower()
    is_correct = judgement_text in {"对", "正确", "是", "yes", "true"}

    if is_correct:
        now = datetime.now()
        await self.repository.finish_game(
            game["game_id"],
            reason="agent_guess_confirmed",
            finished_at=now,
        )
        return {
            "game_id": game["game_id"],
            "status": "finished",
            "agent_guess": pending_guess,
            "message": "系统猜测正确，游戏结束",
        }

    history = game.get("history", []) + [{
        "round_no": game.get("round_count", 0) + 1,
        "actor": "user",
        "turn_type": "judge_agent_guess",
        "input_text": user_input,
        "answer_text": "wrong",
        "result": f"pending_guess={pending_guess}",
    }]

    await self.repository.update_game_state(
        game["game_id"],
        history=history,
        summary=self.memory_policy.build_summary(history),
        round_count=len(history),
        phase="user_turn",
        awaiting_judgement=False,
        pending_guess="",
    )

    return {
        "game_id": game["game_id"],
        "status": "active",
        "message": "系统猜错了，请继续提问",
        "history": history,
    }
```

## 8. Repository 建议

当前 `GameRepository` 已经能做基础更新，但建议补一个统一更新状态的方法，这样 `GameService` 会更干净。

### 8.1 推荐的更新方法

```python
async def update_game_state(
    self,
    game_id: str,
    history: list[dict],
    summary: str,
    round_count: int,
    phase: str,
    awaiting_judgement: bool = False,
    pending_guess: str = "",
    agent_confidence: float = 0.0,
):
    result = await self.collection.update_one(
        {"game_id": game_id},
        {
            "$set": {
                "history": history,
                "summary": summary,
                "round_count": round_count,
                "metadata.phase": phase,
                "metadata.awaiting_judgement": awaiting_judgement,
                "metadata.pending_guess": pending_guess,
                "metadata.agent_confidence": agent_confidence,
                "updated_at": datetime.utcnow(),
            }
        }
    )

    if result.matched_count == 0:
        raise ValueError(f"Game with id {game_id} not found")

    return result
```

### 8.2 结束游戏时建议补充的字段

`finish_game()` 除了把 `status` 改成 `finished`，建议同时写入：

- `finished_at`
- `finish_reason`
- `metadata.phase = finished`
- `metadata.awaiting_judgement = False`

这样前端和后端都更容易判断是否已结束。

## 9. API 层建议

### 9.1 `submit_turn.py` 必须注入 Agent

当前问答接口里 `agent=None`，这是必须优先修复的断点。

文档层面的要求是：

- `get_game_service()` 必须和 `games.py` 保持一致
- 不能让问答接口失去 LLM 能力
- API 层只做参数接收和异常转换，不做业务判断

### 9.2 请求模型建议

如果你要把“用户输入”和“用户确认 Agent 猜测”统一到一个接口，建议把请求体扩展为：

```python
from pydantic import BaseModel, Field
from typing import Literal


class SubmitTurnRequest(BaseModel):
    turn_type: Literal["input", "judge"] = Field(default="input")
    content: str | None = Field(default=None, description="用户输入内容")
    judgement: bool | None = Field(default=None, description="用户是否确认系统猜测正确")
    mode: str = Field(default="agent", description="回答模式")
```

推荐规则：

- `turn_type = input` 时，`content` 必填
- `turn_type = judge` 时，`judgement` 必填
- 这样既能覆盖普通提问，也能覆盖用户确认系统猜测

如果你想先保持最小改动，也可以先沿用 `question` 字段，只是内部把它当成通用输入。

## 11. 回合历史建议

当前 `history` 只保存 `question / answer`，这对后续判断不够。

建议升级成更丰富的结构：

```json
{
  "round_no": 3,
  "actor": "user | agent",
  "turn_type": "question | guess | agent_question | agent_guess | judge_agent_guess | invalid",
  "input_text": "原始输入",
  "answer_text": "系统输出",
  "result": "简要结果",
  "created_at": "时间戳"
}
```

这样做的好处：

- 回放更清晰
- 调试更容易
- 后续统计更容易
- Agent 的上下文也更稳定

## 12. 安全边界建议

### 12.1 不要把答案明文返回给前端

开局时和游戏过程中，都不要把系统答案明文暴露给前端。

### 12.2 不要在日志里打印完整答案

调试日志可以打印游戏 ID、阶段和结果，但不要打印完整答案。

### 12.3 不要让 LLM 直接看到不该暴露的内容

如果某些提示词里需要带答案，应该只在服务内部拼接，不能把相关字段传到客户端。

实现完成后，建议按下面的清单检查。

### 13.1 基础流程

- [ ] 能创建游戏
- [ ] 创建后 `status = active`
- [ ] 创建后 `metadata.phase = user_turn`
- [ ] 能读取当前局历史
- [ ] 能更新回合数和摘要

### 13.2 用户输入识别

- [ ] 用户输入能被区分为 question
- [ ] 用户输入能被区分为 guess
- [ ] 用户输入能被区分为 invalid
- [ ] 模型输出 JSON 失败时有兜底

### 13.3 用户提问

- [ ] 用户问的是/否问题时，系统能返回简短答案
- [ ] 提问后历史能写入数据库
- [ ] 提问后 phase 能切到 agent_turn

### 13.4 用户猜测

- [ ] 用户猜中时，游戏结束
- [ ] 用户猜错时，系统继续进入下一步
- [ ] 猜测是否正确由程序判断，而不是完全交给 LLM

### 13.5 Agent 行为

- [ ] Agent 能决定 ask 或 guess
- [ ] Agent 决定 ask 时，系统能返回一个问题
- [ ] Agent 决定 guess 时，系统能进入 awaiting_judgement
- [ ] 用户确认 Agent 猜测后，状态能正确结束或回退

### 13.6 接口和持久化

- [ ] `submit_turn.py` 注入 Agent 正确
- [ ] `history`、`summary`、`round_count` 能正确更新
- [ ] 结束时 `finished_at`、`finish_reason` 能写入
- [ ] 不会因为空输入或非法 JSON 直接崩溃

## 14. 最后建议

这部分实现的核心不是“让 LLM 更聪明”，而是“让流程更稳定”。

只要记住下面这句就不会跑偏：

**LLM 负责理解和建议，程序负责执行和收口。**
