# 14 学科池与动态提示词实现手册

## 1. 文档目标

这一章用于指导你把“开局词生成”从普通的生活化词汇，升级成更有学科维度、更有语义空间的目标词生成流程。

本章要解决的核心问题是：

- 不要总是生成“苹果”“汽车”这类生活琐碎词汇
- 让小助手在开局时更倾向生成学术感更强、抽象度更高的词语
- 把“学科池”独立维护，避免散落在代码里
- 每次开局从学科池里随机抽取三个学科，作为本局约束
- 根据这三个学科动态拼装提示词，让模型按约束生成目标词

这一章只讲“开局取词”的 prompt 设计，不展开后续问答链路。

### 2.1 当前链路的关键点

当前开局流程已经走到：

- `GameService.create_game()` 调用小助手生成开局词
- `GuessAgent.generate_start_word()` 负责拼接 prompt 并调用 helper 模型
- `PromptLoader.load_prompt()` 负责读取 prompt 文件
- `game_start.txt` 负责约束模型“只返回一个词”

也就是说，你现在要做的不是重构整条链路，而是把“开局词生成”这一步的约束变得更细：

- 词汇来源更学术
- 学科覆盖更随机
- 提示词更动态

## 3. 目标实现效果

实现完成后，每次创建游戏时，后端应该做下面这些事：

1. 从独立的学科池中随机选出三个学科
2. 根据这三个学科动态拼出一段开局提示词
3. 把“学科维度”和“语义空间”写入 prompt
4. 让模型生成一个更偏知识型、概念型、学术型的目标词
5. 对返回值做清洗和校验
6. 将结果继续交给后续的加密与落库流程

### 3.1 什么叫“学科维度”

这里的“学科维度”不是让模型随便联想，而是给它明确的知识边界，例如：

- 计算机科学
- 数学
- 物理
- 历史
- 语言学
- 经济学
- 心理学
- 生物学

模型会在这些维度里选词，而不是回到“水果”“家具”“日用品”这种低信息密度词汇。

### 3.2 什么叫“语义空间”

“语义空间”是为了告诉模型：

- 词语应该属于概念层、术语层、机制层、结构层
- 不要选太口语化、太具体、太生活化的词
- 词语应更接近学科对象、理论名词、抽象概念、专业名词

例如：

- 更合适：`向量`, `熵`, `范式`, `协议`, `演化`, `公理`
- 不够合适：`苹果`, `手机`, `椅子`, `外卖`, `牙刷`

---

## 4. 推荐的实现拆分

这部分建议拆成三层做：

- 学科池配置层
- 学科抽样层
- 动态提示词组装层

### 4.1 学科池配置层

学科池建议独立维护，不要直接散落在 `GuessAgent` 方法里。

推荐两种放置方式：

- 放在 `backend/app/core/constants.py`
- 或者单独新建一个配置文件，例如 `backend/app/core/subject_pool.py`

如果后续还要继续扩展，推荐单独文件维护，原因是：

- 更容易新增和删除学科
- 更容易做单元测试
- 更容易让非核心逻辑与 agent 逻辑分离

### 4.2 学科抽样层

每次开局都应该从学科池里随机抽取三个学科。

建议满足这些要求：

- 每次随机性都要重新抽样
- 尽量避免三个学科重复
- 如果学科池数量不足 3，要直接报错或降级处理
- 抽样结果要可复用给 prompt 组装函数

### 4.3 动态提示词组装层

根据抽到的三个学科，动态生成用户 prompt。

这段 prompt 应该至少包含：

- 当前任务是生成目标词
- 词语必须具备学科感
- 词语需要落在某种语义空间里
- 词语必须只返回一个
- 不要解释，不要附加说明
- 可选：限定长度或词性

## 5. 学科池设计建议

### 5.1 学科池的内容原则

学科池不要只放“学科名称”，最好同时考虑“学科语义边界”。

例如可以先从这些维度入手：

- 计算机科学
- 数学
- 物理
- 化学
- 生物学
- 医学
- 经济学
- 心理学
- 社会学
- 历史学
- 哲学
- 语言学
- 法学
- 地理学
- 天文学

### 5.2 推荐的数据结构

推荐直接用列表维护：

```python
SUBJECT_POOL = [
    "计算机科学",
    "数学",
    "物理",
    "化学",
    "生物学",
    "医学",
    "经济学",
    "心理学",
    "社会学",
    "历史学",
    "哲学",
    "语言学",
    "法学",
    "地理学",
    "天文学",
]
```

如果你想让 prompt 更强，还可以用字典结构，把学科和语义提示一起维护：

```python
SUBJECT_POOL = [
    {"name": "数学", "hint": "抽象结构、定理、公理、函数、空间"},
    {"name": "计算机科学", "hint": "算法、协议、模型、架构、状态"},
    {"name": "历史学", "hint": "制度、王朝、文明、史料、年代"},
]
```

这样做的好处是：

- prompt 更容易写得具体
- 学科约束更稳定
- 便于后续做更强的语义控制

## 6. 随机选择三个学科

### 6.1 选择规则

每次生成开局词时，从学科池中随机选择三个学科。

推荐规则：

- 使用不重复抽样
- 抽样结果按随机顺序保留
- 不要在前端传入固定学科，避免开局过于可预测

### 6.2 推荐实现思路

可以在 `GuessAgent` 里增加一个私有方法，例如：

```python
import random

class GuessAgent:
    START_WORD_SUBJECT_POOL = [
        "数学",
        "物理",
        "计算机科学",
        "历史学",
        "经济学",
        "心理学",
        "语言学",
        "哲学",
        "法学",
        "生物学",
    ]

    def _pick_start_word_subjects(self) -> list[str]:
        if len(self.START_WORD_SUBJECT_POOL) < 3:
            raise ValueError("学科池数量不足，至少需要 3 个学科")
        return random.sample(self.START_WORD_SUBJECT_POOL, 3)
```

### 6.3 为什么要用 `random.sample`

因为它天然就是“无重复抽样”，更符合“随机选三个学科”的需求。

如果你用的是 `choice` 循环，容易出现重复学科，效果会差一些。

## 7. 动态提示词设计

### 7.1 推荐的 prompt 结构

建议把 prompt 分成三段：

- 任务说明
- 学科维度
- 语义空间和输出限制

### 7.2 推荐的用户提示词模板

示例如下：

```text
请根据以下约束生成一个适合猜词游戏的目标词。

学科维度：数学、物理、计算机科学
语义空间要求：优先选择属于概念、理论、机制、结构、方法、定律、模型的词语，避免生活用品、食物、交通工具等低信息密度词汇。

生成要求：
1. 只能返回一个中文词语
2. 词语应具备一定学术感和知识密度
3. 不要输出解释、不要输出列表、不要输出标点
4. 词语长度建议为 2~6 个中文字符
```

### 7.3 为什么要写“语义空间”

只写“学科维度”还不够，因为模型可能仍然生成一些过于泛化的词。

加入“语义空间”后，模型更容易理解你要的是：

- 抽象概念
- 学术术语
- 知识对象
- 理论或机制相关词汇

而不是日常高频词。

### 7.4 推荐把约束写成固定模板 + 动态变量

建议模板化处理，而不是每次在代码里拼很多散乱字符串。

例如：

```python
user_prompt = f"""
请根据以下约束生成一个适合猜词游戏的目标词。

学科维度：{subject_a}、{subject_b}、{subject_c}
语义空间要求：优先选择属于概念、理论、机制、结构、方法、定律、模型的词语，避免生活用品、食物、交通工具等低信息密度词汇。

生成要求：
1. 只能返回一个中文词语
2. 词语应具备一定学术感和知识密度
3. 不要输出解释、不要输出列表、不要输出标点
4. 词语长度建议为 2~6 个中文字符
""".strip()
```

## 8. 推荐的代码落点

### 8.1 `guess_agent.py`

这里是最适合改的地方。

建议在 `GuessAgent` 中增加以下能力：

- 学科池常量
- 学科抽样方法
- 动态 prompt 组装方法
- `generate_start_word()` 内部调用这些方法

推荐结构：

```python
class GuessAgent:
    START_WORD_SUBJECT_POOL = [...]

    def _pick_start_word_subjects(self) -> list[str]:
        ...

    def _build_start_word_prompt(self, difficulty: str | None, subjects: list[str]) -> str:
        ...

    async def generate_start_word(self, difficulty: str | None = None, subject: str | None = None) -> str:
        ...
```

### 8.2 `game_start.txt`

这个文件仍然保留，但它的角色会变化：

- 负责定义通用的系统规则
- 不负责写死具体学科
- 不负责写死具体词表

系统 prompt 应该更像“总规则”，而动态约束放在 user prompt 里。

### 8.3 `game_service.py`

服务层尽量不关心 prompt 细节，只负责调度：

- 调用 `agent.generate_start_word()`
- 取得目标词
- 继续做校验、加密、落库

也就是说，学科池和 prompt 组装最好不要跑到 service 层。

## 9. 推荐实现步骤

### 第一步：独立维护学科池

把学科池单独放到一个文件里，或者至少集中到一个常量位置。

目标是让“学科列表”成为一个可维护的配置，而不是散在方法里的魔法字符串。

### 第二步：实现随机三学科抽样

增加一个抽样函数，每次返回三个不重复学科。

如果你希望结果可测试，可以把随机函数抽象出来，方便单测时注入固定值。

### 第三步：改造动态 prompt

把 `generate_start_word()` 改成：

- 先拿到三个学科
- 再拼接成动态 prompt
- 再调用 helper 模型

### 第四步：收紧输出约束

在 prompt 中增加以下限制：

- 只返回一个词
- 优先学术词
- 禁止生活化词汇
- 禁止解释

### 第五步：保留校验层

即使 prompt 已经约束了，也要继续保留 `validate_start_word()` 或同类校验逻辑。

原因很简单：

- prompt 只能提高概率，不能保证 100% 稳定
- 校验层才是最终防线

## 10. 推荐伪代码

下面给出一个更完整的实现示意：

```python
import random

class GuessAgent:
    START_WORD_SUBJECT_POOL = [
        "数学",
        "物理",
        "计算机科学",
        "历史学",
        "经济学",
        "心理学",
        "语言学",
        "哲学",
        "法学",
        "生物学",
    ]

    def _pick_start_word_subjects(self) -> list[str]:
        if len(self.START_WORD_SUBJECT_POOL) < 3:
            raise ValueError("学科池数量不足，至少需要 3 个学科")
        return random.sample(self.START_WORD_SUBJECT_POOL, 3)

    def _build_start_word_prompt(self, difficulty: str | None, subjects: list[str]) -> str:
        subject_text = "、".join(subjects)
        return f"""
请根据以下约束生成一个适合猜词游戏的目标词。

难度：{difficulty or "medium"}
学科维度：{subject_text}
语义空间要求：优先选择属于概念、理论、机制、结构、方法、定律、模型的词语，避免生活用品、食物、交通工具等低信息密度词汇。

生成要求：
1. 只能返回一个中文词语
2. 词语应具备一定学术感和知识密度
3. 不要输出解释、不要输出列表、不要输出标点
4. 词语长度建议为 2~6 个中文字符
""".strip()

    async def generate_start_word(self, difficulty: str | None = None, subject: str | None = None) -> str:
        system_prompt = self.prompt_loader.load_prompt("game_start")
        subjects = self._pick_start_word_subjects()
        user_prompt = self._build_start_word_prompt(difficulty, subjects)
        messages = [
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt},
        ]
        response = await self.helper_model.ainvoke(messages)
        return self._normalize_answer(response.content)
```

### 10.1 关于 `subject` 参数

如果你已经决定“学科由后端随机选取”，那 `generate_start_word()` 里的 `subject` 参数就可以不再作为外部输入使用。

如果这个参数暂时还要保留，建议把它当成兼容字段，而不是主要逻辑入口。

## 11. 推荐的校验规则

为了避免模型偶尔生成不合适的词，建议在返回前再做一次校验：

- 不能为空
- 不能太短或太长
- 不能包含句号、逗号、冒号等句式痕迹
- 不能出现明显生活化高频词
- 不能是明显的动词短语或句子

如果校验失败，可以：

- 重新抽样一次学科再生成
- 或者直接返回兜底错误，提示小助手暂时不可用

---

## 12. 测试重点

完成实现后，建议至少验证下面几类结果：

- 连续生成 20 次，是否能看到明显不同的学科倾向
- 是否仍然会频繁出现生活化词汇
- 是否能稳定返回单个词
- 学科池数量不足时是否会报错
- prompt 是否真正随每次请求变化

### 12.1 你应该观察的现象

理想状态下，生成词会更偏向：

- 概念名词
- 学术术语
- 抽象对象
- 专业对象

而不是：

- 水果名
- 交通工具名
- 日用品名
- 明显低信息密度词

---

## 13. 最终建议

如果你只做最小改动，优先顺序应该是：

1. 先把学科池独立出来
2. 再做随机三个学科抽样
3. 最后改动态 prompt

这样做的好处是：

- 改动面小
- 好验证
- 方便后续继续增强语义控制

如果你后续还想把“学科维度”做得更强，可以继续升级成：

- 学科 + 子领域双层约束
- 学科权重抽样
- 难度与学科联动
- 按学科分别维护词汇约束模板
