# 9 开局知识词加密手册

## 1. 文档目标

这一章的目标是把“游戏开局即完成知识词选择与答案加密提交”这条主链路定义清楚，作为后续实现的正式指导文档。

本阶段的核心不是“学科选择”，而是“知识词选择”：

- 用户和猜词 Agent 在开局时分别确定一个 10 字以内的知识词或知识点
- 系统在后端完成答案加密
- 知识词信息和加密结果固化到 MongoDB
- 后续回合只基于这份初始化状态继续推进

这一章只聚焦后端初始化链路，不做前端联调，也不展开复杂结算逻辑。

---

## 2. 为什么要单独做这一章

当前项目已经完成了以下能力：

- FastAPI 入口
- MongoDB 连接
- 创建游戏接口
- 问答接口
- Agent 基础结构
- Repository 基础结构

但目前还缺少一个明确的“开局定义”：

- 用户到底选什么
- Agent 怎么定自己的知识词
- 答案什么时候加密
- 加密后存到哪里
- 后续问答如何依赖这份开局状态

如果不开局阶段先定义清楚，后续问答、验证、结算都会互相牵扯，导致结构混乱。

---

## 3. 本阶段的概念定义

### 3.1 知识词

这里的“知识词”指的是：

- 长度不超过 10 个字符
- 可以是概念、名词、术语、短词组
- 例如：反射、蛋白质、数据库、微积分

它不是“学科大类”，也不是“课程名称”。

### 3.2 用户知识词

用户知识词是用户在游戏开始时输入或确认的目标词。

这个词将作为用户方的谜底或目标对象，在整局游戏中保持不变。

### 3.3 Agent 知识词

Agent 知识词是由系统或 Agent 自身在开局时确定的谜底。

这个词同样要在游戏开始时固化，后续问答过程中不能随意变化。

### 3.4 加密结果

加密结果指的是将知识词或答案以密文形式保存到数据库中，避免明文长期出现在持久化层。

---

## 4. 本阶段要解决的核心问题

### 4.1 开局时到底要做什么

开局时至少要完成以下动作：

1. 校验用户知识词是否符合长度约束
2. 为用户侧或系统侧准备知识词
3. 生成并保存加密结果
4. 初始化游戏主记录
5. 进入回合问答状态

### 4.2 为什么要限制 10 字以内

限制长度的原因主要是：

- 保持规则简单
- 方便前端输入和展示
- 便于 Agent 在提示词中稳定处理
- 降低知识词歧义

### 4.3 为什么不再叫“学科选择”

因为这个项目并不是传统意义上的“只能选一门学科”，而是允许用户选择更具体的知识点、概念词或短词组。

所以这里统一称为：

- 知识词选择

而不是：

- 学科选择

## 6. 推荐的开局流程

### 6.1 用户侧流程

建议用户在开局阶段完成以下输入：

- 选择一个 10 字以内的知识词
- 如有需要，输入一个用于加密的口令或密钥
- 确认进入游戏

### 6.2 Agent 侧流程

Agent 在开局阶段完成以下动作：

- 从允许的知识词范围中选择一个词
- 或根据规则确认一个词
- 将结果写入游戏初始化记录

### 6.3 服务端流程

服务端在开局阶段完成以下动作：

1. 接收开局请求
2. 校验知识词长度与合法性
3. 准备加密上下文
4. 加密用户答案或目标词
5. 生成 Agent 侧的开局信息
6. 写入 MongoDB
7. 返回开局结果

## 7. 数据结构建议

### 7.1 游戏主文档

建议游戏主文档至少包含这些字段：

- `game_id`
- `owner_id`
- `status`
- `round_count`
- `created_at`
- `updated_at`
- `history`
- `summary`
- `metadata`
- `user_knowledge_word`
- `agent_knowledge_word`
- `user_answer_encrypted`
- `agent_answer_encrypted`
- `encryption_metadata`

### 7.2 加密元数据

`encryption_metadata` 建议记录：

- 加密算法
- 密钥来源
- 创建时间
- 是否启用用户口令
- 其他必要的审计信息

### 7.3 为什么要把加密结果和元数据分开

因为加密后的内容和加密过程的说明是两类信息：

- 密文用于存储和校验
- 元数据用于排查和扩展

这样以后做验证或回放会更容易。

## 8. 加密策略建议

### 8.1 当前阶段的原则

本阶段建议采用：

- 服务端统一配置密钥
- 统一加密接口
- 明文不长期落库
- 加密过程可复用

### 8.2 为什么要先做服务端加密

因为这一阶段的重点是把“答案不可见”这个约束先固化下来。

如果没有统一加密层，后续在 Repository 和 Service 中到处散落加密逻辑，会非常难维护。

### 8.3 是否需要用户口令

当前可以保留两种可能：

- 简化版：服务端统一加密
- 增强版：用户口令参与密钥派生

是否启用用户口令，可以在下一步实现时再定。

## 9. 知识词选择建议

### 9.1 知识词范围

建议知识词来源是：

- 少量可控的固定词库
- 或一个可配置的知识词列表

#### 9.1.1 设计目标

这一小节的目标不是让用户自由输入任意长词，而是把“知识词”控制在一个可管理、可校验、可复用的范围内。

在当前项目里，知识词范围应满足下面几个条件：

- 长度不超过 10 个字符
- 语义尽量明确
- 便于 Agent 在提示词中稳定使用
- 便于后续数据库查询和回放

#### 9.1.2 推荐的数据来源

知识词范围建议来自以下两种方式之一：

1. 代码内固定列表，适合 demo 阶段快速落地
2. 配置文件或词库文件，适合后续扩展

如果当前阶段先做 demo，建议优先使用固定列表，因为它最简单，也最不容易出错。

#### 9.1.3 为什么要限制来源

限制来源的原因主要有：

- 避免超长输入
- 避免歧义词和脏数据
- 避免不同接口之间出现口径不一致
- 让 Agent 的 prompt 更容易稳定控制

#### 9.1.4 推荐实现思路

建议把知识词范围定义成一个独立常量，并提供一个统一校验函数。

下面是推荐的代码骨架：

```python
# backend/app/core/constants.py

MAX_KNOWLEDGE_WORD_LENGTH = 10

SUPPORTED_KNOWLEDGE_WORDS = [
    "反射",
    "蛋白质",
    "数据库",
    "微积分",
    "光合作用",
    "编译原理",
    "细胞膜",
    "量子力学",
]


def is_supported_knowledge_word(word: str) -> bool:
    if not word:
        return False
    normalized_word = word.strip()
    if len(normalized_word) == 0:
        return False
    if len(normalized_word) > MAX_KNOWLEDGE_WORD_LENGTH:
        return False
    return normalized_word in SUPPORTED_KNOWLEDGE_WORDS
```

#### 9.1.5 开局时如何校验

在开局接口或 GameService 中，应该先做两层校验：

1. 校验是否为空
2. 校验长度是否超过 10 字
3. 校验是否在允许词库中

建议的校验顺序如下：

```python
def validate_knowledge_word(word: str) -> str:
    if not word:
        raise ValueError("knowledge word is required")

    normalized_word = word.strip()
    if len(normalized_word) == 0:
        raise ValueError("knowledge word cannot be empty")

    if len(normalized_word) > 10:
        raise ValueError("knowledge word must be within 10 characters")

    if normalized_word not in SUPPORTED_KNOWLEDGE_WORDS:
        raise ValueError("knowledge word is not supported")

    return normalized_word
```

#### 9.1.6 为什么建议把校验单独抽出来

把校验单独抽出来有三个好处：

- Service 层更干净
- API 层更容易复用同一套规则
- 后续扩展新词库时只需要改一个地方

#### 9.1.7 与 Agent 的关系

Agent 在开局阶段如果也要参与知识词选择，建议只接收已经校验过的候选词，而不要直接接收任意输入。

这样可以保证：

- prompt 不会被脏输入污染
- 选择结果更稳定
- 后续日志更容易排查

#### 9.1.8 与数据库的关系

知识词校验通过后，建议把最终结果写入游戏主文档中的字段，例如：

- `user_knowledge_word`
- `agent_knowledge_word`

这样后续问答阶段就可以直接读取，不需要重新猜测开局信息。

### 9.2 为什么不建议一开始开放自由输入

自由输入会带来这些问题：

- 长度难以控制
- 意义歧义大
- Agent 提示词不稳定
- 难以做一致性校验

所以本阶段建议把知识词范围控制在可枚举、可验证的集合内。

### 9.3 Agent 如何选词

Agent 选词可以有两种方式：

- 规则选词
- LLM 辅助选词

当前阶段建议先保持简单，优先保证可控性。

## 12. 实现顺序建议

建议按下面顺序推进本阶段：

1. 先补加密模块
2. 再定义知识词约束
3. 再扩展游戏文档结构
4. 再调整开局接口
5. 再扩展 Service 和 Repository
6. 最后补验证与解密流程

这个顺序的好处是：

- 先把安全底座打好
- 再把知识词选择规则固定
- 最后再做接口和业务编排

### 12.1 第一步：补加密模块

这一阶段先不要碰业务流程，先把加密工具做成一个独立、可复用、可测试的基础模块。

#### 目标

- 统一加密和解密入口
- 统一密钥派生方式
- 让后续 Service 和 Repository 只调用工具，不关心实现细节

#### 建议文件

- `backend/app/core/security.py`
- `backend/requirements.txt`

#### 详细实现思路

加密工具建议至少提供下面几个函数：

```python
# backend/app/core/security.py

import base64
import hashlib
from cryptography.fernet import Fernet


def derive_key(password: str) -> bytes:
    digest = hashlib.sha256(password.encode("utf-8")).digest()
    return base64.urlsafe_b64encode(digest)


def encrypt_text(text: str, password: str) -> str:
    key = derive_key(password)
    fernet = Fernet(key)
    return fernet.encrypt(text.encode("utf-8")).decode("utf-8")


def decrypt_text(token: str, password: str) -> str:
    key = derive_key(password)
    fernet = Fernet(key)
    return fernet.decrypt(token.encode("utf-8")).decode("utf-8")
```

#### 设计考虑

- `derive_key()` 负责把口令转换成可用密钥
- `encrypt_text()` 负责把明文变成密文
- `decrypt_text()` 负责把密文还原成明文
- 如果以后要替换算法，只需要改这里

#### 依赖补充

`requirements.txt` 里需要确认加入加密库，例如：

```text
cryptography>=41.0.0
```

### 12.4 第四步：调整开局接口

开局接口要从“创建空游戏”升级为“初始化一局游戏”。

#### 目标

- 接收知识词和开局参数
- 调用 Service 完成校验和加密
- 返回开局结果

#### 建议文件

- `backend/app/api/v1/games.py`

#### 推荐请求模型

```python
from pydantic import BaseModel, Field


class StartGameRequest(BaseModel):
    owner_id: str | None = Field(default=None, description="游戏创建者的用户ID")
    user_knowledge_word: str = Field(..., description="用户选择的知识词，长度不超过10字")
    user_password: str | None = Field(default=None, description="可选的用户口令")
```

#### 推荐响应模型

```python
class StartGameResponse(BaseModel):
    game_id: str
    status: str
    round_count: int
    created_at: datetime
    user_knowledge_word: str
    agent_knowledge_word: str
```

#### 路由实现思路

```python
@router.post("", response_model=StartGameResponse, status_code=status.HTTP_201_CREATED)
async def start_game(request: StartGameRequest, service: GameService = Depends(get_game_service)):
    try:
        return await service.create_game(
            owner_id=request.owner_id,
            user_knowledge_word=request.user_knowledge_word,
            user_password=request.user_password,
        )
    except ValueError as exc:
        raise HTTPException(status_code=status.HTTP_400_BAD_REQUEST, detail=str(exc))
    except Exception:
        raise HTTPException(status_code=status.HTTP_500_INTERNAL_SERVER_ERROR, detail="Failed to start game")
```

#### 设计考虑

- 路由层只处理请求和响应
- 开局接口直接返回新的游戏上下文
- `games.py` 不再只是创建空记录，而是开局初始化入口

### 12.5 第五步：扩展 Service 和 Repository

Service 和 Repository 是把上面所有能力串起来的核心。

#### 目标

- 在 Service 中完成业务校验和开局编排
- 在 Repository 中完成 MongoDB 写入和查询

#### 建议文件

- `backend/app/services/game_service.py`
- `backend/app/repositories/game_repository.py`

#### Service 推荐实现

```python
from datetime import datetime
from app.core.constants import validate_knowledge_word
from app.core.security import encrypt_text


class GameService:
    def __init__(self, settings, agent, repository, memory_policy):
        self.settings = settings
        self.agent = agent
        self.repository = repository
        self.memory_policy = memory_policy

    async def create_game(self, owner_id=None, user_knowledge_word: str | None = None, user_password: str | None = None):
        game_id = str(uuid.uuid4())
        now = datetime.utcnow()

        normalized_user_word = validate_knowledge_word(user_knowledge_word)

        agent_knowledge_word = normalized_user_word
        encryption_password = user_password or self.settings.encryption_secret

        user_answer_encrypted = encrypt_text(normalized_user_word, encryption_password)
        agent_answer_encrypted = encrypt_text(agent_knowledge_word, self.settings.encryption_secret)

        game_doc = {
            "game_id": game_id,
            "owner_id": owner_id,
            "status": "active",
            "history": [],
            "summary": "",
            "round_count": 0,
            "created_at": now,
            "updated_at": now,
            "metadata": {},
            "user_knowledge_word": normalized_user_word,
            "agent_knowledge_word": agent_knowledge_word,
            "user_answer_encrypted": user_answer_encrypted,
            "agent_answer_encrypted": agent_answer_encrypted,
            "encryption_metadata": {
                "algorithm": "fernet",
                "key_source": "settings.encryption_secret",
                "created_at": now,
                "user_password_enabled": user_password is not None,
            },
        }

        await self.repository.create_game(game_doc)

        return {
            "game_id": game_id,
            "status": "active",
            "round_count": 0,
            "created_at": now,
            "user_knowledge_word": normalized_user_word,
            "agent_knowledge_word": agent_knowledge_word,
        }
```

#### Repository 推荐实现

```python
from datetime import datetime


class GameRepository:
    def __init__(self, db):
        self.db = db
        self.collection = db["games"]

    async def create_game(self, game_doc: dict):
        existing_game = await self.collection.find_one({"game_id": game_doc["game_id"]})
        if existing_game:
            raise ValueError(f"Game with id {game_doc['game_id']} already exists")
        await self.collection.insert_one(game_doc)
        return game_doc

    async def get_game(self, game_id: str):
        return await self.collection.find_one({"game_id": game_id})

    async def update_game_history(self, game_id: str, history: list[dict], summary: str, round_count: int):
        result = await self.collection.update_one(
            {"game_id": game_id},
            {
                "$set": {
                    "history": history,
                    "summary": summary,
                    "round_count": round_count,
                    "updated_at": datetime.utcnow(),
                }
            },
        )
        if result.matched_count == 0:
            raise ValueError(f"Game with id {game_id} not found")
        return result
```

4. 调用问答接口，确认能读取到该游戏状态

### 13.3 如果验收不通过，优先检查什么

优先检查下面几个点：

- 开局请求里是否传入了合法知识词
- `validate_knowledge_word()` 是否通过
- `encrypt_text()` 是否工作正常
- MongoDB 写入是否成功
- `games.py` 是否调用了正确的 Service 方法
