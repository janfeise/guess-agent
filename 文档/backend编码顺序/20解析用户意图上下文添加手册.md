# 20 解析用户意图上下文添加手册

## 1. 文档目标

这份手册解决的问题是：

**当用户只回复“无关”“是”“不是”这类短句时，`GuessAgent.parse_user_intent()` 必须拿到上一轮对话上下文，否则 LLM 很容易把它判成 `invalid` 或闲聊。**

本次修复只要求加入**最近一轮对话**，不需要把整局历史都塞给意图解析模型。

---

## 2. 问题现象

典型日志如下：

- Agent 先问：“这个概念与天体物理学有关吗？”
- 用户只回：“无关”
- LLM 请求里只有：
  - System Prompt：识别规则
  - User Message：请分析用户输入的意图：无关

此时 LLM 看不到上一轮 Agent 的问题，就不知道“无关”是在回答“是否有关”，还是在表达一种泛化的否定语气。结果很容易被判成：

- `invalid`
- 闲聊
- 无法判断

这就是上下文缺失导致的误判。

---

## 3. 根因分析

当前意图解析链路的问题，不是“模型不聪明”，而是“输入信息不完整”。

### 3.1 只传了当前输入

`parse_user_intent(user_input, history)` 目前实际发给 LLM 的，只有当前这句用户输入。上一轮问了什么，没有进入上下文。

### 3.2 上下文粒度过粗或格式不对

不要直接把整段 `history` 原样拼成字符串丢给模型。这样会带来两个问题：

- prompt 变长，分类稳定性下降。
- 模型会被更早的历史干扰，反而抓不住最近一轮的语义。

### 3.3 非标准 role 会丢上下文

如果消息列表里用了 `communication` 之类的非标准角色，LangChain / ChatOpenAI 侧可能不会按预期传递。

意图解析上下文必须用标准消息角色表达，优先使用：

- `system`
- `user`
- `assistant`

---

## 4. 修复原则

这次修复只遵守三个原则：

1. 只补最近一轮，不补整局历史。
2. 上一轮上下文要以标准消息角色进入模型。
3. 当前用户输入仍然单独保留，不能把上下文和当前输入混成一段不可分辨的文本。

---

## 5. 推荐实现方式

### 5.1 抽一个“最近一轮上下文”构造函数

建议在 `GuessAgent` 内部新增一个专用方法，而不是复用 `build_summary()` 或整局历史拼接逻辑。

示例：

```python
def _build_last_turn_context(self, history: list[dict]) -> str:
    if not history:
        return ""

    last_turn = history[-1]
    question = (
        last_turn.get("question")
        or last_turn.get("pending_question")
        or last_turn.get("input_text")
        or ""
    )
    answer = last_turn.get("answer") or last_turn.get("answer_text") or ""

    if not question and not answer:
        return ""

    return f"最近一轮对话：\nAgent：{question}\nUser：{answer}"
```

如果你的 `history` 结构已经是“每轮一个 dict”，那就取最后一轮；不要取最后两轮或整段历史。

### 5.2 在 `parse_user_intent()` 里显式注入最近一轮

推荐把消息组织成下面这样：

```python
messages = [
    {"role": "system", "content": self.prompt_loader.load_prompt("turn_router")},
]

last_turn_context = self._build_last_turn_context(history)
if last_turn_context:
    messages.append({"role": "user", "content": last_turn_context})

messages.append({"role": "user", "content": f"请分析用户输入的意图：{user_input}"})
```

这里的关键点是：

- 最近一轮对话先给模型
- 当前用户输入后给模型
- 不要把上下文塞进一个非标准字段

### 5.3 如果你想更稳一点，可以把最近一轮拆成两条消息

如果 LLM 对短句分类仍然不稳，可以把上一轮拆成“问题”和“回答”两条标准消息：

```python
messages = [
    {"role": "system", "content": self.prompt_loader.load_prompt("turn_router")},
    {"role": "assistant", "content": previous_question},
    {"role": "user", "content": previous_answer},
    {"role": "user", "content": f"请分析用户输入的意图：{user_input}"},
]
```

这样模型更容易把“无关”理解成对前一个问题的否定回答。

## 6. 应该改哪里

### 6.1 Agent 层

优先改：[backend/app/agents/guess_agent_fixed.py](../../backend/app/agents/guess_agent_fixed.py)

如果你当前分支里还在使用旧的 [backend/app/agents/guess_agent.py](../../backend/app/agents/guess_agent.py)，也要同步补齐同样逻辑，避免入口文件不一致。

重点修改：

- `parse_user_intent()`
- 最近一轮上下文构造函数
- LLM 请求日志

### 6.2 Service 层

优先检查：[backend/app/services/game_service.py](../../backend/app/services/game_service.py)

在这些分支里，调用 `parse_user_intent()` 时必须传入完整的历史记录，不能提前裁掉：

- 普通问题分支
- 用户回答分支
- 意图预判分支

因为“最近一轮上下文”应该由 Agent 层自己裁剪，而不是 Service 层自己猜。

---

## 7. 不要这样做

### 7.1 不要直接传整局历史

整局历史会让意图解析被旧上下文污染，尤其是对“是”“否”“无关”这种短句。

### 7.2 不要只传当前短句

只传“无关”本身，模型几乎没有办法稳定判断它是不是“回答是否有关”的否定答案。

### 7.3 不要使用非标准 role

`communication` 这类字段不应该进入正式的 LLM 消息列表。意图解析要用标准消息角色。

### 7.4 不要用 summary 代替最近一轮

摘要适合长历史压缩，不适合意图分类。意图解析需要的是“刚刚那一轮说了什么”，不是“这一局大概在聊什么”。

---

## 8. 建议的日志格式

实现完以后，日志里最好能看出下面三件事：

- 当前用户输入是什么
- 最近一轮对话是什么
- 最终发给模型的消息顺序是什么

建议记录：

- `game_id`
- `phase`
- `turn_type`
- `history_len`
- `recent_turn_present`
- `intent`

示例：

```python
logger.info(
    "意图解析上下文: game_id=%s history_len=%s recent_turn_present=%s",
    game_id,
    len(history),
    bool(last_turn_context),
)
```

注意不要把敏感明文词、密钥或完整隐私内容直接打进日志。

---

## 9. 验收标准

修完以后，至少要验证下面三种情况。

### 9.1 用户回答短句

场景：

- Agent 问：这个概念与天体物理学有关吗？
- 用户答：无关

期望：

- `parse_user_intent()` 能结合最近一轮上下文把它识别成 `answer=no`
- 不再落到 `invalid`

### 9.2 用户回答是/否

场景：

- Agent 问：你是在确认它属于动物吗？
- 用户答：是

期望：

- 仍然能正确识别为 `answer=yes`

### 9.3 没有上一轮上下文

场景：

- 这局刚开始
- 用户直接输入一个短句

期望：

- 系统不要编造上下文
- 没有历史时就只按当前输入解析

---

## 10. 推荐落地顺序

1. 先在 Agent 层加一个最近一轮上下文构造函数。
2. 再把 `parse_user_intent()` 的消息列表改成“系统提示 + 最近一轮 + 当前输入”。
3. 然后补日志，确认实际发给模型的消息顺序正确。
4. 最后用“无关 / 是 / 不是”这类短回复做回归测试。

---

## 11. 结论

这次修复的核心不是“让模型猜得更聪明”，而是“给模型足够的最近上下文”。

对意图解析来说，**最近一轮对话** 就够了。只要把上一轮的问题和用户回答按标准消息角色传进去，像“无关”这种短回复就能稳定回到 `no`，而不是误判成闲聊或无效输入。
