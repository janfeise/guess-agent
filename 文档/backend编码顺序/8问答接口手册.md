# 8 问答接口手册

## 1. 文档目标

这一章的目标是指导你完成“问答接口”，也就是用户在创建游戏之后，向系统提交一个问题，然后由 Agent 生成答案并把这一轮写回数据库的接口。

写完这一章后，你应该能够明确下面这些事情：

- 问答接口应该放在哪一层
- 接口应该如何接收 `game_id`、`question` 和 `mode`
- 接口如何调用 `GameService.submit_turn()`
- 普通问答和后续流式问答的边界在哪里
- 成功和失败时应该如何返回

这一章是“创建游戏接口”的下一步，也是整个猜词流程最核心的一步。

---

## 2. 问答接口在架构中的位置

推荐调用链路如下：

```text
前端 / 客户端 -> API 路由 -> GameService.submit_turn() -> GameRepository.update_game_history() -> MongoDB
```

### 2.1 API 层负责什么

API 层负责：

- 接收 `game_id` 和问题内容
- 做基础参数校验
- 调用 Service 层
- 把 Service 返回结果转换成 HTTP 响应

### 2.2 Service 层负责什么

Service 层负责：

- 检查游戏是否存在
- 检查游戏是否已结束
- 读取历史回合
- 调用 Agent 生成答案
- 更新历史、摘要和轮次
- 返回本轮问答结果

### 2.3 Repository 层负责什么

Repository 层负责：

- 把更新后的历史写入 MongoDB
- 同步更新 summary、round_count 和 updated_at

---

## 3. 为什么要单独写问答接口

问答接口是核心业务接口，它和创建游戏接口不一样。

创建游戏只负责“开局”，而问答接口负责：

- 读取当前局上下文
- 组织模型输入
- 生成一轮答案
- 持久化这一轮结果

如果不单独写这个接口，后面所有“提问”的流程都会散落在别的地方，维护起来会很乱。

---

## 4. 接口职责边界

### 4.1 问答接口应该做的事情

问答接口应该负责：

- 接收问题输入
- 接收 `game_id`
- 接收可选的 `mode`
- 调用 `GameService.submit_turn()`
- 返回答案和更新后的历史

### 4.2 问答接口不应该做的事情

问答接口不应该负责：

- 不直接调用 Agent
- 不自己拼接历史
- 不自己更新 MongoDB
- 不做游戏状态判断
- 不做摘要压缩

这些事情都应该交给 Service 和 Repository。

---

## 5. 推荐路由设计

### 5.1 路由前缀

建议继续沿用游戏资源前缀：

- `/api/v1/games`

### 5.2 问答路由

推荐使用以下路由：

- `POST /api/v1/games/{game_id}/turns`

这个命名的好处是：

- `game_id` 清晰地表示“对哪一局游戏提问”
- `turns` 表示这是一个回合操作
- 路由语义更贴近游戏场景

### 5.3 为什么不用普通的 `/answer`

因为在这个项目里，问答不是一次独立的聊天，而是“某一局游戏中的一轮回合”。

用 `turns` 更能表达业务语义，也方便后续统计轮次。

---

## 6. 请求结构设计

### 6.1 必要请求字段

问答接口至少需要这些字段：

- `game_id`：路由参数
- `question`：用户问题

### 6.2 可选请求字段

当前项目里还可以加一个可选字段：

- `mode`：调用模式，默认 `agent`

如果后续你要区分主模型和小助手，这个字段会很有用。

### 6.3 推荐请求格式

```json
{
  "question": "现在我应该猜什么？",
  "mode": "agent"
}
```

### 6.4 为什么 question 不能省略

问答接口的核心就是 question。

如果没有问题，就没有这轮回合的意义。

---

## 7. 返回结构设计

### 7.1 成功返回什么

建议返回这些字段：

- `game_id`
- `question`
- `answer`
- `history`

如果你想让前端更容易展示，还可以加：

- `round_count`
- `updated_at`

### 7.2 推荐返回示例

```json
{
  "game_id": "b8f2e6f0-3f60-4a7c-9b7a-9a0d7f2d0f1a",
  "question": "现在我应该猜什么？",
  "answer": "你可以先从常见类别开始排除，比如动物、物品或地点。",
  "history": [
    {
      "question": "现在我应该猜什么？",
      "answer": "你可以先从常见类别开始排除，比如动物、物品或地点。"
    }
  ]
}
```

### 7.3 为什么要返回 history

返回 history 的目的不是让前端自己算逻辑，而是让前端能：

- 直接刷新对话列表
- 保持当前局上下文显示一致
- 减少前端自己拼接状态的工作

---

## 8. 需要的请求模型

如果你使用 FastAPI，建议为问答接口单独定义请求模型。

### 8.1 请求模型职责

请求模型负责：

- 定义 `question`
- 定义可选 `mode`
- 自动生成更清晰的接口文档

### 8.2 建议字段

- `question`：必填字符串
- `mode`：可选字符串，默认 `agent`

### 8.3 推荐模型结构

```python
from pydantic import BaseModel, Field


class SubmitTurnRequest(BaseModel):
    question: str = Field(..., description="用户提交的问题")
    mode: str = Field(default="agent", description="调用模式，默认 agent")
```

### 8.4 为什么要单独定义模型

因为这样可以让输入结构更稳定，也便于后续扩展更多字段，比如：

- `client_message_id`
- `hint_level`
- `debug`

---

## 9. 问答接口的实现流程

推荐把接口实现分成下面几个步骤。

### 9.1 接收参数

先接收 `game_id` 和请求体中的 `question`、`mode`。

### 9.2 调用 Service

API 层不自己处理历史，不自己调用 Agent，只需要调用：

- `GameService.submit_turn(game_id=..., question=..., mode=...)`

### 9.3 返回结果

Service 返回结果后，API 层原样返回或做少量包装即可。

---

## 10. 详细实现结构

这一部分给出问答接口的推荐骨架。

```python
from fastapi import APIRouter, Depends, HTTPException, status

router = APIRouter(prefix="/games", tags=["games"])


@router.post("/{game_id}/turns")
async def submit_turn(game_id: str, request: SubmitTurnRequest, service: GameService = Depends(get_game_service)):
    ...
```

### 10.1 依赖注入的作用

和创建游戏接口一样，问答接口也应该通过依赖注入拿到 `GameService`，而不是在路由里自己实例化所有对象。

### 10.2 为什么路由函数只保留最少参数

因为真正的业务逻辑都在 Service 层，路由函数越简单，越容易维护。

---

## 11. 问答接口与 GameService 的联动方式

### 11.1 Service 层调用链

`GameService.submit_turn()` 内部通常会做这些事情：

1. 查询游戏记录
2. 判断是否存在
3. 判断是否已结束
4. 读取历史回合
5. 调用 Agent 生成回答
6. 更新历史、摘要、轮次
7. 返回结果

### 11.2 API 层只负责触发

API 层不需要知道历史怎么拼，不需要知道摘要怎么更新，只需要把请求参数传进去。

---

## 12. 成功和失败响应建议

### 12.1 成功响应

建议返回：

- `200 OK`

### 12.2 失败响应

常见失败情况包括：

- 游戏不存在
- 游戏已经结束
- 问题为空
- Agent 调用失败
- 数据库更新失败

### 12.3 推荐状态码

- `400 Bad Request`：请求参数不合法，或者业务上无法继续
- `404 Not Found`：游戏不存在
- `500 Internal Server Error`：内部异常

如果你当前想先简化，也可以先统一用 `400` 和 `500`，后续再细化。

---

## 13. 异常处理建议

### 13.1 问答接口常见异常

常见异常包括：

- `game_id` 不存在
- 游戏状态为 `finished`
- `question` 为空字符串
- Agent 返回错误
- Repository 更新失败

### 13.2 推荐处理方式

建议 API 层做一层简单的异常翻译：

- `ValueError` 转成业务错误响应
- 其他异常转成内部错误响应

### 13.3 为什么不能把异常直接抛给前端

因为那样会暴露内部实现细节，前端也很难统一处理。

---

## 14. 代码级实现参考

下面这部分是最关键的代码实现参考。

### 14.1 路由文件建议结构

建议把创建游戏和问答接口放在同一个路由文件里，比如：

```text
backend/app/api/v1/games.py
```

如果你后面想拆分，也可以继续拆成多个文件，但当前 demo 阶段放在一起更简单。

### 14.2 路由骨架

```python
from fastapi import APIRouter, Depends, HTTPException, status
from pydantic import BaseModel, Field


class SubmitTurnRequest(BaseModel):
    question: str = Field(..., description="用户提交的问题")
    mode: str = Field(default="agent", description="调用模式，默认 agent")


router = APIRouter(prefix="/games", tags=["games"])


@router.post("/{game_id}/turns")
async def submit_turn(game_id: str, request: SubmitTurnRequest, service: GameService = Depends(get_game_service)):
    try:
        return await service.submit_turn(game_id=game_id, question=request.question, mode=request.mode)
    except ValueError as exc:
        raise HTTPException(status_code=status.HTTP_400_BAD_REQUEST, detail=str(exc))
    except Exception:
        raise HTTPException(status_code=status.HTTP_500_INTERNAL_SERVER_ERROR, detail="Failed to submit turn")
```

### 14.3 这段代码的职责拆分

- `SubmitTurnRequest`：定义用户输入
- `submit_turn()`：处理 HTTP 请求
- `GameService.submit_turn()`：处理业务逻辑
- `GameRepository.update_game_history()`：处理数据库写入

### 14.4 为什么这里要传 `game_id` 到路径里

因为这是一局游戏中的一轮提问，`game_id` 本身就是资源定位的一部分。

把 `game_id` 放在路径上比放在请求体里更清晰，也更符合 REST 风格。

---

## 15. 与创建游戏接口的关系

问答接口和创建游戏接口是连续的两步。

### 15.1 创建游戏

先调用创建游戏接口，拿到 `game_id`。

### 15.2 提交问答

再拿着这个 `game_id` 调问答接口。

### 15.3 为什么必须先创建

因为问答接口需要一个明确的游戏上下文，而这个上下文就是 `game_id` 所标识的一局游戏。

---

## 16. 调试顺序建议

建议按下面顺序验证问答接口：

1. 先创建一局游戏
2. 再提交第一轮问题
3. 再确认返回的 answer 是否合理
4. 再检查数据库里的 history 是否更新
5. 再确认 round_count 和 summary 是否同步变化

### 16.1 先验证业务链路

只要第一轮能正常返回，就说明创建游戏和问答接口之间的链路已经跑通。

### 16.2 再验证落库

确认 MongoDB 里的 `history` 数组已经多了一条记录。

### 16.3 再验证状态约束

确认已经结束的游戏不能继续提交问答。

---

## 17. 本章补充结论

问答接口是整个猜词项目最核心的业务接口之一。

它的关键职责可以总结成一句话：

**接收一轮问题，交给 Service 生成答案，再把这一轮写回数据库。**

如果这一层写得清楚，后面的流式问答接口、结束游戏接口、回合查询接口都会很好扩展。
