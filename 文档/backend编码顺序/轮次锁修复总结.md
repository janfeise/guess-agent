# 轮次锁修复总结 (Executive Summary)

## 问题

用户反馈：用户提问后，系统应该根据收集的信息进行**决策**（继续提问或猜测），但当前实现直接返回 `user_turn`，导致用户可以无限提问而系统永远没有机会主动行动。

**错误场景**：

```
用户: 是理科吗?
系统: 是的。【现在轮到你继续提问】❌ 不对！应该轮到系统决策
用户: 是物理吗?
系统: 是的。【现在轮到你继续提问】❌ 继续错误
...
```

## 修复方案（核心改动）

### ✅ 修复要点

在两个关键处理函数中**恢复 `decide_agent_action()` 调用**：

| 函数                      | 修复内容                                                                                                                |
| ------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| `_handle_user_question()` | 在回答用户问题后，调用 `decide_agent_action()` 获取系统决策，根据返回的 action（question 或 guess）条件路由到不同的阶段 |
| `_handle_user_answer()`   | 在处理用户回答后，调用 `decide_agent_action()` 获取系统决策，根据返回的 action 条件路由到不同的阶段                     |

### ✅ 新增逻辑

```python
# 在 _handle_user_question() 和 _handle_user_answer() 中都添加：

# 系统决策
agent_decision = await self.agent.decide_agent_action(raw_history, summary)
agent_action = agent_decision.get("action", "question")  # "question" 或 "guess"

# 根据决策条件路由
if agent_action == "question":
    # 添加系统的新问题到 history
    # 设置 phase = "waiting_answer"，期望用户回答
else:  # agent_action == "guess"
    # 添加系统的猜测到 history
    # 设置 phase = "awaiting_judgement"，期望用户判断
```

## 正确的游戏循环

### 修复后的流程

```
【第1轮】
用户提问 → 系统回答 → 系统决策 →
  ├─ 继续提问 → phase="waiting_answer" → 用户回答
  └─ 猜测 → phase="awaiting_judgement" → 用户判断

【第2轮】
用户回答/判断 → 系统决策 →
  ├─ 如果是回答：继续提问/猜测 → phase="waiting_answer" 或 "awaiting_judgement"
  └─ 如果是判断：结束或回到 user_turn
```

## 问题排查 (Troubleshooting)

### Q: 修复后游戏还是一直 waiting_answer？

**A**: 检查 `decide_agent_action()` 是否返回正确的 action。查看日志中的：

```
INFO: LLM 原始返回 [decide_agent_action/parsed]: {...}
```

如果 `action` 总是 "question"，说明 LLM 认为信息还不足。可能需要调整 prompt 或 `max_rounds`。

### Q: 用户提问后的 phase 还是 user_turn？

**A**: 检查：

1. `decide_agent_action()` 是否被调用（查看日志）
2. 返回值中 action 字段是否正确
3. 条件分支是否进入了正确的代码路径

### Q: 多了很多 LLM 调用导致延迟增加？

**A**: 这是正常的。修复加入了 `decide_agent_action()` 调用，对应增加延迟。可以：

1. 使用更快的模型作为决策模型
2. 优化 prompt 以加快 LLM 响应
3. 接受约 2-3秒的额外延迟是游戏交互中可以接受的
