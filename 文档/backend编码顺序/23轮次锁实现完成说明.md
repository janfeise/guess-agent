# 轮次锁完成说明

## 一、概述

轮次锁（Turn Lock）已在代码层、数据层和API层完成实现。核心目标是强制执行游戏的交替回合制，防止系统在用户应该行动时继续自行操作。

## 二、核心设计

### 2.1 状态机模型

游戏有三个主要阶段（记录在 `metadata.phase` 中）：

```
user_turn (初始状态)
  ↓
awaiting_answer (系统提问后，等待用户回答)
  ↓
user_turn (用户回答后，回到用户轮)
  ↓
awaiting_judgement (系统猜测后，等待用户判断)
  ↓
user_turn (用户判断后，回到用户轮)
```

### 2.2 轮次锁字段

每个游戏的 `metadata` 中存储以下字段来表示当前轮次锁状态：

| 字段                 | 类型        | 含义             | 初始值        |
| -------------------- | ----------- | ---------------- | ------------- |
| `phase`              | str         | 当前阶段         | `"user_turn"` |
| `next_actor`         | str         | 下一个行动者     | `"user"`      |
| `expected_turn_type` | str         | 期望的意图类型   | `"question"`  |
| `last_actor`         | str         | 上一个行动者     | `"system"`    |
| `last_intent`        | str \| None | 上一个意图       | `None`        |
| `awaiting_answer`    | bool        | 是否等待用户回答 | `False`       |
| `awaiting_judgement` | bool        | 是否等待用户判断 | `False`       |

### 2.3 意图类型

系统支持四种用户意图：

- `question`: 用户提问 是/否 问题
- `guess`: 用户猜测答案
- `answer`: 用户回答系统的问题
- `judge`: 用户判断系统的猜测是否正确
- `invalid`: 无效输入

#### `_get_turn_lock_state(game)` - 读取轮次锁状态

#### `_is_turn_allowed(phase, intent)` - 校验意图是否允许

根据当前阶段和用户意图进行校验：

```python
def _is_turn_allowed(phase, intent):
    # user_turn: 允许 question 和 guess
    # awaiting_answer: 只允许 answer
    # awaiting_judgement: 允许 judge 和 answer
```

#### `_reject_turn(game_id, phase, message, history)` - 拒绝非法回合

当用户意图违反轮次锁时，调用此函数返回拒绝消息。

### 3.2 核心流程：submit_turn()

`submit_turn()` 是游戏回合的主分发函数，现在实现了 **阶段优先** 的校验逻辑：

```
1. 读取游戏
2. 提取 phase 从 metadata
3. 使用 parse_user_intent 分类用户意图
4. 【轮次锁校验】根据 phase 判断 intent 是否允许
5. 如果不允许，调用 _reject_turn() 返回拒绝消息
6. 如果允许，路由到对应的 handler 函数
```

这是 **轮次锁的核心实现点**：校验发生在路由之前，不依赖 LLM 的解释。

### 3.4 数据层扩展

`GameRepository.update_game_state()` 扩展了两个新参数：

```python
async def update_game_state(
    self,
    game_id: str,
    # ... 其他参数
    next_actor: str | None = None,
    expected_turn_type: str | None = None,
):
    # 条件性更新：只有当参数不为 None 时才更新
    if next_actor is not None:
        update_dict["$set"]["metadata.next_actor"] = next_actor
    if expected_turn_type is not None:
        update_dict["$set"]["metadata.expected_turn_type"] = expected_turn_type
```

这确保轮次锁状态与数据库持久化。

## 四、工作流示例

### 场景：用户连续回答

**问题**：用户刚回答完系统的问题，又输入 "是"（作为另一个问题的回答）

**轮次锁的保护**：

```
1. 游戏状态：phase = "user_turn"
2. 用户输入 "是"
3. parse_user_intent 识别为 "answer"
4. 【轮次锁校验】user_turn 不允许 answer
5. 拒绝：_reject_turn() → "现在轮到你提问或猜词，不是回答阶段"
```

### 场景：系统猜测错误后

**预期行为**：用户判断系统猜测后，应该回到用户回合

**轮次锁流程**：

```
1. 系统提问 → phase = "awaiting_answer"
2. 用户回答 → update_game_state(..., phase="user_turn")
3. 系统不自动猜测，等待用户提问或猜测
4. 用户猜测 → 系统判断 → phase = "awaiting_judgement"
5. 用户判断 → update_game_state(..., phase="user_turn")
```

- 轮次锁校验是纯逻辑操作，在 `submit_turn()` 中执行，性能开销极小 (~1ms)
