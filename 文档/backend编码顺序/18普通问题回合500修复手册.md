# 18 普通问题回合 500 修复手册

## 1. 文档目标

这份手册只解决一个问题：

**用户提交普通问题时，`POST /api/v1/games/{game_id}/turns` 返回 500，应该怎么修。**

你现在的代码已经把猜词核心流程往前推进了一步，但普通提问这条主链路还没有完全收口，所以在真实请求时会报 500。

这份手册的作用不是重新设计系统，而是帮你把当前缺口补齐，保证普通问题能稳定走通。

---

## 2. 这个 500 的根因

先说结论：

### 2.1 当前最直接的根因

在 [backend/app/services/game_service.py](../../backend/app/services/game_service.py) 中，`submit_turn()` 在处理普通问题时，会走到：

```python
return await self._handle_user_question(game, question, history, mode)
```

但当前类里**没有实现 `_handle_user_question()`**。

这意味着：

- 普通问题路径一旦进入这个分支，就会直接抛异常
- 异常没有在 service 层被转换成业务响应
- 路由层最后只能返回 500

### 2.2 相关但次要的风险点

除了缺失 `_handle_user_question()`，当前还有这些容易在提问链路里出问题的点：

- `GuessAgent.answer_question()` 返回结构不稳定时，service 层没有兜底
- `GameRepository.update_game_state()` 需要确保能更新 `phase`、`history`、`summary`、`round_count`
- 接口层要兼容 `question` / `content` 两种字段，否则旧客户端可能传空
- `SubmitTurnResponse.phase` 的类型定义应该是字符串，不是整数

---

## 3. 普通问题路径应该长什么样

正确的普通问题链路应该是：

```text
客户端提交问题
  -> submit_turn()
  -> 读取游戏状态
  -> 判断当前不是 awaiting_judgement
  -> parse_user_intent()
  -> intent = question
  -> _handle_user_question()
  -> agent.answer_question()
  -> 写回 history / summary / round_count / phase
  -> agent.decide_agent_action()
  -> 如果 agent 提问，返回下一问
  -> 如果 agent 猜测，进入 awaiting_judgement
```

这一条链路里，普通问题的关键不是“返回一个答案”这么简单，而是：

- 回答用户问题
- 顺便推进下一轮状态
- 保持历史和阶段一致

---

## 4. 你需要修哪些文件

### 4.1 [backend/app/services/game_service.py](../../backend/app/services/game_service.py)

这是主修文件。普通问题 500 的根修复点就在这里。

你需要补：

- `_handle_user_question()`
- 让 `answer_question()` 的结果有稳定兜底
- 让普通问题后的状态更新收口到仓储层

### 4.2 [backend/app/agents/guess_agent.py](../../backend/app/agents/guess_agent.py)

这个文件主要负责：

- 把用户问题转成模型输入
- 把模型输出转成可用结果
- 在模型输出不标准时返回可执行的兜底结构

### 4.3 [backend/app/repositories/game_repository.py](../../backend/app/repositories/game_repository.py)

这个文件负责把普通问题后的历史和状态真正写回数据库。

你要确保：

- `update_game_state()` 可用
- `phase` 能被更新
- `history`、`summary`、`round_count` 能同步写回

### 4.4 [backend/app/api/v1/submit_turn.py](../../backend/app/api/v1/submit_turn.py)

这个文件负责把请求参数正确送入 service。

你要确认：

- `question` 和 `content` 能互相兼容
- 回合接口不会因为空字段直接走异常

---

## 5. 服务层应该怎么补

这一节最关键，建议你直接照着补。

### 5.1 新增 `_handle_user_question()`

这个函数是普通问题路径的核心。

#### 目标

- 把用户问题交给 Agent 回答
- 把结果写回历史
- 推进下一轮状态
- 返回结构化响应

#### 建议实现骨架

```python
async def _handle_user_question(self, game: dict, question: str, history: list[dict], mode: str = "agent"):
    target_word = self.reveal_target_word(game, self.settings)
    answer_result = await self.agent.answer_question(question, target_word)

    if isinstance(answer_result, dict):
        answer_text = answer_result.get("answer", "unknown")
        answer_reason = answer_result.get("reason", "")
    else:
        answer_text = str(answer_result)
        answer_reason = ""

    updated_history = history + [{
        "round_no": game.get("round_count", 0) + 1,
        "actor": "user",
        "turn_type": "question",
        "input_text": question,
        "answer_text": answer_text,
        "result": answer_reason,
    }]

    summary = self.memory_policy.build_summary(updated_history)
    agent_action = await self.agent.decide_agent_action(updated_history, summary)

    if agent_action.get("action") == "guess":
        updated_history.append({
            "round_no": len(updated_history) + 1,
            "actor": "agent",
            "turn_type": "agent_guess",
            "input_text": agent_action.get("guess_word", ""),
            "answer_text": "awaiting_judgement",
            "result": f"confidence={agent_action.get('confidence', 0.0)}",
        })
        await self.repository.update_game_state(
            game["game_id"],
            history=updated_history,
            summary=self.memory_policy.build_summary(updated_history),
            round_count=len(updated_history),
            phase="awaiting_judgement",
            awaiting_judgement=True,
            pending_guess=agent_action.get("guess_word", ""),
            agent_confidence=agent_action.get("confidence", 0.0),
        )
        return {
            "game_id": game["game_id"],
            "status": game.get("status", "active"),
            "phase": "awaiting_judgement",
            "answer": answer_text,
            "agent_guess": agent_action.get("guess_word", ""),
            "history": updated_history,
            "message": "系统已给出猜测，请判断是否正确",
        }

    updated_history.append({
        "round_no": len(updated_history) + 1,
        "actor": "agent",
        "turn_type": "agent_question",
        "input_text": agent_action.get("question", ""),
        "answer_text": "",
        "result": f"confidence={agent_action.get('confidence', 0.0)}",
    })

    await self.repository.update_game_state(
        game["game_id"],
        history=updated_history,
        summary=self.memory_policy.build_summary(updated_history),
        round_count=len(updated_history),
        phase="user_turn",
        awaiting_judgement=False,
        pending_guess="",
        agent_confidence=0.0,
    )

    return {
        "game_id": game["game_id"],
        "status": game.get("status", "active"),
        "phase": "user_turn",
        "answer": answer_text,
        "next_question": agent_action.get("question", ""),
        "history": updated_history,
        "message": "已回答你的问题，系统继续推进下一轮",
    }
```

### 5.2 统一普通问题返回结构

建议普通问题返回至少包含：

- `game_id`
- `status`
- `phase`
- `answer`
- `next_question`
- `history`
- `message`

这样前端不会只拿到一段文本，而是可以直接判断下一步显示什么。

### 5.3 让 helper 方法变成静态方法

你当前 `game_service.py` 里有几个辅助方法写法不规范，会导致调用混乱。

建议改成：

```python
@staticmethod
def normalize_guess_text(text: str) -> str:
    return text.strip().lower()

@staticmethod
def is_guess_correct(user_guess: str, target_word: str) -> bool:
    return GameService.normalize_guess_text(user_guess) == GameService.normalize_guess_text(target_word)

@staticmethod
def reveal_target_word(game_doc: dict, settings) -> str:
    system_password = decrypt(game_doc["system_password_encrypted"], settings.encryption_secret)
    return decrypt(game_doc["system_word_encrypted"], system_password)
```

这样普通问题和猜测问题都能共用同一套辅助函数。

## 6. Agent 层要怎么配合

普通问题路径不会只停在“回答用户”这一步，还要让 Agent 决定下一轮做什么。

### 6.1 `answer_question()` 的返回要稳定

推荐让 `answer_question()` 返回字典结构：

```python
{
    "answer": "yes | no | unknown",
    "reason": "简短理由"
}
```

### 6.2 `decide_agent_action()` 也要稳定

推荐返回：

```python
{
    "action": "ask | guess",
    "question": "...",
    "guess_word": "...",
    "confidence": 0.0,
    "reason": "..."
}
```

### 6.3 JSON 解析要有兜底

如果模型没有返回 JSON，不要直接炸掉。

建议兜底逻辑是：

- 如果能解析 JSON，就用 JSON
- 如果不能解析 JSON，按普通文本降级
- 如果还是无法判断，就返回 `invalid` 或 `unknown`

## 7. 接口层要怎么配合

普通问题 500 也可能来自接口参数不兼容，所以路由层要一起修。

### 7.1 请求模型建议

当前建议保留兼容字段：

```python
class SubmitTurnRequest(BaseModel):
    turn_type: Literal["input", "judge"] = Field(default="input")
    question: str | None = Field(default=None)
    content: str | None = Field(default=None)
    judgement: bool | None = Field(default=None)
    mode: str = Field(default="agent")
```

### 7.2 路由转发建议

```python
@router.post("/{game_id}/turns")
async def submit_turn(game_id: str, request: SubmitTurnRequest, service: GameService = Depends(get_game_service)):
    try:
        user_text = request.content or request.question or ""
        if request.turn_type == "judge":
            user_text = "是" if request.judgement else "否"
        return await service.submit_turn(game_id, user_text, request.mode)
    except ValueError as exc:
        raise HTTPException(status_code=status.HTTP_400_BAD_REQUEST, detail=str(exc))
    except Exception:
        raise HTTPException(status_code=status.HTTP_500_INTERNAL_SERVER_ERROR, detail="Failed to submit turn")
```

### 7.3 为什么这里要兼容旧字段

因为你现在很可能已经有旧客户端或手工请求还在传 `question`，如果直接切掉，调试时会多出一层假故障。

---

## 8. 修复顺序

建议你按下面顺序做，不要跳。

### 第一步：补 `_handle_user_question()`

这是 500 的直接根因，必须先补。

### 第二步：让 `answer_question()` 返回稳定结构

确保普通问题路径不会因为模型结果格式不一致而崩。

### 第三步：把辅助方法改成静态方法

避免 `self` / 非 `self` 调用混乱。

### 第四步：检查 `update_game_state()` 是否可写回

历史、摘要、阶段必须能真正落库。

### 第五步：统一接口请求字段

让路由兼容旧请求和新请求。

## 9. 验证方法

修完以后，你至少要验证下面这几个场景。

### 9.1 普通提问

请求示例：

```json
{
  "question": "它属于计算机领域吗？",
  "mode": "agent"
}
```

预期：

- 返回 200
- 有 `answer`
- 有 `history`
- `phase` 被正确更新

### 9.2 用户猜测

请求示例：

```json
{
  "question": "我猜是编译器",
  "mode": "agent"
}
```

预期：

- 猜中时结束
- 猜错时回到下一轮

### 9.3 系统猜测确认

请求示例：

```json
{
  "turn_type": "judge",
  "judgement": true
}
```

预期：

- 返回结束结果
- 不再允许继续提交

## 10. 你最应该检查的三处

如果你只想先修三处，优先级应该是：

1. [backend/app/services/game_service.py](../../backend/app/services/game_service.py) 里的 `_handle_user_question()`
2. [backend/app/agents/guess_agent.py](../../backend/app/agents/guess_agent.py) 里的 JSON 解析兜底
3. [backend/app/api/v1/submit_turn.py](../../backend/app/api/v1/submit_turn.py) 里的请求字段兼容

这三处通了，普通问题 500 基本就能被消掉。

---

## 11. 结论

这次报错不是“模型坏了”，也不是“数据库坏了”，而是普通问题链路里缺了一个关键处理函数，导致服务层在普通问题分支上直接断掉。

你修复时只要记住一句话：

**普通问题路径必须有完整闭环：解析意图 -> 回答问题 -> 更新状态 -> 返回下一步。**
