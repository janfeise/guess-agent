# 7 创建游戏接口手册

## 1. 文档目标

这一章的目标是指导你完成“创建游戏”这个最基础、最先要打通的后端接口。

写完这一章后，你应该能够明确下面这些事情：

- 创建游戏接口应该放在哪一层
- 请求参数应该长什么样
- 接口应该返回什么内容
- 接口如何调用 `GameService.create_game()`
- 创建成功和创建失败时应该如何响应

这一章只聚焦“创建游戏”这一件事，不展开其他游戏操作接口。

---

## 2. 创建游戏接口在架构中的位置

在当前项目里，创建游戏接口属于 API 层。

推荐调用链路如下：

```text
前端 / 客户端 -> API 路由 -> GameService.create_game() -> GameRepository.create_game() -> MongoDB
```

### 2.1 API 层负责什么

API 层负责：

- 接收请求
- 校验最基础的输入
- 调用 Service 层
- 组织 HTTP 响应

### 2.2 Service 层负责什么

Service 层负责：

- 生成 `game_id`
- 组装初始游戏数据
- 调用 Repository 写入数据库
- 返回创建结果

### 2.3 Repository 层负责什么

Repository 层负责：

- 将游戏主文档写入 MongoDB
- 保证 `game_id` 唯一
- 返回数据库写入结果

---

## 3. 为什么要先做创建游戏接口

创建游戏接口是整条游戏链路的入口。

如果这个接口没有先打通，后续这些事情都没法稳定推进：

- 提交回合
- 查询游戏状态
- 流式返回答案
- 结束游戏

先把创建游戏做通，才能保证后续接口都有一个明确的 `game_id` 可以依赖。

---

## 4. 接口职责边界

### 4.1 创建游戏接口应该做的事情

创建游戏接口应该负责：

- 接收创建请求
- 读取可选参数，比如 `owner_id`
- 调用 `GameService.create_game()`
- 返回新建游戏信息

### 4.2 创建游戏接口不应该做的事情

创建游戏接口不应该负责：

- 不直接操作 MongoDB
- 不拼接游戏历史
- 不调用 Agent
- 不处理回合逻辑
- 不做复杂业务判断

接口层要尽量薄，只做请求转发和响应包装。

---

## 5. 推荐路由设计

### 5.1 路由前缀

建议把游戏接口放在：

- `/api/v1/games`

### 5.2 创建游戏路由

建议使用：

- `POST /api/v1/games`

这个设计比较符合 REST 风格，也容易理解：

- `POST` 表示创建资源
- `games` 表示创建的是一局游戏

### 5.3 路由命名建议

如果你使用 FastAPI 的 `APIRouter`，建议路由函数命名为：

- `create_game`

这样和 Service 层的方法名也能保持一致，降低理解成本。

---

## 6. 请求结构设计

### 6.1 最小请求体

当前 demo 阶段，创建游戏接口可以先只接收一个可选字段：

- `owner_id`

如果你暂时不需要用户系统，也可以直接不传请求体，接口自动创建一局空游戏。

### 6.2 推荐请求格式

```json
{
  "owner_id": "user-001"
}
```

### 6.3 如果暂时没有 owner_id

如果你当前还没有用户体系，可以把请求体设计成空对象，也就是：

```json
{}
```

### 6.4 为什么不要一开始就塞太多参数

因为创建游戏接口的核心任务只是“开局”。

如果一开始就加入太多参数，会导致：

- 接口不清晰
- 后续不好维护
- 参数验证复杂化

demo 阶段先把最小链路打通最重要。

---

## 7. 返回结构设计

### 7.1 创建成功返回什么

创建成功后，建议返回这些字段：

- `game_id`
- `status`
- `round_count`
- `created_at`

如果你想让前端拿到更多信息，也可以补充：

- `updated_at`
- `summary`

### 7.2 推荐返回示例

```json
{
  "game_id": "b8f2e6f0-3f60-4a7c-9b7a-9a0d7f2d0f1a",
  "status": "active",
  "round_count": 0,
  "created_at": "2026-04-11T10:20:30"
}
```

### 7.3 为什么返回值不建议太大

因为创建接口只是初始化动作，不需要一次性把完整历史都返回。

如果返回太多，会让接口看起来混乱，也没有必要。

---

## 8. 需要的请求模型

如果你使用 FastAPI，建议为创建请求单独定义一个请求模型。

### 8.1 请求模型职责

请求模型负责：

- 定义允许传入哪些字段
- 做基础类型校验
- 让接口文档自动生成更清晰的说明

### 8.2 建议字段

当前阶段最简单的请求模型可以只包含：

- `owner_id`：可选字符串

### 8.3 为什么要单独定义模型

因为这样可以让接口输入更明确，也方便后续扩展。

例如未来你可能会加：

- `game_mode`
- `difficulty`
- `metadata`

有单独模型以后，这些扩展就很容易加。

---

## 9. 创建游戏接口的实现流程

推荐把接口实现分成下面几个步骤。

### 9.1 接收请求

API 路由先接收请求参数。

如果没有请求体，就直接进入创建逻辑；如果有请求体，就先做校验。

### 9.2 调用 Service

API 层不要自己生成 `game_id`，也不要自己组装数据库文档，而是直接调用：

- `GameService.create_game(owner_id=...)`

### 9.3 返回结果

Service 返回创建结果后，API 层把它包装成 HTTP 响应即可。

## 10. 参考实现结构

这一部分给出创建游戏接口的推荐骨架。

```python
from fastapi import APIRouter, Depends

router = APIRouter(prefix="/games", tags=["games"])


@router.post("")
async def create_game(request: dict):
    ...
```

### 10.1 推荐依赖方式

实际实现时，建议通过依赖注入拿到 `GameService` 实例，而不是在路由里直接 new 对象。

这样做的好处是：

- 依赖关系更清晰
- 便于测试
- 便于统一管理配置和数据库连接

### 10.2 推荐路由函数职责

路由函数只负责：

- 读取输入
- 调用 Service
- 返回结果

不要把业务逻辑写进路由函数。

---

## 11. 接口实现思路

### 11.1 如果没有用户体系

如果你暂时没有登录系统，可以直接让接口创建一局空游戏。

这种情况下，`owner_id` 可以不传，Service 内部默认处理为 `None`。

### 11.2 如果有用户体系

如果后续接入了用户系统，那么创建游戏接口可以把当前用户 ID 作为 `owner_id` 传入。

这样后续就能按用户查询游戏列表。

### 11.3 为什么创建接口要尽量无状态

因为创建游戏本身是一个初始化动作，不需要依赖前置游戏数据。

越无状态，接口越稳定，测试也越容易。

---

## 12. 与 GameService 的联动方式

创建游戏接口和 `GameService.create_game()` 是一对一关系。

### 12.1 接口层

接口层负责接收 HTTP 请求。

### 12.2 Service 层

Service 层负责生成 `game_id`、组装初始文档、调用 Repository。

### 12.3 Repository 层

Repository 层负责把游戏文档插入 MongoDB。

这三层职责要严格分开，不然后面维护会很痛苦。

---

## 13. 成功和失败响应建议

### 13.1 成功响应

如果创建成功，建议返回：

- `200 OK`

或者更符合创建语义的：

- `201 Created`

### 13.2 失败响应

常见失败情况包括：

- 数据库写入失败
- `game_id` 冲突
- 输入参数非法

这些错误应该在 API 层转换成合适的 HTTP 状态码。

### 13.3 推荐状态码

- `400 Bad Request`：请求参数不合法
- `500 Internal Server Error`：数据库或内部异常

如果你后续做得更细，还可以加入更明确的业务错误码。

---

## 14. 异常处理建议

### 14.1 创建接口常见异常

常见异常包括：

- 创建参数为空
- `GameRepository.create_game()` 抛出重复 `game_id` 异常
- MongoDB 插入失败

### 14.2 推荐处理方式

建议接口层只做一层很薄的异常转换：

- 业务异常转换成 `400`
- 未知异常转换成 `500`

不要把所有异常都吞掉，否则排查会很困难。

---

## 15. 调试顺序建议

建议按下面顺序验证创建游戏接口：

1. 先确认路由能注册成功
2. 再确认请求能进入接口函数
3. 再确认 Service 能拿到 `owner_id`
4. 再确认 Repository 能写入 MongoDB
5. 最后确认返回值里有 `game_id`

### 15.1 先测创建结果

创建成功后，直接看返回的 `game_id` 是否存在。

### 15.2 再查数据库

确认 MongoDB 里确实出现了这条记录，并且字段完整。

### 15.3 再看初始状态

确认 `history` 为空、`round_count` 为 0、`status` 为 `active`。

---

## 16. 参考实现顺序

如果你准备动手写这个接口，建议按下面顺序来：

1. 先确定路由前缀和方法名
2. 再定义请求模型
3. 再接入 `GameService`
4. 再调用 `create_game()`
5. 最后补错误处理和返回结构

这个顺序可以让你先跑通主链路，再逐步增强细节。

---

## 17. 与后续接口的关系

创建游戏接口是后续接口的起点。

等这个接口写完后，下一步通常就是：

- 根据 `game_id` 获取游戏
- 提交普通回合
- 提交流式回合
- 结束游戏

所以这一章其实是在为后面整个游戏 API 铺地基。

---

## 18. 本章结论

创建游戏接口应该尽量简单、稳定、无状态。

它最重要的职责就是：

- 接收请求
- 调用 `GameService.create_game()`
- 返回新建游戏的基础信息

只要这个接口稳定，后面的所有游戏流程就都有了统一的起点。

## 19. 创建游戏接口详细代码实现参考

这一节把创建游戏接口拆成“文件级”和“方法级”两部分来讲，目标是让你可以直接照着写 `backend/app/api/v1/games.py`，而不是只停留在接口概念上。

### 19.1 建议文件结构

建议把创建游戏接口放在单独的路由文件里，例如：

```text
backend/
└── app/
  └── api/
    └── v1/
      ├── health.py
      └── games.py
```

这样做的好处是：

- 健康检查和游戏业务解耦
- 后续如果再加提交回合、结束游戏，也能继续放在同一个模块里
- 路由文件职责更清晰

如果后面接口越来越多，也可以继续拆分成：

- `games_create.py`
- `games_turns.py`
- `games_admin.py`

但当前 demo 阶段不需要拆得太碎。

---

### 19.2 路由模块应该包含什么

一个完整的创建游戏接口模块，建议至少包含下面四部分：

1. 路由对象
2. 请求模型
3. 依赖注入函数
4. 路由处理函数

这四部分分别负责不同事情：

- 路由对象：定义 URL 前缀和标签
- 请求模型：定义输入字段
- 依赖注入函数：构造 `GameService`
- 路由处理函数：接收请求并返回结果

---

### 19.3 请求模型的详细实现

创建游戏接口的请求模型建议尽量简单。

#### 推荐模型字段

- `owner_id`：可选字符串

#### 推荐实现思路

```python
from pydantic import BaseModel, Field


class CreateGameRequest(BaseModel):
  owner_id: str | None = Field(default=None, description="创建者或所属用户 ID")
```

#### 设计考虑

- 先把输入限制住，避免接口接收任意字典
- 给字段加描述，方便自动生成接口文档
- 如果后续要加字段，只需要扩展这个模型即可

#### 为什么不用 `dict`

如果直接用 `dict`，虽然短，但会有这些问题：

- 输入结构不清楚
- 文档自动生成效果差
- 后续难以做字段级校验

所以建议一开始就用 Pydantic 模型。

---

### 19.4 响应模型的详细实现

虽然创建接口也可以直接返回字典，但如果你希望接口更规范，建议再定义一个响应模型。

#### 推荐字段

- `game_id`
- `status`
- `round_count`
- `created_at`

#### 推荐实现思路

```python
from pydantic import BaseModel
from datetime import datetime


class CreateGameResponse(BaseModel):
  game_id: str
  status: str
  round_count: int
  created_at: datetime
```

#### 设计考虑

- 响应模型让返回值更稳定
- 前端更容易知道会拿到哪些字段
- 后续接口演进时不容易乱

如果你暂时不想再多写一个模型，也可以先返回字典，但文档上建议把结构定下来。

---

### 19.5 依赖注入函数的详细实现

创建游戏接口不能自己在路由里手动 new 所有对象，而应该通过依赖注入统一拿到 `GameService`。

#### 推荐依赖对象

构造 `GameService` 时通常需要：

- `settings`
- `agent`
- `repository`
- `memory_policy`

#### 推荐实现思路

```python
from fastapi import Depends


def get_game_service() -> GameService:
  settings = get_settings()
  db = get_database()
  repository = GameRepository(db)
  agent = get_guess_agent(settings)
  memory_policy = MemoryPolicy(max_rounds=settings.max_rounds)
  return GameService(settings, agent, repository, memory_policy)
```

#### 设计考虑

- 依赖注入让路由更干净
- 如果以后要替换 Agent 或 Repository，只改这里就行
- 统一入口有利于测试和维护

#### 说明

如果你目前已经在别的地方封装了 `get_game_service()`，这里的文档只需要保持同样的思路即可。

---

### 19.6 路由处理函数的详细实现

路由处理函数是整个创建游戏接口最核心的 API 入口。

#### 推荐处理流程

1. 接收请求模型
2. 从依赖注入中拿到 `GameService`
3. 调用 `service.create_game(owner_id=...)`
4. 捕获业务异常并转换成 HTTP 错误
5. 返回创建结果

#### 推荐实现思路

```python
from fastapi import APIRouter, Depends, HTTPException, status

router = APIRouter(prefix="/games", tags=["games"])


@router.post("", status_code=status.HTTP_201_CREATED)
async def create_game(request: CreateGameRequest, service: GameService = Depends(get_game_service)):
  try:
    result = await service.create_game(owner_id=request.owner_id)
    return result
  except ValueError as exc:
    raise HTTPException(status_code=status.HTTP_400_BAD_REQUEST, detail=str(exc))
  except Exception as exc:
    raise HTTPException(status_code=status.HTTP_500_INTERNAL_SERVER_ERROR, detail="Failed to create game")
```

#### 设计考虑

- 使用 `201 Created` 更符合创建资源的语义
- `ValueError` 可以先统一当成业务错误处理
- 未知异常直接返回 500，避免把内部细节暴露给前端

#### 为什么这里要捕获异常

因为 Service 层和 Repository 层都可能抛错。

接口层应该把这些内部错误翻译成用户能理解的 HTTP 响应，而不是直接把 Python 异常暴露出去。

---

### 19.7 路由返回值的详细说明

创建游戏成功后，路由建议直接返回 Service 的结果，或者返回经过轻微整理后的结果。

#### 推荐返回内容

- `game_id`
- `status`
- `round_count`
- `created_at`

#### 为什么不建议把完整文档全部返回

因为创建动作只需要告诉前端：

- 新游戏已经创建成功
- 这局游戏的 ID 是什么
- 当前状态是什么

把完整 history 返回出去不仅没有必要，还会让返回体变重。

---

### 19.8 接口文件的完整参考骨架

下面给出一个更接近最终文件结构的参考骨架，方便你理解整个创建游戏接口是怎么拼起来的。

```python
from fastapi import APIRouter, Depends, HTTPException, status
from pydantic import BaseModel, Field


class CreateGameRequest(BaseModel):
  owner_id: str | None = Field(default=None, description="创建者或所属用户 ID")


class CreateGameResponse(BaseModel):
  game_id: str
  status: str
  round_count: int
  created_at: datetime


router = APIRouter(prefix="/games", tags=["games"])


@router.post("", status_code=status.HTTP_201_CREATED)
async def create_game(request: CreateGameRequest, service: GameService = Depends(get_game_service)):
  try:
    return await service.create_game(owner_id=request.owner_id)
  except ValueError as exc:
    raise HTTPException(status_code=status.HTTP_400_BAD_REQUEST, detail=str(exc))
  except Exception:
    raise HTTPException(status_code=status.HTTP_500_INTERNAL_SERVER_ERROR, detail="Failed to create game")
```

这个骨架里最重要的是理解职责分离：

- `CreateGameRequest` 管输入
- `CreateGameResponse` 管输出
- `get_game_service()` 管依赖
- `create_game()` 路由函数管流程

---

### 19.9 与 `main.py` 的注册方式

创建游戏接口写好以后，还需要在入口文件里注册路由。

#### 推荐注册方式

```python
from app.api.v1.games import router as games_router

app.include_router(games_router, prefix="/api/v1")
```

#### 设计考虑

- 路由统一挂在 `/api/v1` 下，方便版本控制
- `games` 路由文件只管游戏接口，不管健康检查
- 后续新增更多接口时也可以同样注册

---

### 19.10 创建接口的测试思路

写完创建游戏接口后，建议按下面顺序测试：

1. 先确认路由是否能够访问
2. 再确认请求体能正确解析成模型
3. 再确认 `GameService.create_game()` 是否被调用
4. 再确认 MongoDB 是否落库成功
5. 最后确认返回值里是否有 `game_id`

#### 推荐测试重点

- 请求体为空时是否也能创建成功
- `owner_id` 传入时是否会被保存
- 数据库中是否出现 `games` 记录
- `status` 和 `round_count` 是否符合初始化要求

## 20. 本章补充结论

如果你按照上面的方式实现创建游戏接口，这一层的代码会很清晰：

- 路由只负责接收和返回
- 请求模型只负责输入约束
- 响应模型只负责输出约束
- `GameService` 负责业务动作
- Repository 负责数据落库

这也是后面提交回合、结束游戏、查询游戏接口继续扩展时最稳妥的写法。
